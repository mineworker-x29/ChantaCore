# v0.21.1 Self-Capability Registry Awareness

Korean name: 자기 능력 레지스트리 관조

Track: Deep Self-Introspection

## Version Definition

v0.21.1 is the first concrete OCEL-native deep self-introspection subject. It
creates read-only awareness of ChantaCore's capability registry truth.

Capability truth > persona claim.

Deep self-introspection is not self-modification. Capability awareness is not
capability enablement. Capability registry view is not permission grant.
Registry view is read-only.

Capability awareness is not capability enablement.

## Objective

This version lets ChantaCore inspect what capabilities exist, which are
implemented or contract-only, which are future-track or blocked, whether a
capability is read-only or candidate-only, whether it has risk/gate/observability
surfaces, and whether optional self-claims contradict registry truth.

## Relationship To v0.21.0 And v0.20.9

v0.21.0 introduced the contract-only `deep_self_introspection` layer. v0.21.1
implements the `subject:capability_registry` read model while keeping the other
v0.21 seed subjects contract-only.

v0.20.9 provides the OCEL-native Self-Awareness Foundation v1 consolidation
that acts as source evidence for deep self-introspection.

## Non-Goals

v0.21.1 does not enable capabilities, enable skills, enable external candidates,
enable adapters, create permission grants, activate write/edit/apply_patch,
activate shell/test/lint, activate network/web/MCP/plugin access, activate
external harness execution, mutate memory/persona/overlay, promote candidates,
materialize candidates, auto-fix self-claims, use LLM judge/planner, perform
runtime probing, or perform provider/API probing.

## Data Models

`SelfCapabilityRegistryViewRequest` filters the registry view by layer/status
and controls whether risk, gate, observability, and OCEL mapping surfaces are
included.

`SelfCapabilityRecord` is the normalized capability read model. It includes
skill id, layer, status, execution/materialization/promotion flags, effect
types, OCEL mappings, source refs, and evidence refs.

`SelfCapabilityRiskProfileView`, `SelfCapabilityGateView`, and
`SelfCapabilityObservabilityView` expose read-only safety, gate, and visibility
state for each capability.

`SelfCapabilityTruthReport` records registry truth findings, unsafe claim
counts, dangerous capability counts, missing gate/observability counts,
withdrawal conditions, and validity horizon. It remains `report_only`.

## Service Design

`SelfCapabilityRegistrySourceService` reads existing self-awareness contracts,
deep self-introspection contracts, observation/digestion skill ids, and injected
external candidate records if present. External candidates are forced to
future-track and non-executable.

`SelfCapabilityRegistryAwarenessService` builds snapshots and truth reports. It
does not mutate registries, enable skills, create permission grants, or write
canonical state.

## Truth Check Rules

Truth check passes when dangerous capability count is zero, implemented
capabilities have OCEL mapping and gate/envelope visibility, future-track
capabilities are not marked implemented, contract-only capabilities are not
executable, and candidate-only capabilities are not materialized or promoted.

Truth check warns for partial observability, future-track capabilities that are
correctly marked, or unknown optional capability claims.

Truth check fails for dangerous capabilities, write/shell/network/MCP/plugin or
external execution flags, memory/persona/overlay mutation flags, future-track
marked implemented, contract-only executable, or candidate-only materialized or
promoted.

## OCEL Mapping

Object types include capability registry, capability record, registry snapshot,
truth report, truth finding, risk profile view, gate view, observability view,
skill contract, deep self-introspection subject, self-awareness capability,
external candidate capability, execution envelope, PIG report, and OCPX
projection.

Event types include registry view requested, snapshot created, risk/gate/
observability views created, truth check requested, truth report created,
warning created, and violation detected.

Relation types include registry view, capability containment, skill contract
description, risk/gate/observability view links, truth checks, issue findings,
claim contradiction, registry verification, workbench visibility, envelope
recording, and derivation from the deep self contract.

Effect types are `read_only_observation` and `state_candidate_created`.

## CLI Commands

The repository package name is `chanta_core`.

- `python -m chanta_core.cli deep-self capability registry`
- `python -m chanta_core.cli deep-self capability registry --status implemented`
- `python -m chanta_core.cli deep-self capability registry --layer self_awareness`
- `python -m chanta_core.cli deep-self capability truth-check`
- `python -m chanta_core.cli deep-self capability truth-check --claims claims.json`
- `python -m chanta_core.cli deep-self capability risks`
- `python -m chanta_core.cli deep-self capability observability`

CLI output shows capability counts, dangerous count, execution/materialization/
promotion counts, missing observability count, registry truth status, and
`Capability truth > persona claim`. It does not print raw file content, raw
secrets, private material, or full private paths.

## PIG/OCPX Report

The PIG projection uses version `v0.21.1`, layer
`deep_self_introspection`, subject `capability_registry`, and principle
`Capability truth > persona claim`. It includes implemented, contract-only,
future-track, dangerous, execution, materialization, and canonical promotion
counts.

The OCPX projection uses state `self_capability_registry_awareness`, source read
models `SelfAwarenessReleaseState` and `DeepSelfIntrospectionContractState`, and
target read models `SelfCapabilityRegistryState`, `SelfCapabilityTruthState`,
`SelfCapabilityRiskState`, and `SelfCapabilityContradictionState`.

semantics, CLI views, PIG/OCPX projections, non-goal drift, and v0.21.0/v0.20.x

## Acceptance Criteria

- Capability registry snapshot builds.
- Capability truth report builds.
- Risk/gate/observability views build.
- Dangerous capabilities are detected.
- Contract-only, future-track, executable, materialized, and promoted
  contradictions are detected.
- No capability or skill is enabled.
- No permission grant is created.
- No registry mutation occurs.
- No mutation, execution, LLM judge, or external execution is introduced.
- OCEL/PIG/OCPX visibility exists.
- v0.21.0 and v0.20.x tests still pass.

## Withdrawal Conditions

Withdraw v0.21.1 if registry awareness mutates a registry, enables a capability
or skill, creates permission grants, activates unsafe execution, uses LLM judge
or planner, performs runtime/provider probing, promotes or materializes a
candidate, exposes private material, prints raw file content, prints raw
secrets, or treats persona claims as stronger than registry truth.
