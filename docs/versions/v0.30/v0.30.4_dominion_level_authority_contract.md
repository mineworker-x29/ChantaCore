# v0.30.4 Dominion Level / Authority Contract

Parent track: External Dominion Control Plane Foundation

## Relationship To Earlier Layers

v0.30.0 defines the External Dominion contract, `DominionLevel`, and the v0.30 grant ceiling. D0-D3 are the only grantable levels in v0.30.x, and D4-D9 remain future-track.

v0.30.1 defines External Target Inventory, Identity, Availability, and Trust Boundary. Inventory is not permission, identity is not trust, and trust boundary is not execution authority.

v0.30.2 defines `CapabilityObservationReport` and `ExternalCapabilityDescriptor`. Observation is not permission, certification, an internal skill, or a Dominion target.

v0.30.3 defines digestion candidates and assimilation decisions. Digestion candidate is not digestion, assimilation decision is not implementation, and `dominion_required` is not Dominion authority.

v0.30.4 makes Dominion authority explicit as contract vocabulary only.

## Definition Of Dominion Authority

Dominion authority is the governance vocabulary for deciding whether an external target, capability, candidate, report, profile, mission, delegation, gateway, provider, or RPA surface may later be observed, described, planned, simulated, deferred, denied, blocked, reviewed, or marked future-track.

Dominion is not external control yet. Dominion authority request is not authority grant. Dominion authority scope is not runtime permission. Dominion authority decision is not execution. Dominion grant boundary is not runtime implementation.

## Request, Evaluation, Decision, Review, Execution

`DominionAuthorityRequest` records what level is requested and over what scope.

`DominionAuthorityEvaluation` is advisory. It summarizes the maximum allowed level, dangerous boundary/effect surfaces, risk signals, recommended decision, and recommended granted level.

`DominionAuthorityDecision` records the governance outcome. It may deny, defer, require review, observe, describe, plan, simulate, mark future-track, block, or no-op. It does not execute anything and does not grant D4+.

`DominionAuthorityReviewRequirement` records symbolic review requirements. Review required is not approval. Approval required is not approval granted.

Actual execution, delegation, provider invocation, network access, credential access, command execution, browser automation, RPA control, and gateway control are outside v0.30.4.

## DominionLevel Ladder And v0.30.4 Grant Ceiling

- `D0_OBSERVE`: observe-only vocabulary.
- `D1_DESCRIBE`: describe-only vocabulary.
- `D2_PLAN`: plan-only vocabulary.
- `D3_SIMULATE`: bounded simulation vocabulary.
- `D4_EXECUTE_READ`: future-track.
- `D5_EXECUTE_WRITE_PROPOSAL`: future-track.
- `D6_EXECUTE_SANDBOX`: future-track.
- `D7_EXECUTE_NETWORK_PREVIEW`: future-track.
- `D8_DELEGATE_AGENT`: future-track.
- `D9_GATEWAY_CONTROL`: future-track.

v0.30.4 grant ceiling is `D3_SIMULATE`. D4-D9 remain future-track in v0.30.4.

## Authority Scope

`DominionAuthorityScope` defines:

- scope id and kind;
- target id;
- capability, candidate, and report ids;
- allowed and prohibited boundary surfaces;
- allowed and prohibited effect surfaces;
- maximum level;
- optional expiry;
- evidence references.

Scope kind is descriptive. Gateway, provider, RPA, delegation, and unknown scopes are conservative. Gateway/provider/RPA/network/credential/command/browser/delegation surfaces are prohibited by default and must not be treated as runtime permission.

## Request Status

- `unknown`: conservative.
- `requested`: request only, not granted.
- `evidence_linked`: requires evidence refs.
- `review_required`: review is not approval.
- `denied`: valid non-progressing status.
- `deferred`: valid non-progressing status.
- `blocked`: stops later v0.30.4 authority flow.
- `future_track`: not readiness.

## Decision Types

- `deny`: no grant.
- `defer`: no grant.
- `no_op`: no-action outcome.
- `require_review`: review required, not approval.
- `observe_only`: at most D0.
- `describe_only`: at most D1.
- `plan_only`: at most D2.
- `simulate_only`: at most D3.
- `future_track`: required or expected for D4-D9 requests.
- `blocked`: stops progression and requires blocked reasons.

No decision type means actual execution.

## Review Requirement

`DominionAuthorityReviewRequirement` records review reasons and symbolic required reviewers. It does not approve anything and does not grant authority.

## D4-D9 Future-Track Rule

D4-D9 can be requested for governance modeling, but cannot be granted in v0.30.4. A D4-D9 request must become deny, defer, no-op, require_review, future_track, or blocked, with `granted_level=None` or a D0-D3 level only.

Future-track is not readiness.

## Safe Defaults

Safe descriptive candidates may receive D0, D1, or D2 outcomes. A simulation-only candidate may receive D3 only under conservative conditions with no dangerous allowed boundary/effect surfaces.

Provider invocation, gateway control, command execution, browser automation, RPA control, credential access, network access, and delegation surfaces are high risk. They route to review, future-track, denial, deferral, no-op, or blocked outcomes, not runtime authority.

## Flow

`AssimilationDecision / DigestionCandidate -> DominionAuthorityRequest -> DominionAuthorityEvaluation -> DominionAuthorityDecision -> ReviewRequirement or v0.30.5 dry-run handoff`

An assimilation decision with `dominion_required` may generate an authority review request, but it must not grant authority.

## Hard Prohibitions

v0.30.4 must not:

- execute external targets;
- invoke external agent harnesses;
- call provider SDKs;
- perform network calls;
- access credentials or secrets;
- use subprocess, shell, or command execution;
- implement RPA runtime control;
- implement browser automation;
- implement gateway/channel message send;
- implement external delegation execution;
- persist raw external output as memory;
- add automatic retry or rollback execution;
- mark any target as live-runtime-ready;
- bypass v0.29 provider adapter gates;
- raise v0.30 grantability beyond `D3_SIMULATE`;
- allow D4-D9 as actual runtime grants;
- treat `approval_required` as `approval_granted`;
- treat `human_review_required` as approval;
- treat v0.30.3 `dominion_required` as authority granted.

## Gate Inheritance

v0.30.4 inherits v0.29 provider adapter gates and all v0.30.0, v0.30.1, v0.30.2, and v0.30.3 boundaries.

Runtime readiness flags remain false:

- `provider_invocation_runtime_ready = False`
- `limited_preview_execution_ready_now = False`
- `live_adapter_runtime_ready = False`
- `external_agent_dominion_runtime_ready = False`
- `network_access_ready = False`
- `credential_access_ready = False`
- `command_execution_ready = False`
- `rpa_runtime_control_ready = False`
- `gateway_control_ready = False`

No runtime readiness flag changes in v0.30.4.

## Acceptance Criteria

- Dominion authority request status taxonomy exists.
- Dominion authority decision type taxonomy exists.
- Dominion authority scope kind taxonomy exists.
- `DominionAuthorityScope`, `DominionAuthorityRequest`, `DominionAuthorityEvaluation`, `DominionAuthorityDecision`, and `DominionAuthorityReviewRequirement` exist.
- Helpers are pure.
- Requests do not become grants.
- Evaluations remain advisory.
- Decisions execute nothing.
- Review requirements approve nothing.
- Approval required and review required do not become approval granted.
- D4+ remains future-track.
- Focused v0.30.0-v0.30.4 tests pass.

## Withdrawal Conditions

Withdraw this release interpretation if:

- any helper executes, fetches, invokes, registers, installs, writes runtime artifacts, accesses credentials, or sends gateway/channel messages;
- any authority decision grants D4+;
- any review requirement grants authority;
- `approval_required` or `human_review_required` is treated as approval granted;
- any runtime readiness flag becomes true;
- D4-D9 are described as ready.

## Handoff To v0.30.5

v0.30.5 External Delegation Candidate / Dry-Run Plan may consume v0.30.4 decisions. It must treat them as governance records only. A v0.30.4 authority decision is not execution and is not delegation runtime.
