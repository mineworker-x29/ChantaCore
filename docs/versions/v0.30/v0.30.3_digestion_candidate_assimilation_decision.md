# v0.30.3 Digestion Candidate / Assimilation Decision Contract

Parent track: External Dominion Control Plane Foundation

## Relationship To Earlier v0.30 Layers

v0.30.0 defines the External Dominion contract and keeps v0.30 grantability capped at `D3_SIMULATE`.

v0.30.1 adds External Target Inventory, Identity, Availability, and Trust Boundary records. Inventory is not permission, identity is not trust, trust boundary is not execution authority, and availability is not invocation readiness.

v0.30.2 adds `CapabilityObservationReport` and `ExternalCapabilityDescriptor`. Observation reports normalize external capability surfaces into capability maps and risk maps. A capability observation report is not permission, certification, an internal skill, or a Dominion target.

v0.30.3 builds only the next contract step: observed capability to digestion candidate, feasibility report, assimilation decision, and optional internalization plan.

## Definition Of Digestion

Digestion means reviewing an observed external capability for ChantaCore-internal pattern value. It may identify a tool contract pattern, skill manifest pattern, mission manifest pattern, policy rule pattern, memory schema pattern, result envelope pattern, prompt assembly pattern, profile isolation pattern, gateway manifest pattern, provider adapter pattern, delegation packet pattern, trace event pattern, or approval boundary pattern.

Digestion candidate is not digestion. A digestible pattern is not copied code. It is a bounded governance record that preserves evidence, risk, assumptions, and withdrawal conditions.

## Digestion Candidate

`DigestionCandidate` records:

- what external capability is being considered;
- which v0.30.2 observation report and capability ids support it;
- which descriptive candidate kind applies;
- which future internal artifact candidate kind could be reviewed later;
- which evidence references support the candidate;
- which risks, effect surfaces, and boundary surfaces remain;
- whether the candidate is digestible, deferred, rejected, blocked, Dominion-required, or future-track.

An internal artifact candidate kind is not an active internal artifact. A candidate does not create a skill, tool, mission, policy, memory writer, provider adapter, gateway adapter, or runtime registration.

## Assimilation Decision

`AssimilationDecision` records whether a digestion candidate should be treated as:

- `candidate`;
- `defer`;
- `reject`;
- `dominion_required`;
- `blocked`;
- `future_track`;
- `no_op`.

Assimilation decision is not implementation. It does not grant permission, DominionAuthority, D4+ capability, runtime readiness, or runtime control. `dominion_required` is a routing signal for v0.30.4+ review only.

## Internalization Plan

`InternalizationPlan` is a plan-only artifact. It may list later contracts, tests, reviews, and out-of-scope runtime work for a candidate decision.

Internalization plan is not runtime registration. It must keep `no_execution_guarantee=True` and `no_runtime_registration_guarantee=True`.

## Feasibility Status Definitions

- `unknown`: insufficiently classified.
- `digestible_pattern_only`: evidence-linked pattern may be reviewed, but is not internalization.
- `schema_extractable`: schema may be extracted later, but generated code is not created.
- `requires_internal_design`: further internal ChantaCore design is required.
- `requires_review`: more observation or review is needed.
- `requires_dominion_review`: route to later Dominion reasoning without creating DominionTarget.
- `not_digestible`: not currently fit for internal digestion.
- `unsafe_to_digest`: unsafe as an internal candidate under current boundaries.
- `blocked`: stops v0.30.3 progression.
- `future_track`: not readiness.

## Assimilation Decision Type Definitions

- `candidate`: candidate-only, not an active artifact.
- `defer`: valid when evidence or design is insufficient.
- `reject`: valid when incompatible, too risky, or unfit.
- `dominion_required`: later Dominion review signal, not DominionAuthority.
- `blocked`: normal v0.30.3 flow stops.
- `future_track`: future gate required, not readiness.
- `no_op`: valid no-action outcome.

Rejected digestion is valid. It is a governance outcome, not proof that the external target is dangerous.

## Safe Defaults

Read-only, evidence-linked descriptive patterns may become candidate-only records. Provider invocation, network access, credential access, command execution, browser automation, RPA control, gateway send, external delegation, and memory mutation remain conservative and route toward defer, reject, blocked, Dominion-required, or future-track outcomes.

No-op remains a valid outcome.

## Flow

`CapabilityObservationReport -> DigestionCandidate -> DigestionFeasibilityReport -> AssimilationDecision -> InternalizationPlan -> v0.30.4 Dominion handoff or v0.31 Internal Triad future handoff`

The flow preserves these distinctions:

- observed external capability is not a digestion candidate;
- digestion candidate is not an internal artifact;
- feasibility report is not certification;
- assimilation decision is not permission;
- internalization plan is not implementation;
- Dominion handoff is not Dominion authority.

## Hard Prohibitions

v0.30.3 must not:

- execute external targets;
- implement live observation scanning;
- fetch `source_ref`;
- invoke external agent harnesses;
- call provider SDKs;
- perform network calls;
- access credentials or secrets;
- use subprocess, shell, or command execution;
- implement RPA runtime control;
- implement browser automation;
- implement gateway or channel message send;
- persist raw external output as memory;
- add automatic retry or rollback execution;
- mark any target as live-runtime-ready;
- bypass v0.29 provider adapter gates;
- raise v0.30 max grantability beyond `D3_SIMULATE`;
- convert observed capabilities into permissions;
- convert digestion candidates into active runtime artifacts;
- convert assimilation decisions into DominionAuthority.

## Gate Inheritance

v0.30.3 inherits v0.29 provider adapter gates and all v0.30.0, v0.30.1, and v0.30.2 boundaries.

Runtime readiness flags remain false:

- `provider_invocation_runtime_ready = False`
- `limited_preview_execution_ready_now = False`
- `live_adapter_runtime_ready = False`
- `external_agent_dominion_runtime_ready = False`
- `network_access_ready = False`
- `credential_access_ready = False`
- `command_execution_ready = False`
- `rpa_runtime_control_ready = False`
- `gateway_control_ready = False`

No runtime readiness flag changes in v0.30.3.

## Acceptance Criteria

- Digestion candidate taxonomy exists.
- Feasibility status taxonomy exists.
- Assimilation decision taxonomy exists.
- Internal artifact candidate taxonomy exists.
- Block and rejection reason taxonomy exists.
- `DigestionCandidate`, `DigestionFeasibilityReport`, `AssimilationDecision`, and `InternalizationPlan` validate their boundaries.
- Helper functions are pure.
- Dangerous surfaces do not become active internal runtime artifacts.
- Decisions do not grant permission or DominionAuthority.
- `D4+` remains future-track.
- Focused v0.30.0-v0.30.3 tests pass.

## Withdrawal Conditions

Withdraw this release interpretation if:

- any helper executes, fetches, invokes, registers, installs, writes runtime artifacts, accesses credentials, or sends gateway/channel messages;
- any candidate creates active internal skills, tools, missions, policies, memory writers, providers, gateways, browser runtimes, RPA runtimes, or delegation runtimes;
- any decision grants permission, DominionAuthority, or D4+ levels;
- any runtime readiness flag becomes true;
- rejected digestion is treated as proof of external danger rather than a governance outcome;
- future-track is treated as readiness.

## Handoff To v0.30.4

v0.30.4 should define Dominion Level / Authority Boundary for external capabilities that cannot be safely digested. v0.30.3 can route a candidate as `dominion_required`, but it does not create `DominionTarget`, grant Dominion authority, or implement runtime control.
