## Provider Config Contract

v0.41.3 defines provider config as metadata only:

- provider kind is descriptive and does not import provider SDKs
- allowed modes are `metadata_only` and `text_only`
- tool calling is closed
- function calling is closed
- completion in provider doctor is closed
- completion in run remains closed until v0.41.4
- remote network probe is closed

The profile provider config may be read from the explicit Default Personal
profile home if present. Missing config is reported as metadata, not treated as
runtime authority.

## Provider Doctor Contract

Allowed command:

```powershell
chanta-cli provider doctor --profile default-personal --home "%TEMP%\chantacore-v0413-test" --no-completion
```

The provider doctor may validate provider kind, base URL, model metadata, and
redacted secret references. It must not send a user prompt, submit a completion
request, start AgentLoop, execute skills, enable tool calling, enable function
calling, or print credential values.

## No-completion Policy

`--no-completion` is required for v0.41.3 provider doctor behavior.

Provider completion is recorded as explicitly blocked with target version
`v0.41.4`. A v0.41.3 provider doctor report may state configuration readiness
metadata for v0.41.4, but that metadata is not runtime authority.

## Safe Loopback `/models` Probe Policy

Optional command:

```powershell
chanta-cli provider doctor --profile default-personal --home "%TEMP%\chantacore-v0413-test" --no-completion --probe-models
```

The optional probe is allowed only when all conditions hold:

- `provider doctor` is the command
- `--no-completion` is present
- `--probe-models` is explicitly present
- provider base URL is loopback only: `localhost`, `127.0.0.1`, or `::1`
- path is `/models` or `/v1/models`
- timeout is short and bounded at no more than three seconds
- no user input is sent
- no prompt is sent
- no completion, chat completion, or responses endpoint is called
- no credentials are printed

Remote provider network probing remains closed in v0.41.3.

## Secret Redaction Contract

Secret references may record environment variable names and presence/absence.
Secret values are not loaded, persisted, or printed. Display strings must remain
redacted, such as `<redacted-present>` or `<not-set>`.

The secret redaction report must keep:

- all_values_redacted=True
- unsafe_secret_exposure_detected=False

## Read-only Skill Registry Contract

v0.41.3 defines a static read-only skill registry as metadata only. It does not
load executable skill implementations, dispatch skills, or grant runtime
authority.

Required default skill names:

- `profile_status`

- `provider_status`
- `restore_summary`
- `trace_recent`
- `trace_summary`
- `status_summary`
- `docs_reference_search`
- `list_available_skills`
- `unsafe_command_check`

All skill specs are inspectable but keep `execution_allowed=False`.

## Skill List Contract

Allowed command:

```powershell
chanta-cli skills list --profile default-personal --home "%TEMP%\chantacore-v0413-test"
```

The command lists registered static skill metadata. It does not invoke a
provider, submit a prompt, execute a skill, mutate files, call the network, or
access credentials.

## Skill Inspect Contract

Allowed command:

```powershell

```

The command returns a single static skill spec if present. It does not execute
the skill and does not load a SkillLoader or SkillExecutor.

## Known Limitations

- Provider config parsing is intentionally minimal metadata loading.
- Provider doctor does not perform completion.
- Optional `/models` probing is loopback-only and may fail without failing the
release boundary.
- Read-only skill specs are static and inspectable only.
- Skill execution remains closed.
- v0.41.6 user-test readiness remains false.

## Withdrawal Conditions

Withdraw or roll back v0.41.3 if any of these occur:

- `run` or `ask` is implemented
- user prompt is sent to a provider
- completion request is sent
- `/chat/completions`, `/completions`, or `/responses` is called
- AgentLoop starts
- SkillLoader or SkillExecutor is implemented
- read-only skills execute
- runtime trace is emitted as user runtime
- shell/subprocess opens
- remote network endpoint is accessed
- credential values are read or printed

- child session is created
- files outside explicit profile/session home are mutated
- existing user files are overwritten
- user-test release readiness is claimed
