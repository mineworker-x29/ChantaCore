# v0.25.5 Internal Provider Invocation Orchestrator

Korean name: 내부 Provider 호출 Orchestrator

Track: Bounded General Agent Surface & Internal Tool Routing

Previous: v0.25.4 Tool Routing Plan & Provider Selection

Next: v0.25.6 Response Assembly & Evidence Binder

## Version Definition

v0.25.5 implements internal provider invocation orchestration for the public
ChantaCore agent surface. It consumes v0.25.4 route plans and invokes only
registered internal provider boundaries by reference. It creates invocation
plans, invocation steps, preconditions, boundary checks, dispatch records,
result references, traces, result bundles, and evidence seeds for v0.25.6.

v0.25.x remains Bounded General Agent Surface & Internal Tool Routing.

## Relationships

v0.25.4 creates route plans and provider selections. v0.25.5 requires those
plans before any provider invocation can be orchestrated.

v0.25.3 safety gate must allow routing before v0.25.4 can create an executable
provider route.

v0.25.2 intent classification and task framing provide the deterministic task
shape that v0.25.3 and v0.25.4 consume.

v0.25.1 turn envelope and v0.25.0 agent surface contract remain prerequisites.

v0.24.9 internal provider consolidation and v0.24 provider capability surfaces
remain the provider-owned boundary source.

## Direction

ChantaCore directly implements its OCEL-native agent spine. OpenCode,
OpenClaw, and Hermes remain reference architectures only. They do not become
runtime dependencies, adapters, bridges, fork-specific integrations, or copied
runtime logic in v0.25.5.

Workspace workbench remains deferred to v0.26. Memory candidate and continuity
remain deferred to v0.27. External provider and agent adapters remain deferred
to v0.29+. External agent dominion remains deferred to v0.30+.

## Principles

- Provider invocation is not arbitrary tool execution.
- Provider invocation must respect provider-owned policy boundaries.
- Route plan is required before provider invocation.
- Safety gate is required before route plan.
- Provider invocation result is not final answer.
- Evidence seed is not response assembly.
- Local runtime execution must remain inside the v0.24 gated provider boundary.
- Response assembly is deferred to v0.25.6.
- Ask/REPL is deferred to v0.25.7.

## Non-goals

v0.25.5 does not implement final response assembly, ask, REPL, memory
continuity, memory promotion, persistent memory writes, persona mutation,
external provider adapters, external agent adapters, Schumpeter split logic,
company-specific wrappers, network fetch, credential storage/output, raw
provider output inline rendering, or LLM-based orchestration judgment.

It also does not directly read files, search repositories, inspect process
state, run local commands, call process APIs directly, rerun commands, or write
repository state. Local runtime flow can only be represented through the v0.24
sequence:

1. local_runtime_command_candidate_provider
2. local_runtime_static_safety_preflight_provider
3. gated_local_runtime_execution_provider
4. local_runtime_output_failure_explanation_provider

## Service Design

AgentProviderInvocationPrerequisiteSourceService loads the agent surface
contract, route report, route plan, provider selection, safety gate reference,
internal provider registry, capability surface, and skill registry view.

AgentProviderInvocationPlanService maps v0.25.4 route steps to v0.25.5
invocation steps. Local runtime execution routes are expanded into the v0.24
candidate, safety/preflight, gated execution, and output/failure sequence.

AgentProviderInvocationPreconditionService checks route plan validity,
allow-route gate presence, provider registration, capability availability,
provider boundary availability, input reference availability, local runtime gate
requirements, and no-direct-access requirements.

AgentProviderInvocationBoundaryCheckService checks registry, policy, effects,
scope, local runtime gate, output redaction, evidence references, external
adapter absence, and direct execution absence.

AgentProviderDispatchService records dispatch through the internal provider
interface only. It does not inline raw provider output and does not directly
perform provider work outside provider-owned boundaries.

AgentProviderInvocationResultService creates sanitized result refs. The result
ref, bundle, and evidence seed are inputs for v0.25.6.

## Status Rules

passed means a valid route plan required provider invocation, provider
preconditions and boundary checks passed, internal provider dispatch records and
result references were created, trace/bundle/evidence seed were created, and no
forbidden direct execution or response assembly occurred.

warning means no provider invocation was required, an optional/partial result
was encountered, or response assembly is needed with limitations.

failed means a required provider result, bundle, or evidence seed could not be
created without a dangerous bypass.

blocked means the route plan, allow-route gate, provider boundary, or safety
boundary is missing, or a forbidden direct/external/response/memory/persona/LLM
condition is detected.

## Acceptance Criteria

- AgentProviderInvocationReport can be created.
- AgentProviderInvocationPlan can be created from a valid route plan.
- AgentProviderInvocationStep can be created from route steps.
- Internal providers are represented through provider-owned boundaries.
- Provider result refs are captured.
- Provider invocation trace is recorded.
- Provider result bundle is created.
- Evidence seed is created for v0.25.6.
- Local runtime route preserves the v0.24 sequence.
- No direct subprocess occurs.
- No direct file access outside provider occurs.
- No direct local command execution occurs.
- No external provider or agent adapter is implemented.
- No final response assembly occurs.
- No ask/REPL execution occurs.
- No memory promotion or persistent memory write occurs.
- No persona mutation occurs.
- No OpenCode/OpenClaw/Hermes runtime dependency is added.
- No Schumpeter split is introduced.
- No credential value is stored or output.
- No raw provider output or raw secret is inlined.
- No LLM judge is used.

## Withdrawal Conditions

Withdraw v0.25.5 readiness if provider dispatch bypasses the registered
provider boundary, direct file/search/process/command access appears in the
agent surface orchestrator, final response assembly appears in v0.25.5, raw
provider output is inlined, external provider or agent adapters are introduced,
memory/persona mutation appears, or LLM judgment is used for orchestration.

## Validity Horizon

Valid until v0.25.6 response assembly begins or provider invocation policy
changes.
