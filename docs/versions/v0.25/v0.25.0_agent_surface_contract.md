## Definition

v0.25.0 introduces the contract-only Agent Surface foundation. It defines the
shape of agent surface modes, request envelopes, response envelopes, outcomes,
permissions, effects, routing boundaries, evidence rules, observability, safety
boundaries, roadmap boundaries, and reference architecture policy.

v0.25.x is bounded general agent surface work. It is not a full
OpenCode/Hermes/OpenClaw-class agent platform.

## Objective

The release prepares a future general agent surface that can receive a user
request, structure a request envelope, define a response envelope, define
allowed outcomes, prepare future intent classification, prepare future safety
gating, prepare future internal provider routing, prepare evidence-bound
response assembly, and prepare future OCEL-native agent trace visibility.

## Relationship To Earlier Releases

v0.24.9 consolidated the Internal Provider / Local Runtime Provider Foundation
v1 and produced the substrate readiness signal for v0.25 entry. v0.23.9 remains
the Internal Dominion Foundation v1. v0.22.9, v0.21.9, and v0.20.9 remain their
respective safety, introspection, and self-awareness foundations.

If any prerequisite foundation is unavailable, v0.25.0 should report that as a
finding rather than fake readiness.

## Policy

- Agent surface is not arbitrary autonomous execution.
- User request is not automatic permission.
- Ask/REPL surface is not unrestricted shell.
- Tool routing is not arbitrary tool execution.
- No-action is a valid agent outcome.
- Clarification is safer than unsafe routing.
- Provider result must be evidence-bound.
- Local runtime execution must go through the v0.24.7 gate.
- Memory continuity is deferred to v0.27.
- External agent domination is deferred to v0.30+.
- ChantaCore directly implements its own OCEL-native agent spine.

## Long-Term Direction

ChantaCore directly implements its own agent surface. OpenCode, Hermes, and
OpenClaw may inform architecture, but they are references only in v0.25.0.

- OpenCode patterns may inform session, tool registry, permission, provider
  abstraction, and CLI/TUI style decisions.
- Hermes patterns may inform future identity isolation, profile isolation,
  delegation, and bounded memory design.
- OpenClaw patterns may inform future personal assistant UX, skill ecosystem
  lessons, and external-agent risk handling.
- OpenCode, OpenClaw, and Hermes are not runtime dependencies.
- External provider and agent adapters are deferred to v0.29+.
- External Agent Dominion Bridge is deferred to v0.30+.

## Policies

The outcome policy permits `answered`, `routed_future`, `needs_more_input`,
`clarification_requested`, `no_action`, `blocked`, `deferred`, and `failed`.
No-action, clarification, and blocked are valid agent outcomes.

The permission policy is deny-by-default. A user request is not permission.
Provider invocation, route planning, and local runtime execution remain future
gated work.

The effect policy allows only `read_only_observation`, `state_candidate_created`,
and `agent_surface_contract_declared` in v0.25.0. Executable agent effects are
future effects, not active effects.

The routing boundary disables route planning, provider invocation, local runtime
execution, ask, and REPL in v0.25.0.

The evidence policy requires fact, provider-result, inference, and uncertainty
separation. Raw provider output dumps, raw secret output, and private full path
output are forbidden.

## OCEL Mapping

Object mappings include the agent surface contract, mode descriptors, request
and response envelope contracts, outcome/permission/effect/routing/evidence
policies, observability contract, safety boundary, roadmap boundary, reference
architecture policy, findings, and report.

Event mappings include contract request, mode declaration, envelope contract
creation, policy creation, boundary creation, reference architecture creation,
contract report creation, and blocked contract events.

Effects are limited to read-only observation, state candidate creation, and
agent surface contract declaration.

## PIG/OCPX

The PIG report records the agent surface principles, future direction,
reference architecture policy, and disabled safety boundary. The OCPX projection
uses `state=agent_surface_contract_declared` and projects contract, mode,
permission, effect, routing, evidence, roadmap, and v0.25 readiness states.

## Acceptance Criteria

- Agent surface contract can be created.
- Request and response envelope contracts are defined.
- Outcome policy includes no-action, needs-more-input, and blocked outcomes.
- Permission policy states user request is not permission.
- Effect policy is contract-only.
- Routing boundary disables actual routing and provider invocation.
- Evidence policy requires fact/provider-result/inference/uncertainty separation.
- Observability contract requires OCEL/PIG/OCPX visibility.
- Reference architecture policy documents direct implementation strategy.
- OpenCode/OpenClaw/Hermes are reference patterns only.
- `ready_for_v0_25_1` can be true.
- `ready_for_v0_26` remains false.
- No ask/REPL execution, provider invocation, local command execution, external
  adapter, memory continuity, workspace workbench, Schumpeter split, credential
  output, or LLM judge is introduced.

## Withdrawal Conditions

This version judgment should be withdrawn if v0.25.0 introduces ask/REPL
execution, provider invocation, tool route execution, local command execution,
memory continuity, workspace workbench, external adapters, OpenCode/OpenClaw/
Hermes runtime dependencies, Schumpeter split, credential exposure, raw secret
output, or an LLM judge.

## Validity Horizon

Valid until v0.25.1 turn envelope implementation begins or agent surface policy
changes.
