## Release Definition

v0.37.5 creates repair suggestion metadata only.

Repair suggestion metadata records what type of future repair may be worth considering, what evidence supports that consideration, what scope may be relevant, what risk remains, whether do-nothing remains valid, and what human review is required. It does not generate executable changes.

Repair scope metadata describes a possible area of concern, such as implementation, import path, configuration, allowlist policy, sandbox environment, test, documentation, or no scope. Scope is not file edit permission.

Repair risk assessment records confusion risks around repair patch proposals, repair execution, automatic repair, retry loops, dependency install, live workspace write, patch application, external agents, Dominion, and production certification.

Do-nothing repair comparison preserves the do-nothing alternative as a non-scored comparator. It is not a final scorecard.

Human review requirement records that later review is required. It is not human approval.

## Boundary Distinctions

- Repair suggestion metadata is not a repair patch.
- Repair suggestion metadata is not a patch proposal.
- Future repair proposal input is not repair proposal generation.
- Suggested action is not execution.
- Human review requirement is not approval.
- Do-nothing comparison is not a final scorecard.
- Vera-Codex trial input is not Vera-Codex trial execution.
- Repair scope metadata is not code edit permission.
- Repair risk metadata is not approval to repair.
- Missing dependency suggestion does not allow dependency install.
- Timeout suggestion does not allow immediate retry.

## Models

`SandboxRepairSuggestionPolicy` allows suggestion metadata, scope metadata, risk assessment, human review requirement, and future repair proposal input metadata. It blocks test execution, subprocess, shell, dependency install, network access, repair patch proposal generation, repair diff generation, code hunk generation, automatic repair, repair execution, Vera-Codex trial execution, cold performance evaluation, external agent execution, and Dominion runtime.

`SandboxRepairSuggestionInput` is a suggestion request, not a repair execution request. It records prohibited runtime actions including patch proposal, diff generation, code edit, apply, test execution, subprocess, shell, install, network, repair execution, external agent, and Dominion.

`SandboxRepairEvidenceRef` is bounded evidence metadata.

`SandboxRepairScope` is not edit permission. `edit_allowed` remains false.

`SandboxRepairRiskAssessment` requires human review for high, critical, or blocked risks.

`SandboxRepairDoNothingComparison` keeps the do-nothing alternative present. `scoring_performed` remains false.

`SandboxRepairHumanReviewRequirement` is not approval. `human_approval_present` remains false.

`SandboxRepairSuggestedAction` is metadata only. It does not execute, generate patches, edit code, apply patches, run tests, or install dependencies.

`SandboxRepairSuggestionEnvelope` may be eligible for future repair proposal input, future Vera-Codex trial input, and future cold evaluation input. It never creates repair patch proposals, unified diffs, code hunks, repair execution, test reruns, dependency installs, production certification, or execution readiness.

## Acceptance Criteria

- Repair suggestion metadata can be created from supplied v0.37.4 feedback report metadata.
- Evidence refs, scope, risk assessment, do-nothing comparison, and human review requirement are present.
- Missing dependency suggests inspection without install.
- Timeout suggests inspection without immediate retry.
- Weak or insufficient evidence produces do-nothing/review/inconclusive metadata.
- No suggestion produces patch proposals, diffs, code hunks, edits, applies, test runs, dependency installs, or repair execution.

## Withdrawal Conditions

Withdraw v0.37.5 readiness if any helper:

- Creates patch proposals, unified diffs, or code hunks.
- Edits files or applies patches.
- Calls apply_patch or git apply at runtime.
- Executes tests, subprocesses, shell, command strings, package managers, dependency install, or network access.
- Performs automatic repair or repair execution.
- Starts retry loops or multi-cycle loops.
- Treats human review requirement as approval.
- Treats do-nothing comparison as final scorecard.
- Runs Vera-Codex trial or cold performance evaluation.
- Invokes external coding agents or Dominion runtime.
- Sets unsafe readiness flags true.
