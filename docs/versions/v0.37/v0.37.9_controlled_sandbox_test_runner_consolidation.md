# v0.37.9 Controlled Sandbox Test Runner & Vera-Codex Agent Evaluation Consolidation

Foundation release name: Controlled Sandbox Test Runner & Vera-Codex Agent Evaluation v1.

Parent track: Controlled Sandbox Test Runner & Vera-Codex Agent Evaluation.

v0.37.9 is consolidation only. It records the release-readiness, coverage, audit, boundary, risk, gap, manifest, and v0.38 handoff state of v0.37.0 through v0.37.8. It does not add new runtime authority.

## Relationship To Prior Gates

v0.37.9 inherits the v0.30, v0.31, v0.32, v0.33, v0.34, v0.35, v0.36, and v0.37 gates. It also preserves the v0.36.9 Human-approved Patch Apply Sandbox v1 distinction: sandbox apply metadata and human-approved sandbox workflows are not live workspace apply authority.

The consolidated v0.37 stages are:

- v0.37.0 Controlled Sandbox Test Runner Boundary Foundation.
- v0.37.1 Allowlisted Test Command Policy & Test Invocation Contract.
- v0.37.2 Sandbox Test Execution Engine.
- v0.37.3 Test Result Envelope & Output Classifier.
- v0.37.4 Test Feedback Report & Failure Diagnosis.
- v0.37.5 Repair Suggestion Metadata, No Auto-Repair.
- v0.37.6 Vera-Codex One-shot Agent Operation Trial.
- v0.37.7 Cold Agent Performance Evaluation & Scorecard.
- v0.37.8 CLI Test Runner & Agent Evaluation Surface.

## What v0.37.x Accomplished

v0.37.x opened a bounded Controlled Sandbox Test Runner & Vera-Codex Agent Evaluation v1 foundation:

- Controlled sandbox test runner boundary.
- Allowlisted test command policy and invocation contract.
- Structured command specs.
- Sandbox cwd policy.
- Timeout, resource, output, and environment contracts.
- Controlled sandbox test execution through v0.37.2 helper boundaries only.
- Controlled subprocess only with structured argv, shell false, validated sandbox cwd, timeout, bounded output, and minimal env.
- Test result envelope and output classifier.
- Process exit and failure class classification.
- Evidence snippet extraction and redaction.
- Test feedback report and failure diagnosis metadata.
- Evidence strength assessment, root-cause hypothesis metadata, and suggested next action metadata.
- Do-nothing alternative signal.
- Repair suggestion metadata, repair scope metadata, repair risk assessment, and human-review requirement.
- Future repair proposal input metadata without repair proposal generation.
- Vera-Codex one-shot trial metadata, evidence bundle, bounded decision trace without chain-of-thought, stop reason, and mandatory human handoff.
- Cold agent performance evaluation, evidence-grounded scorecard, boundary compliance assessment, do-nothing comparison, failure condition assessment, and PASS / PASS_WITH_WARNINGS / INCONCLUSIVE / FAIL / BLOCKED verdict metadata.
- CLI Test Runner & Agent Evaluation Surface with bounded dispatch to v0.37 helpers.
- Explicit denial of shell, raw pytest, npm, pip, install, network, live, repair, external-agent, and Dominion commands.
- Digestion-first and Dominion-fallback metadata.
- External-agent-control pattern awareness as blocked or future-gated metadata.

## What v0.37.x Did Not Accomplish

v0.37.x does not enable arbitrary shell. It does not enable direct or uncontrolled subprocess. It does not enable dependency install, network access, live workspace write, live code edit, patch application, apply_patch, git apply, repair patch proposal generation, repair diff generation, code hunk generation, automatic repair, repair execution, retry loop, or multi-cycle agentic loop.

v0.37.x does not enable model provider invocation, OpenAI/Anthropic/Claude API calls, Codex CLI invocation, Claude Code invocation, OpenCode/Hermes/OpenClaw execution, external coding agent orchestration, tool execution, general agent runtime, independent autonomous agent runtime, Dominion runtime, credential/env/secret reads, persistent trace file/database/log writes, OCEL file write, JSONL trace write, UI runtime, external control, authority grant, D4-D9, or production certification.

## Stage Summaries

Boundary foundation: v0.37.0 defines the test runner boundary and confirms that test runner readiness is not general execution readiness.

Allowlisted command policy: v0.37.1 defines allowed test command metadata and invocation contracts. It does not authorize arbitrary commands.

Sandbox test execution engine: v0.37.2 owns the only controlled test subprocess helper boundary in v0.37.x. v0.37.9 does not add another subprocess helper.

Test result envelope: v0.37.3 wraps execution output into an evidence envelope. Envelope readiness is not test success.

Test feedback and failure diagnosis: v0.37.4 creates feedback and diagnosis metadata. Diagnosis readiness is not repair permission.

Repair suggestion metadata: v0.37.5 creates suggestion metadata only. Repair suggestion readiness is not repair patch proposal readiness.

Vera-Codex one-shot trial: v0.37.6 creates one-shot trial metadata. Vera one-shot readiness is not model provider invocation or autonomous runtime.

Cold agent performance evaluation: v0.37.7 creates evidence-grounded scorecard metadata. Cold scorecard readiness is not production certification.

CLI Test Runner & Agent Evaluation Surface: v0.37.8 exposes bounded CLI-facing handlers. CLI surface readiness is not shell readiness.

## V038 Handoff

Recommended v0.38 track: Bounded Repair Proposal Loop.

Recommended first step: Bounded Repair Proposal Boundary Foundation.

v0.38 should begin with repair proposal boundary foundation, repair proposal evidence contract, repair candidate and proposed diff metadata, code hunk proposal metadata, no apply by default, no live workspace write, no auto-repair, human approval before any repair apply, sandbox-only repair proposal validation, do-nothing alternative comparison, and cold evaluation feedback integration.

v0.38 should not immediately open automatic repair. v0.38 should not immediately open live apply. v0.38 should not immediately open apply_patch or git apply. v0.38 should first create bounded repair proposal artifacts, not execute them.

## Hard Prohibitions

v0.37.9 helpers must remain pure metadata/report/handoff helpers. They must not write files, run tests, invoke subprocess, shell out, install dependencies, use network, call providers, invoke external agents, create patches/diffs/hunks, apply patches, perform automatic repair, implement multi-cycle loops, implement Dominion runtime, read credentials/secrets/env, persist traces, mutate registries or memory, implement UI runtime, grant authority, or set unsafe readiness flags true.

## Acceptance Criteria

Acceptance requires the v0.37.9 models, helpers, tests, and documentation to distinguish bounded foundation readiness from runtime readiness. Controlled sandbox test execution may be ready only through v0.37.2 helper boundaries. Repair suggestion metadata may be ready, but repair proposal/diff/hunk/repair execution must remain false. Vera one-shot readiness may be true, but provider/external-agent/autonomous runtime must remain false. Cold scorecard readiness may be true, but production certification must remain false.

## Withdrawal Conditions

Withdraw v0.37.9 consolidation if arbitrary shell, direct/uncontrolled subprocess, dependency install, network access, live workspace write, patch application, repair proposal generation, automatic repair, model provider invocation, external agent execution, Dominion runtime, persistent trace write, authority grant, production certification, or ready_for_execution is introduced.
