# v0.37.2 Sandbox Test Execution Engine

Parent track: Controlled Sandbox Test Runner & Vera-Codex Agent Evaluation.

v0.37.2 builds on v0.37.1 Allowlisted Test Command Policy & Test Invocation Contract, v0.37.0 Controlled Sandbox Test Runner Boundary Foundation, and v0.36.9 Human-approved Patch Apply Sandbox v1. It introduces controlled sandbox test execution only. It does not introduce arbitrary shell, uncontrolled subprocess authority, dependency install, live workspace write, repair authority, Vera-Codex trial execution, cold evaluation execution, external agent execution, Dominion runtime, or production certification.

## Definition

Controlled sandbox test execution means one allowlisted structured argv subprocess, under a validated sandbox cwd, with `shell=False`, configured timeout, bounded stdout/stderr capture, minimal sanitized environment, dependency install blocked, network blocked by policy, and no repair or retry loop.

Controlled sandbox test subprocess is not arbitrary subprocess. Allowlisted argv is not shell command text. Sandbox cwd is not live workspace cwd. A process exit code is not final test result classification; v0.37.3 owns the test result envelope and classifier. Successful process exit is not production certification. Test failure does not allow repair. Timeout does not allow retry. Missing dependency does not allow dependency install.

## Network Policy Limitation

v0.37.2 records `network_policy_blocked=True`, `network_access_allowed=False`, and `network_hard_isolation_certified=False`. It does not claim OS-level network hard isolation. That claim can only be made by a later release that implements and proves a hard isolation mechanism.

`ControlledTestSubprocessPolicy` requires structured argv, `shell=False`, timeout, sandbox cwd, minimal environment, and bounded output. It blocks shell=True, command strings, package managers, external agents, and network access.

`SandboxTestExecutionInput` is a controlled sandbox test request. It includes sandbox cwd, timeout, source refs, and prohibited runtime actions.

`ControlledTestCommandLine` stores structured argv only. It must be allowlisted, `shell=False`, and contain no blocked argument.

`ControlledTestEnvironment` stores minimal sanitized environment metadata. It does not inherit credential or network environment.

`ControlledTestCwdValidation` validates sandbox cwd and blocks live workspace cwd, reference cwd, and outside-sandbox cwd.

`ControlledTestSubprocessInvocation` combines command line, cwd validation, environment, timeout, output limit, and subprocess kind. It is controlled sandbox subprocess only.

`ControlledTestSubprocessResult` stores process-level status, exit kind, return code, timeout flag, duration, bounded stdout/stderr, truncation/redaction flags, and safety booleans. It is not production certification.

`SandboxTestOutputCapture` stores bounded/redacted output metadata. Secret-like content forces redaction.

`SandboxTestExecutionResult`, `SandboxTestExecutionValidationReport`, `SandboxTestExecutionReport`, and `SandboxTestExecutionRunPreview` are process result and reporting metadata. v0.37.3 handles result envelope classification.

`SandboxTestExecutionNoUnsafeCommandGuarantee` records no arbitrary shell, shell=True, uncontrolled subprocess, unallowlisted command, install, network permission, live write, patch application, repair, retry, multi-cycle loop, Vera-Codex trial, cold evaluation, provider, external agent, Dominion, persistent trace, UI, or authority grant.

## Hard Prohibitions

v0.37.2 may execute allowlisted sandbox test subprocesses only. It must use `shell=False`, structured argv, sandbox cwd, timeout, bounded stdout/stderr, and minimal environment.

v0.37.2 does not execute arbitrary shell. It does not run unallowlisted commands. It does not run npm, pnpm, yarn, pip, poetry, go, cargo, make, package scripts, bash, powershell, cmd, git, curl, wget, ssh, or external agent commands. It does not install dependencies. It does not allow network access by policy. It does not write live workspace files, edit live code, apply patches, call apply_patch, or call git apply.

v0.37.2 does not perform automatic repair, repair loops, retry loops, or multi-cycle loops. It does not run Vera-Codex trial. It does not run cold performance evaluation. It does not invoke model providers, external coding agents, Claude Code, Codex CLI, OpenCode, Hermes, OpenClaw, or Dominion runtime. It does not read credentials, secrets, tokens, API keys, or `.env` files. It does not persist traces, write OCEL/JSONL files, implement UI runtime, mutate registries or memory, grant DominionAuthority, or grant D4-D9.

`ready_for_sandbox_test_execution_engine`, `ready_for_controlled_test_subprocess`, `ready_for_allowlisted_test_execution`, `ready_for_pytest_module_execution`, `ready_for_unittest_module_execution`, `ready_for_bounded_test_output_capture`, `ready_for_test_timeout_enforcement`, `ready_for_minimal_test_environment`, and `ready_for_future_test_result_envelope_input` may be true only in the controlled sandbox-test sense.

## Digestion And Dominion

Digestion-first policy remains applied. Dominion fallback remains future-gated and blocked. v0.37.2 records Dominion runtime, external agent execution, infinite loops, recursive self-invocation, automatic repair loops, independent autonomous agent runtime, and multi-cycle loops as blocked metadata only. Vera-Codex evaluation receives future input metadata only and is not executed.

Reference paths under OpenCode, Hermes, or OpenClaw must not be used as sandbox test cwd. Reference code is not executed, imported, installed, tested, or used as a live adapter.

## Acceptance Criteria

- Controlled subprocess appears only in approved helpers.
- Commands are structured argv and `shell=False`.
- CWD is validated as sandbox cwd before execution.
- Timeout is enforced.
- stdout/stderr are bounded and redacted.
- Minimal environment is used.
- Unallowlisted, shell, install, network, external agent, Dominion, repair, multi-cycle, live cwd, reference cwd, and outside cwd requests are blocked.
- Process zero, nonzero, timeout, blocked, and failed-safe states can be represented.
- Results do not certify production and do not grant repair, retry, or install permission.

## Withdrawal Conditions

Withdraw v0.37.2 readiness if subprocess use appears outside approved helpers, `shell=True` is introduced, command strings are passed to a shell, package managers or external agents can execute, live workspace/reference/outside cwd can run, timeout is optional, output is unbounded, credential environment is inherited, network permission is granted, hard network isolation is claimed without proof, repair/retry loops are added, Vera-Codex or cold evaluation executes, traces persist, or any unsafe readiness flag becomes true.
