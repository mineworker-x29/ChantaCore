## Release Definition

v0.37.4 creates feedback and diagnosis metadata only.

A test feedback report is a bounded metadata report derived from supplied result evidence. It records observations, evidence strength, root-cause hypotheses, suggested next-action metadata, a do-nothing alternative signal, and future handoff input metadata.

A failure diagnosis is a bounded classification of supplied failure evidence. It is not repair execution, not proof of root cause, not production certification, and not permission to retry, install, repair, invoke an agent, or run evaluation.

A root-cause hypothesis is an evidence-bounded explanation candidate. It must carry evidence strength and confidence. It is not a proven root cause.

A do-nothing alternative signal records whether doing nothing remains preferred, competitive, required, inferior, or not evaluable yet. It is not a final scorecard.

## Boundary Distinctions

- Test feedback report is not test execution.
- Failure diagnosis is not repair execution.
- Root-cause hypothesis is not proof.
- Suggested next action is not action execution.
- Future repair input is not a repair suggestion artifact.
- Repair suggestion artifact belongs to v0.37.5.
- Do-nothing signal is not a final scorecard.
- Vera-Codex trial input is not Vera-Codex trial execution.
- Cold evaluation input is not cold performance evaluation execution.
- Failed or inconclusive evidence must not be reported as success.
- Missing dependency does not grant install permission.
- Timeout does not grant retry permission.
- Suggested next actions are metadata only.

## Models

`SandboxTestFeedbackPolicy` allows feedback, diagnosis, root-cause hypothesis metadata, suggested next-action metadata, do-nothing signal metadata, future repair suggestion input, and future Vera-Codex trial input. It blocks test execution, subprocess, shell, dependency install, network access, repair patch proposal, automatic repair, Vera-Codex trial execution, cold performance evaluation, external agent execution, and Dominion runtime.

`SandboxTestFeedbackInput` is a feedback request, not an execution request. It references supplied result envelope, outcome, failure, execution, and invocation metadata.

`SandboxFailureObservation` records failure observations from supplied evidence. It is metadata only.

`SandboxEvidenceAssessment` records evidence strength, confidence, supporting evidence, contradictory evidence, and insufficient evidence.

`SandboxRootCauseHypothesis` records a hypothesis with evidence strength and confidence. It defaults to `proven=false` and `repair_allowed=false`.

`SandboxSuggestedNextAction` records metadata only. `executes_now`, `repair_allowed`, `test_rerun_allowed`, and `dependency_install_allowed` remain false.

`SandboxDoNothingAlternativeSignal` preserves the do-nothing alternative. `scoring_performed` remains false in v0.37.4.

`SandboxFailureDiagnosisReport` combines observations, evidence assessments, and hypotheses. It does not allow repair, retry, or dependency install.

`SandboxTestFeedbackReport` combines the diagnosis report, suggested actions, do-nothing signal, source refs, and future input eligibility. It never performs test execution, subprocess execution, repair, retry, install, production certification, or general execution.

## Acceptance Criteria

- Feedback and diagnosis models exist.
- Result envelope metadata can produce feedback input.
- Failure observations can be created from supplied result envelope metadata.
- Evidence strength is explicit.
- Root-cause hypotheses remain hypotheses, not proof.
- Suggested next actions are metadata only and execute nothing.
- Missing dependency suggests inspection without install.
- Timeout suggests review or future rerun gate, not retry now.
- Inconclusive evidence remains inconclusive and preserves do-nothing.
- Do-nothing signal is represented and is not a scorecard.
- Future repair, Vera-Codex, and cold evaluation input metadata is represented without execution.

## Withdrawal Conditions

Withdraw v0.37.4 readiness if any helper:

- Executes tests or subprocesses.
- Uses shell, command execution, package manager execution, dependency install, or network access.
- Reads credentials, secrets, token files, or environment credentials.
- Writes live workspace files, patches, traces, databases, JSONL, or OCEL files.
- Creates repair patch proposals.
- Performs automatic repair or retry.
- Treats failed or inconclusive evidence as success.
- Treats root-cause hypothesis as proof.
- Treats suggested next action as execution.
- Treats do-nothing signal as final scorecard.
- Runs Vera-Codex trial or cold performance evaluation.
- Invokes external coding agents or Dominion runtime.
- Sets unsafe readiness flags true.
