## Definitions

A controlled sandbox test runner is a future allowlisted, sandbox-cwd-only, no-shell, timeout-controlled, bounded-output, no-network test execution mechanism. v0.37.0 defines the boundary only.

An allowlisted test command policy is a future structured command contract. It is not shell execution and is not arbitrary command execution.

A controlled future subprocess is not arbitrary command execution. If introduced later, it must be allowlisted, sandbox cwd only, no shell, timeout-controlled, bounded output, no network, and no dependency install by default.

A test result envelope is future test metadata. It is not production certification.

Failure diagnosis is future feedback metadata. It is not automatic repair. Repair suggestion metadata is not repair execution.

Vera-Codex one-shot agent evaluation is a bounded operator/evaluator function: one explicit evaluation task, one sandbox evidence bundle, one cold assessment, no automatic repair, no retry loop, no self-directed multi-step autonomy, no external agent orchestration, no model self-approval, and mandatory human handoff. It is not autonomous runtime.

Cold performance evaluation is future evidence-based scorecard metadata. It is not self-praise.

Do-nothing alternative comparison is mandatory evidence discipline for future evaluation. It is not passive inaction and scoring remains future-gated.

## Boundary Policies

SandboxTestRunnerBoundaryPolicy allows boundary definition and future gates only. It blocks test execution, controlled test subprocess, pytest execution, npm test execution, unittest execution, shell, arbitrary command, dependency install, network access, live workspace write, patch application, automatic repair, multi-cycle loop, Vera-Codex trial execution, external agent execution, and Dominion runtime.

SandboxTestCommandBoundaryPolicy requires allowlist, structured command spec, no shell, sandbox cwd, timeout, no network, package scripts rejected by default, and dependency install rejected. allow_test_command_execution remains false in v0.37.0.

SandboxTestOutputBoundaryPolicy defines bounded and redacted future output requirements. Output capture remains false in v0.37.0.

VeraCodexEvaluationBoundaryPolicy requires test result evidence, validation report evidence, do-nothing comparison, cold evaluation, no self-praise without evidence, no failed-test-as-success claim, and human handoff. Trial execution, model provider invocation, autonomous runtime, external agent execution, and Dominion runtime remain false.

DoNothingAlternativeBoundaryPolicy requires do-nothing baseline, before/after comparison, risk delta comparison, test delta comparison, and rejection of improvement claims without evidence. Scoring execution remains false in v0.37.0.

Allowed surfaces are design-stage only and executable_in_v0370 remains false. They do not run tests, shell, install dependencies, access network, invoke models, or invoke external agents.

Prohibited surfaces block test execution and runtime readiness.

SandboxTestRunnerBoundary combines the policies, allowed surfaces, prohibited surfaces, flags, status, readiness level, gaps, blocked reasons, evidence, and withdrawal conditions. It is not test execution.

Permission request, decision, denied action, and gate evaluation artifacts are metadata only. Decisions never allow test execution, shell, subprocess, install, network, repair, evaluation execution, external agents, or Dominion runtime.

SandboxTestRunnerRiskRegister, SandboxTestNoExecutionGuarantee, V037RoadmapOverview, and V0370ReadinessReport preserve the boundary. The no-execution guarantee has all no_* booleans true.
