# v0.37.1 Allowlisted Test Command Policy & Test Invocation Contract

Parent track: Controlled Sandbox Test Runner & Vera-Codex Agent Evaluation.

v0.37.1 builds on v0.37.0 Controlled Sandbox Test Runner Boundary Foundation and the v0.36.9 Human-approved Patch Apply Sandbox v1 handoff. It creates allowlisted test command policy and test invocation contract metadata only. It does not execute tests and does not implement the v0.37.2 controlled sandbox test execution engine.

## Scope

v0.37.1 defines:

- structured allowlisted test command specifications,
- future-execution-only test command allowlist metadata,
- denylist metadata for unsafe command names and arguments,
- sandbox cwd policy metadata,
- timeout and resource limit policy metadata,
- bounded output capture contract metadata,
- minimal environment contract metadata,
- test invocation request, decision, validation report, policy report, run preview, and readiness report artifacts,
- do-nothing baseline requirements for future Vera-Codex evaluation.

Allowlisted test command policy is not test execution. A test invocation contract is not subprocess execution. A structured command spec is not a shell command. Future sandbox test execution eligibility is not execution. Timeout, resource, output, and environment contracts are not runtime process control in v0.37.1. A missing dependency is not permission to install a dependency. Do-nothing baseline requirement is evidence discipline for later evaluation, not passive inaction.

## Relationship To Earlier Releases

v0.36.9 consolidated Human-approved Patch Apply Sandbox v1. It allowed sandbox-only patch apply readiness under a validated sandbox root while keeping live workspace write, live patch application, shell, test execution, external agents, Dominion runtime, and persistent trace writes blocked.

v0.37.0 defined the boundary foundation for the controlled sandbox test runner and Vera-Codex evaluation track. v0.37.1 narrows that boundary into structured command policy and invocation contracts that can be handed to v0.37.2, but still does not run anything.

## Taxonomy

Command kind taxonomy includes python pytest module/file/node metadata, python unittest metadata, python module invocation metadata, metadata-only no-op, and blocked shell, package script, dependency install, network, external agent, Dominion, repair, multi-cycle, and unknown command kinds.

Source taxonomy includes v0.37.0 boundary metadata, v0.36.9 consolidation, v0.36.8 CLI sandbox apply surface, v0.36.7 trace packet, v0.36.6 agentic run packet, v0.36.5 validation report, structured command request, CLI-like argv metadata, manual operator input, test fixture, and unknown.

Status and readiness taxonomy distinguishes command spec creation, allowlist creation, denylist creation, invocation contract creation, invocation validation, future sandbox test input eligibility, blocked, review-required, future-gated, no-op, and safe-failed states. These statuses are not subprocess or test execution readiness.

Decision and risk taxonomy allows command spec metadata, allowlist metadata, invocation contract metadata, and future sandbox test execution input metadata. It denies, blocks, requires review, or future-gates unsafe requests. Risks include arbitrary shell, uncontrolled subprocess, shell metacharacters, live workspace cwd, outside sandbox cwd, package scripts, dependency install, network access, unbounded output, missing timeout, missing allowlist, malformed specs, unsafe arguments, credential or secret exposure, external agent execution, Dominion runtime, automatic repair, and multi-cycle loops.

Executable, argument, cwd, timeout, resource, output, environment, network, and dependency postures are explicit metadata surfaces. Python executable metadata may be future-eligible. Shells, package managers, git, node/npm by default, external agents, live workspace roots, reference roots, external roots, unbounded timeout, unbounded resources, raw secret output, inherited credential environment, network access, and dependency install are blocked or future-gated.

## Artifacts

`SandboxTestCommandFlagSet` records readiness for policy, specs, allowlist, denylist, contracts, sandbox cwd policy, timeout policy, output capture contract, environment contract, and future sandbox test execution input. It may mark v0.37.2, v0.37.3, and v0.37.6 handoff readiness only as design-stage handoff. It keeps execution, test execution, controlled subprocess, pytest, unittest, npm test, shell, command, dependency install, network, live write, patch application, repair, Vera-Codex trial execution, cold evaluation execution, external agent, Dominion, provider, credential, persistent trace, UI, authority, and production flags false.

`SandboxTestAllowedExecutable` and `SandboxTestAllowedModule` describe future metadata. They are not executable now.

`SandboxTestCommandSpec` may be eligible for future sandbox execution input, but `executable_now`, shell, dependency install, network, and live workspace permissions remain false.

`SandboxTestCommandAllowlist` is future-execution-only and not executable now.

`SandboxTestCommandDenylist` blocks shell, install, network, external-agent, Dominion, repair, multi-cycle, and unknown command patterns.

`SandboxTestSandboxCwdPolicy` requires sandbox root posture and blocks live, reference, and external cwd.

`SandboxTestTimeoutPolicy` and `SandboxTestResourceLimitPolicy` require bounded timeout and resources and block unbounded operation.

`SandboxTestOutputCaptureContract` represents future bounded/redacted output capture only. It blocks raw secret output and does not capture output now.

`SandboxTestEnvironmentContract` requires minimal sandbox environment metadata and blocks inherited credential and network environment posture.

`SandboxTestInvocationContract` can mark an invocation as eligible for future sandbox test execution input. It does not allow execution, test execution, subprocess, shell, dependency install, or network. It preserves do-nothing baseline evidence as a future evaluation requirement.

`SandboxTestInvocationRequest`, `SandboxTestInvocationDecision`, `SandboxTestDeniedCommand`, `SandboxTestInvocationValidationReport`, `SandboxTestCommandPolicyReport`, and `SandboxTestInvocationRunPreview` are metadata artifacts. They do not execute or remediate anything.

`SandboxTestCommandNoExecutionGuarantee` records that no execution, tests, controlled subprocess, pytest, unittest, npm test, shell, command, install, network, live write, patch apply, automatic repair, multi-cycle loop, Vera-Codex trial, cold evaluation, provider call, external agent, Dominion runtime, persistent trace write, UI runtime, or authority grant is introduced.

`V0371ReadinessReport` records policy/contract readiness and design-stage handoff only. It is not runtime readiness.

## Flow

v0.37.0 boundary -> `SandboxTestCommandAllowlist` + `SandboxTestCommandDenylist` -> `SandboxTestCommandSpec` -> `SandboxTestInvocationContract` -> `SandboxTestInvocationRequest` -> `SandboxTestInvocationDecision` -> `V0371ReadinessReport` -> v0.37.2 Sandbox Test Execution Engine.

## Digestion And Dominion

Digestion-first policy remains applied. Dominion is fallback-only and future-gated. v0.37.1 records Dominion runtime, external agent execution, infinite agent loops, recursive self-invocation, automatic repair loops, independent autonomous agent runtime, and multi-cycle loops as blocked or future-gated metadata only. Vera-Codex evaluation remains bounded future metadata requiring test evidence, validation evidence, do-nothing comparison, and human handoff.

OpenCode, Hermes, OpenClaw, Claude Code, Codex CLI, external coding agents, and Dominion runtimes are not executed, imported, installed, invoked, or adapted as live runtime components.

## Safe Defaults

Safe defaults are:

- python module invocation metadata only,
- sandbox cwd only,
- bounded timeout required,
- bounded resource metadata required,
- bounded/redacted output contract future-gated,
- minimal sandbox environment future-gated,
- network blocked,
- dependency install blocked,
- missing dependency does not permit install,
- do-nothing baseline preserved for future evaluation,
- human handoff required for future Vera-Codex evaluation.

It does not install dependencies. It does not access network. It does not write live workspace files, edit live code, write sandbox files, apply patches, call apply_patch, or call git apply.

v0.37.1 does not run Vera-Codex trial, cold performance evaluation, automatic repair, repair loops, retry loops, or multi-cycle agentic loops. It does not invoke model providers, external coding agents, Claude Code, Codex CLI, OpenCode, Hermes, OpenClaw, or Dominion runtime. ## Acceptance Criteria

- Allowlisted command policy and invocation contract metadata exist.
- Command specs may be future-eligible but are not executable now.
- Safe pytest-style metadata can be classified as eligible for future sandbox test execution input.
- Shell, install, network, external-agent, Dominion, repair, multi-cycle, live/reference/external cwd, missing timeout, and unknown command requests are denied or blocked.
- Dependency install remains blocked even when a dependency is missing.
- Do-nothing baseline is preserved for future evaluation.
- No execution authority or production certification is introduced.

## Withdrawal Conditions

Withdraw v0.37.1 readiness if any helper executes tests, starts a subprocess, shells out, installs dependencies, accesses network, writes workspace files, applies patches, invokes providers or external agents, imports or executes reference code, runs Vera-Codex evaluation, performs automatic repair, starts a loop, persists traces, grants authority, or sets an unsafe readiness flag true.

## Handoff

v0.37.1 hands off to:

- v0.37.2 Sandbox Test Execution Engine for a future allowlisted sandbox-only controlled execution engine,
- v0.37.3 Test Result Envelope for future result representation,
- v0.37.6 Vera-Codex One-shot Agent Operation Trial for future bounded evaluation using test evidence, validation evidence, do-nothing comparison, and human handoff.

The handoff is design-stage only.
