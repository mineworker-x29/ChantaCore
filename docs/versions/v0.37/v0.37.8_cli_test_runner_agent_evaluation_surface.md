# v0.37.8 CLI Test Runner & Agent Evaluation Surface

Parent track: Controlled Sandbox Test Runner & Vera-Codex Agent Evaluation.

v0.37.8 creates bounded CLI-facing handler metadata and in-memory output only. It may expose safe parser/handler functions and may dispatch selected commands to existing v0.37 helper boundaries. It does not execute shell commands, does not use subprocess directly, does not pass argv to shell, does not run arbitrary pytest/npm/pip commands, does not install dependencies, does not access network, does not write live workspace files, does not edit live code, does not apply patches, does not call apply_patch, does not call git apply, does not generate repair patches/diffs/hunks, does not perform automatic repair, does not invoke model providers, does not invoke Codex CLI, Claude Code, or external agents, does not implement Dominion runtime, does not implement infinite agent loops, and does not persist trace files or database records.

- v0.37.6 Vera-Codex One-shot Trial supplies one-shot trial packet metadata. v0.37.8 may expose `vera-trial-run-once` through that helper boundary. Vera trial is not model provider invocation.

- v0.36.9 Human-approved Patch Apply Sandbox v1 remains apply-boundary context only. v0.37.8 does not apply patches.

## Definition

The CLI Test Runner & Agent Evaluation Surface is a bounded parser, policy decision, result, output, and report layer for the v0.37 track. CLI argv parsing is not OS command execution. A command specification is not an OS command. A dispatch target is static helper-boundary metadata, not dynamic import, eval, exec, shell, or provider execution.

Safe CLI outputs are bounded in-memory results. Rendering output does not write to disk.

Denied command is a safe outcome. Denial does not execute remediation and does not retry, repair, install, invoke providers, or call external agents.

## Safe Command Set

Safe commands are:

`test-run-help`, `test-run-status`, `test-invocation-preview`, `test-run-sandbox`, `test-result-summary`, `feedback-report`, `repair-suggestion-preview`, `vera-trial-preview`, `vera-trial-run-once`, `cold-scorecard`, `evaluation-bundle-preview`, and `no-op`.

`test-run-sandbox` may dispatch only through v0.37.2 helper boundaries after policy, invocation contract, sandbox root, allowlist, shell false, timeout/output/environment, no install, no network, and no live workspace path conditions are represented.

`vera-trial-run-once` may dispatch only through v0.37.6 helper boundaries. It remains one trial, one cycle, no provider call, no external agent call, no tool execution, no repair, no retry loop, and mandatory human handoff.

`cold-scorecard` may dispatch only through v0.37.7 helper boundaries and does not certify production readiness.

## Models

`CLITestRunnerFlagSet` records safe CLI readiness while keeping execution, shell, direct subprocess, uncontrolled subprocess, command execution, install, network, live write, patch application, apply_patch, git apply, repair, provider, tool, external agent, Dominion, persistent trace, UI, authority, and production flags false.

`CLITestRunnerSourceRef` references argv, parsed args, runtime context, sandbox root, invocation contract, test execution result, result envelope, feedback report, repair suggestion, Vera trial packet, cold evaluation report, fixture, or unknown sources. A source ref does not execute.

`CLITestRunnerArgumentSpec` bounds argument values and records prohibited patterns. Argument specs are not execution.

`CLITestRunnerCommandSpec` describes safe or denied commands. Dispatch target is internal helper metadata only, not dynamic execution.

`CLITestRunnerSurfacePolicy` allows safe commands and controlled helper dispatch while blocking direct subprocess, shell, arbitrary command, dependency install, network, live workspace write, patch application, code edit, apply_patch, git apply, repair proposal, repair execution, automatic repair, retry loop, multi-cycle loop, model provider invocation, tool execution, external agent execution, Claude Code, Codex CLI, Dominion runtime, persistent trace write, and UI runtime.

`CLITestRunnerSurface` is not shell and keeps `ready_for_execution` false.

`CLITestRunnerRuntimeContext` records whether required artifacts are present. It does not contain credentials and does not grant unsafe runtime permissions.

`CLITestRunnerInvocation` records parsed argv. The argv is never passed to subprocess or shell.

`CLITestRunnerInvocationDecision` records an allow or deny decision. Safe allow decisions are bounded helper dispatch decisions only.

`CLITestRunnerDeniedCommand` records a denied command as a safe outcome.

`CLITestRunnerCommandResult`, `CLITestRunnerRunOutput`, and `CLITestRunnerRunReport` return bounded/redacted in-memory outputs while keeping direct subprocess, shell, install, network, live write, repair, model invocation, external agent, production certification, and ready_for_execution false.

`CLITestRunnerNoUnsafeSideEffectGuarantee` records no unsafe side effects. `no_controlled_test_dispatch` may be false only when `test-run-sandbox` occurred through the v0.37.2 helper boundary.

`V0378ReadinessReport` may mark v0.37.9 handoff and CLI readiness true while keeping unsafe readiness false.

## Acceptance Criteria

- Unknown commands are denied by default.
- Shell-like/metacharacter args are denied.
- Raw pytest/npm/pip/install/network/live/repair/external-agent/Dominion commands are denied.
- Safe CLI commands return bounded in-memory outputs.
- `test-run-sandbox` dispatches only through v0.37.2 helper boundaries.
- `vera-trial-run-once` dispatches only through v0.37.6 helper boundaries and does not call providers.
- `cold-scorecard` dispatches only through v0.37.7 helper boundaries and does not certify production.
- `repair-suggestion-preview` does not create patch, diff, or hunk metadata.
- No direct subprocess, shell, install, network, live write, patch, repair, provider, external agent, Dominion, persistence, UI runtime, or authority grant is introduced.

## Withdrawal Conditions

Withdraw v0.37.8 readiness if CLI argv is passed to shell/subprocess, direct subprocess is added in the v0.37.8 module, arbitrary pytest/npm/pip commands are executed, dependency install or network is enabled, live workspace writes are performed, patch application or repair generation is introduced, provider or external agent calls are added, Dominion runtime is added, persistent trace writes are introduced, `ready_for_execution` becomes true, or production certification is claimed.

## Handoff

v0.37.8 hands off to v0.37.9 Controlled Sandbox Test Runner & Agent Evaluation Consolidation only. The handoff is design-stage metadata, not execution authority and not production certification.
