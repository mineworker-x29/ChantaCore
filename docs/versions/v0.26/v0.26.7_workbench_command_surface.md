# v0.26.7 Workbench Command Surface

## Version Definition

- Version: `v0.26.7`
- Name: Workbench Command Surface
- Korean name: Workbench Command Surface
- Track: Workspace Agent Workbench
- Previous: `v0.26.6` Run Dashboard / Session Monitor
- Next: `v0.26.8` Workbench Snapshot / OCEL Export

`v0.26.7` creates command-surface artifacts for inspection, candidate creation,
decision recording, boundary tracing, history, and audit visibility. It is not a
command executor.

## Objective

The objective is to make workbench command choices inspectable and approvable
without crossing into execution. The central artifact is the
`WorkbenchCommandCandidate`, accompanied by a do-nothing alternative, rationale,
evidence, risk summary, approval requirement, and non-executing boundary trace.

## Relationships

`v0.26.7` reads refs and summaries from `v0.26.6` dashboard/session monitor
reports, `v0.26.5` approval console records, `v0.26.4` evidence inspector
reports, `v0.26.3` provider browser reports, and `v0.26.2` trace explorer
reports when available.

It also references `v0.25.7` ask/REPL boundaries, `v0.25.5` provider invocation
contracts, and the `v0.24.7` gated local runtime boundary as future or external
boundaries only. It does not invoke those surfaces.

## Grok/Codex Feedback Absorption Rule

ChantaCore may absorb external agent-usability feedback only as OCEL-visible
guidance, candidate, rationale, approval, inspection, dashboard, session-view,
telemetry, and command-candidate surfaces.

Allowed absorption includes skill, action, route, provider, ask pipeline,
file-edit, snapshot request, do-nothing candidates, command rationale,
evidence refs, PIG guidance refs, safety findings, risk summaries, approval
requirements, human intervention refs, command boundary traces, command
decisions, history, and audit trails.

Forbidden absorption includes direct command execution, approval-token
execution, provider invocation, local command execution, file mutation, patch
application, autonomous loops, automatic rerun, automatic repair, PIG as memory,
PIG as persona or policy mutation, PIG as execution authority, provider boundary
bypass, external adapter runtime dependency, raw transcript persistence, and raw
provider output dumps.

## Core Principles

- Workbench command is not direct execution.
- Command candidate is not command execution.
- Skill candidate is not skill execution.
- Action candidate is not action execution.
- Route candidate is not route execution.
- Provider candidate is not provider invocation.
- File edit candidate is not file mutation.
- Ask pipeline candidate is not ask execution.
- Snapshot request candidate is not snapshot export.
- Approval-linked command is not approval execution.
- Do-nothing candidate is a first-class safe alternative.
- PIG guidance may inform candidates, but is not memory, policy mutation, or execution.

## Service Design

The service layer is refs-only by default:

- prerequisite source service loads workbench view state, command panel,
  dashboard, approval, evidence, provider, trace, skill registry, provider
  registry, and PIG refs when available.
- policy and type-policy services define allowed candidate types and forbidden
  executable types.
- candidate services create skill, action, route, provider, file edit, ask
  pipeline, snapshot request, and do-nothing candidates.
- rationale, evidence, risk, PIG, safety, approval, and boundary services attach
  review context to command candidates.
- decision, decision-record, envelope, result, history, and audit services
  record command-surface state without dispatch.
- report and finding services compute readiness and block on forbidden drift.

## Source View

`WorkbenchCommandSourceView` contains refs and summaries only. It may include
dashboard, approval, evidence, provider, trace, skill registry, provider
registry, approval candidate, PIG guidance, and user intent refs. It must not
include raw provider output, raw transcripts, or raw secrets.

## Candidate Types

`WorkbenchCommandTypePolicy` allows inspection, approval-candidate creation,
snapshot-request candidate creation, skill/action proposal, ask pipeline
candidate, provider invocation candidate, local runtime candidate, file edit
candidate, and do-nothing candidate types. Executable types remain forbidden in
this release.

## Evidence, Risk, PIG, Safety

Every command candidate carries an evidence bundle, risk summary, PIG guidance
view when available, safety finding view, approval requirement, and boundary
trace. Risky command candidates require approval requirements and do-nothing
alternatives.

## Boundary Trace And Envelope

`WorkbenchCommandBoundaryTrace` records the relevant future or external
boundaries, including ask surface, provider invocation orchestrator, gated local
runtime, workbench view-only boundary, and future snapshot/export. It records
that no boundary was bypassed and no execution was performed now.

`WorkbenchCommandExecutionEnvelope` is a non-executing boundary envelope. It may
record candidate-only, inspection-only, approved-ref-only, rejected, deferred,
or blocked status, but it never dispatches.

## Decisions, History, Audit

Command decisions support approve candidate, reject candidate, defer candidate,
request more evidence, request clarification, choose do nothing, and mark
inspection only. Decision records are OCEL-visible and do not trigger execution.
History entries and audit trails are sanitized and do not include raw secrets,
raw provider output, or raw transcripts.

## Status Rules

The report passes when command source, type policy, command candidates,
do-nothing candidates, boundary traces, decision records, history, and audit
trail are created without forbidden behavior.

It warns for missing optional refs or partial source coverage. It fails in
strict mode when mandatory source or candidate artifacts are missing. It blocks
on any execution, mutation, provider invocation, ask/repl execution, approval
execution, autonomous loop, memory mutation, external adapter, Schumpeter split,
raw material persistence, or LLM judge finding.

## OCEL Mapping

Object, event, relation, allowed effect, and forbidden effect constants are
registered in the command surface module. Allowed effects include read-only
observation, command surface creation, command candidate creation, do-nothing
candidate creation, skill/action/route candidate creation, boundary trace
creation, decision recording, history recording, audit trail creation, and state
candidate creation. Forbidden effects include command dispatch, provider
invocation, local execution, file mutation, patch application, ask/repl
execution, final response emission, rerun, retry, repair, autonomous loop,
approval execution, memory mutation, external adapters, PIG mutation/execution,
credential exposure, raw output, and Schumpeter split.

## PIG/OCPX Report

The PIG report has `version=v0.26.7`, layer
`workspace_agent_workbench`, and subject `workbench_command_surface`. Its
principles state that command candidates are not execution and that PIG guidance
is not memory, policy mutation, or execution.

The OCPX projection state is `workbench_command_surface_created`. Source read
models include dashboard, approval, evidence, provider, trace, skill registry,
provider registry, and PIG guidance states. Target read models include command
surface, command candidate, skill/action candidate, boundary trace, decision
record, audit trail, and V026 readiness states.

## Acceptance Criteria

- Command surface artifacts build.
- Do-nothing candidate exists.
- Risky candidates have evidence, risk, approval requirement, and boundary
  trace.
- Decision records and envelopes never execute.
- CLI commands show all forbidden execution/mutation/provider/memory/raw-output
  flags as false.
- `ready_for_v0_26_8` is computed.
- `ready_for_v0_27=false`.

## Withdrawal Conditions

Withdraw readiness if direct command execution, command dispatch, provider
invocation, local runtime execution, file mutation, patch application, ask/REPL
execution, final response emission, rerun, retry, repair, autonomous loops,
approval execution, memory mutation, external adapters, PIG memory/policy/
execution use, Schumpeter split, raw material persistence, credential exposure,
or LLM judge behavior is introduced.
