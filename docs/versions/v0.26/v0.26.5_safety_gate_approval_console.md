# v0.26.5 Safety Gate / Approval Console

Korean name: Safety Gate·Approval Console

Track: Workspace Agent Workbench

Previous: v0.26.4 Evidence / Report Inspector

Next: v0.26.6 Run Dashboard / Session Monitor

## Objective

v0.26.5 creates OCEL-visible Safety Gate / Approval Console view and
decision-record artifacts. It reads existing safety, evidence, provider, route,
trace, PIG guidance, action-candidate, risk, and human-intervention references.
It records approval, rejection, deferral, and manual-review decisions without
executing them.

## Relationships

v0.26.4 provides Evidence / Report Inspector references that can be attached to
approval candidates.

v0.26.3 provides Provider / Capability Browser references for provider boundary,
readiness, risk, and rationale context.

v0.25.3 provides deterministic safety gate reports and gate outcome envelopes.

## Feedback Absorption

Grok / Codex usability feedback may be absorbed only as OCEL-visible guidance,
candidate, rationale, approval, inspection, and telemetry surfaces.

Allowed in v0.26.5:

- approval candidate creation
- rejection and deferral decision records
- manual review records
- human intervention point records
- safety gate rationale views
- risk, evidence, policy, user intent, and PIG guidance reference attachments
- approval scope and expiry definitions

Forbidden in v0.26.5:

- automatic approval
- approval as execution
- approval token as command
- PIG as memory, persona mutation, policy mutation, or execution authority
- autonomous multi-step loops, automatic rerun, or automatic repair
- provider boundary bypass, command execution, file mutation, external adapter
  runtime dependency, raw transcript persistence, and raw provider output dump

## Principles

Approval Console is not execution.

Approval candidate is not command.

Approval token is not command token.

Approval decision is not provider invocation.

Approve / reject / defer must be OCEL-visible.

Approval scope is required.

Approval expiry is required.

User intent ref is required.

Policy ref is required.

Evidence refs are required.

Risk summary is required.

PIG guidance is not memory, policy mutation, or execution.

Human intervention point is not approval by itself.

## Service Design

WorkbenchApprovalConsolePrerequisiteSourceService loads existing workbench view
state, safety gate and approval console panel models, safety gate report,
gate outcome envelope, evidence inspector report, provider browser report,
trace explorer report, route plan references, provider selection references,
action candidate refs, PIG guidance refs, and human intervention refs. It only
loads refs and summaries.

WorkbenchSafetyApprovalPolicyService creates the v0.26.5 policy with execution,
adapter, memory, persona, raw-output, and PIG mutation flags disabled.

WorkbenchSafetyGateSourceViewService and WorkbenchSafetyGateViewService create
refs-only safety source and safety gate views.

WorkbenchSafetyRationaleConsoleViewService creates safety rationale console
items without mutating safety policy.

WorkbenchApprovalRequirementService, WorkbenchApprovalCandidateService,
WorkbenchApprovalCandidateEvidenceBundleService, WorkbenchActionRiskSummaryService,
WorkbenchApprovalScopeService, WorkbenchApprovalExpiryService, and
WorkbenchApprovalTokenService create the approval candidate package. The token
is a non-executable reference.

WorkbenchApprovalDecisionService and WorkbenchApprovalDecisionRecordService
record approve, reject, and defer decisions without execution.

WorkbenchManualReviewRecordService records manual review.

WorkbenchPIGGuidanceAttachmentService attaches PIG refs without memory, policy,
or execution effects.

WorkbenchApprovalHumanInterventionPointService creates intervention points.

WorkbenchApprovalAuditTrailService creates a refs-only audit trail.

WorkbenchApprovalConsoleReportService creates the final report and PIG/OCPX
projection.

## Status Rules

Passed means the safety gate view, rationale console, approval console,
approval candidates, mandatory approval requirements, approval records, manual
review records, and audit trail can be created without introducing execution,
mutation, external adapters, raw output, or LLM judging.

Warning means optional refs are missing, no approval candidate is needed, or a
candidate remains safely incomplete.

Failed means required sources or mandatory approval refs are missing under
strict mode.

Blocked means approval execution, token execution, automatic approval, command
execution, provider invocation, route or stage rerun, ask execution, final
response emission, local command execution, direct file/subprocess access,
command rerun, automatic repair, PIG memory/policy/execution treatment,
memory/persona mutation, external/vendor adapter implementation, raw
provider/transcript/secret output, Schumpeter split, reference runtime
dependency, or LLM judge use is detected.

## CLI Commands

- `python -m chantacore.cli workbench safety view`
- `python -m chantacore.cli workbench safety rationale`
- `python -m chantacore.cli workbench approval console`
- `python -m chantacore.cli workbench approval candidates`
- `python -m chantacore.cli workbench approval requirements`
- `python -m chantacore.cli workbench approval scope`
- `python -m chantacore.cli workbench approval expiry`
- `python -m chantacore.cli workbench approval decide --candidate-id <id> --decision approve|reject|defer`
- `python -m chantacore.cli workbench approval audit`
- `python -m chantacore.cli workbench approval report --report-id <id>`

Decision commands create records only. They do not execute approved actions,
invoke providers, run commands, or mutate files.

tokens become command tokens, automatic approval appears, command/provider/route
execution is added, mandatory approval refs are bypassed, PIG guidance mutates
memory or policy, raw material is emitted, or v0.26.5 starts implementing
Command Surface or Run Dashboard behavior.
