# v0.26.6 Run Dashboard / Session Monitor

## Version Definition

- Version: `v0.26.6`
- Name: Run Dashboard / Session Monitor
- Korean name: Run Dashboard Session Monitor
- Track: Workspace Agent Workbench
- Previous: `v0.26.5` Safety Gate / Approval Console
- Next: `v0.26.7` Workbench Command Surface

## Objective

`v0.26.6` creates refs-only Run Dashboard and Session Monitor artifacts over
existing agent-surface, trace, approval, evidence, provider, PIG guidance,
failure, run, and session references.

It makes run and session state inspectable. It does not render a UI, start a
daemon, watch continuously, execute work, rerun failed work, repair failures, or
promote session context into memory.

## Relationship to Earlier Versions

`v0.26.5` creates Safety Gate / Approval Console records. `v0.26.6` may summarize
approval, rejection, deferral, manual review, and human intervention status from
those records.

`v0.25.7` creates Ask / REPL Surface reports. `v0.26.6` may read their report,
pipeline, turn, emission, and session refs as existing source refs only.

`v0.25.8` creates Agent Trace / Usability Telemetry reports. `v0.26.6` may read
their trace, stage, decision, route, provider, response, and metric refs as
source refs only.

## Feedback Absorption Rule

ChantaCore may absorb Grok/Codex-style agent-usability feedback only as
OCEL-visible guidance, candidate, rationale, approval, inspection, dashboard,
session-view, and telemetry surfaces.

Allowed absorption in this version includes refs-only session summaries, run
status summaries, pipeline status summaries, trace summaries, PIG guidance
summaries, repeated route/provider/safety/failure pattern views, approval status
summaries, long-task context refs, human intervention summaries, and trace
coverage summaries.

Forbidden absorption includes raw transcript session memory, PIG as memory,
persona mutation, automatic policy mutation, execution authority, autonomous
monitor loops, watcher daemons, retries, repairs, provider boundary bypasses,
command execution, external adapter runtime dependencies, raw transcript
persistence, and raw provider output dumps.

## Core Principles

- Run Dashboard is not background execution.
- Session Monitor is not memory continuity.
- Session summary is not raw transcript storage.
- Repeated pattern view is not autonomous optimization.
- Failure summary is not automatic repair.
- Run status card is not rerun button.
- Approval status card is not approval execution.
- Provider status card is not provider invocation.
- PIG guidance summary is not memory, policy mutation, or execution.
- Long-task context refs are not persistent memory.

## Source View

The source view references ask/REPL reports, pipeline runs/results, surface
emissions, REPL sessions/turns, trace reports, approval reports, evidence
reports, provider reports, and PIG guidance refs.

It records `raw_provider_output_included=false`,
`raw_transcript_included=false`, `raw_secret_included=false`, and
`private_full_path_included=false`.

## Dashboard and Session Views

Run cards expose run refs and status summaries. They never enable rerun,
command, or provider invocation.

Session cards and trace summaries are refs-only. They never include raw
transcripts and never enable memory continuity.

Repeated decision, route, provider, safety, and failure pattern views are
descriptive. They do not optimize, rerun, invoke providers, mutate safety
policy, retry, or repair.

Context-ref views hold long-task context references only. They are not memory
promotion.

## Metrics

The metric set computes descriptive counts for runs, sessions, completed runs,
warnings, failures, blocks, approval activity, provider activity, safety blocks,
unsupported claims, uncertainty, limitations, and repeated patterns.

Boundary counters for direct bypass, command rerun, automatic repair,
autonomous loop, background execution, memory promotion, and raw transcript
persistence remain zero.

## Status Rules

`passed` is available when source, dashboard, session monitor, summaries,
patterns, context refs, and metrics are created without boundary violations.

`warning` is available for partial optional sources, missing optional PIG refs,
or insufficient repeated-pattern refs.

`failed` is used for required strict-mode source or view construction failures.

`blocked` is used for background monitoring, watcher behavior, auto-refresh
execution, rerun/retry/repair, command execution, approval execution, provider
invocation, ask/REPL execution, final response emission, local command
execution, direct file/subprocess access, memory continuity/promotion, PIG as
memory/policy/execution, external adapters, raw output persistence, Schumpeter
split, OpenCode/OpenClaw/Hermes runtime dependency, or LLM judge.
