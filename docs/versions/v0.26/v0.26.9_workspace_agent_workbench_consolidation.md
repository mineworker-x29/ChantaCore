# v0.26.9 Workspace Agent Workbench Consolidation

- Version: `v0.26.9`
- Name: Workspace Agent Workbench Consolidation
- Korean name: Workspace Agent Workbench tonghap.release readiness
- Release name: Workspace Agent Workbench Foundation v1
- Track: Workspace Agent Workbench
- Previous: `v0.26.8` Workbench Snapshot / OCEL Export
- Next: `v0.27.0` Memory Candidate & Continuity Contract

## Objective

`v0.26.9` consolidates `v0.26.0` through `v0.26.8` into Workspace
Agent Workbench Foundation v1. It creates release-readiness artifacts only:
foundation snapshot, subject components, capability map, coverage matrix,
safety and interaction boundary reports, event quality and trace coverage
consolidation reports, usability readiness, process-intelligence feedback-loop
readiness, gap register, `v0.27` readiness, refs-only memory-candidate handoff,
release manifest, and consolidation report.

## Relationship To Prior Work

- `v0.26.0` introduced the Workspace Agent Workbench Contract.
- `v0.26.1` introduced Workbench View State and Panel Models.
- `v0.26.2` introduced Trace Explorer and Pipeline Timeline.
- `v0.26.3` introduced Provider / Capability Browser.
- `v0.26.4` introduced Evidence / Report Inspector.
- `v0.26.5` introduced Safety Gate / Approval Console.
- `v0.26.6` introduced Run Dashboard / Session Monitor.
- `v0.26.7` introduced Workbench Command Surface.
- `v0.26.8` introduced Workbench Snapshot / OCEL Export.
- `v0.25.9` remains the previous Bounded General Agent Surface Foundation v1.

## Feedback Absorption Rule

Grok/Codex usability feedback may be absorbed only as OCEL-visible guidance,
candidate, rationale, approval, inspection, dashboard, session-view, telemetry,
command-candidate, snapshot, export, and consolidation surfaces.

## Boundary Principles

- Consolidation is not new Workbench feature implementation.
- Workbench release readiness is not memory continuity.
- `v0.27` readiness is not `v0.27` implementation.
- Memory candidate readiness is not memory candidate extraction.
- PIG guidance visibility is not PIG memory.
- Command candidate visibility is not command execution.
- Approval decision visibility is not approval execution.
- Event quality report is not automatic optimization.
- Trace coverage report is not autonomous improvement.

## Included Scope

The release defines models and services for:

- foundation snapshot and subject components
- capability map and coverage matrix
- safety boundary and interaction boundary reports
- event quality and trace coverage consolidation
- usability readiness and process-intelligence feedback-loop readiness

## Non-Goals

`v0.26.9` does not implement memory candidate extraction, memory scoring,
memory promotion, persistent memory write, persona mutation, session continuity,
new Workbench panel behavior, command execution, provider invocation, local
command execution, file mutation, ask/repl execution, final response emission,
rerun/retry/repair, autonomous loops, background daemons, continuous watchers,
external/vendor adapters, pm4py/ocpa runtime dependency, OpenCode/OpenClaw/
Hermes runtime dependency, Schumpeter split, raw transcript/provider/secret
persistence or export, credential output, private path export, or LLM judging.

## Service Design

The service layer loads refs for v0.26 reports, builds subject components,
maps capabilities, computes coverage, verifies zero dangerous boundary counts,
builds interaction and quality readiness, records future-track usability gaps,
creates a refs-only v0.27 handoff packet, and emits a release manifest and
consolidation report.

## Foundation Snapshot

The foundation snapshot includes `v0.26.0` through `v0.26.9`, points back to
`v0.25.9`, and marks every v0.26 subject as OCEL-visible. It does not mark any
component as memory-capable, external-adapter-capable, or execution-surface.

## Capability Map

The map covers contract, view state, trace inspection, provider inspection,
evidence inspection, approval record, dashboard monitor, command candidate,
snapshot export, and consolidation. Memory capability count, external adapter
count, and execution surface count are zero.

## Coverage Matrix

The matrix tracks model, service, CLI, tests, boundary tests, docs, OCEL, PIG,
OCPX, safety boundary, forbidden search, and latest artifact availability for
the v0.26 subjects.

## Safety Boundary Report

Dangerous counts are zero for UI rendering, provider invocation, command
execution, local command execution, file mutation, ask execution, final response
emission, rerun/retry/repair, autonomous loops, background monitoring, memory,
external adapters, pm4py/ocpa dependency, raw persistence, Schumpeter split,
and LLM judge.

## Interaction Boundary Report

Trace view, provider browser, evidence inspector, approval console, run
dashboard, session monitor, command candidate surface, snapshot/export,
do-nothing candidate, approval/rejection/deferral visibility, and human
intervention point visibility are consolidated as available. Execution boundary
is preserved.

## Event Quality And Trace Coverage

Event quality consolidation includes decision, skill, action, route, provider,
safety, PIG, approval, failure, and human intervention coverage fields.
Trace coverage consolidation covers trace explorer, provider browser, evidence
inspector, approval console, dashboard, command surface, and snapshot/export.
These reports are descriptive and do not optimize automatically.

## Usability And Feedback Loop

Usability readiness confirms Default Agent visibility for skill/action/provider
candidate refs, evidence, safety rationale, approval flow, command candidates,
session context refs, PIG guidance, and do-nothing alternatives.
The process-intelligence feedback loop report marks OCEL event quality, OCPX
readiness, PIG visibility, workbench inspection, approval records, command
candidates, snapshot/export, and v0.27 memory-candidate inputs as ready.

## Gap Register

Future-track gaps remain tracked for memory candidate extraction, session
continuity, durable memory, memory promotion gate, Schumpeter split preparation,
external provider adapters, external agent dominion bridge, pm4py/ocpa adapter,
production UI, and autonomous multi-step loop. These are not v0.26 blockers
unless they contaminate the v0.26 boundary.

## v0.27 Readiness And Handoff

The `WorkbenchV027ReadinessReport` can mark `ready_for_v0_27=true` when the
Workbench substrate is releasable. This does not create memory. The handoff
packet is refs-only and lists snapshot refs, OCEL export package refs, session
context refs, trace summaries, evidence summaries, PIG guidance refs, approval
decision refs, command candidate refs, failure cause refs, human intervention
refs, and event quality refs as possible inputs for the next track.

## Release Manifest

The manifest includes `v0.26.0` through `v0.26.9`, the Foundation v1 release
name, included Workbench capabilities, allowed read-only effects, and forbidden
effects. Excluded capabilities include memory extraction/promotion/persistence,
persona mutation, session continuity engine, external adapters, Schumpeter
split, pm4py/ocpa runtime adapter, autonomous loops, automatic retry/repair,
direct execution, direct file mutation, and raw transcript/provider output
persistence/export.

## OCEL Mapping

Object types include the foundation snapshot, subject component, capability map,
coverage matrix, safety and interaction reports, quality and trace reports,
usability readiness, feedback loop report, gap register, v0.27 readiness,
handoff packet, release manifest, finding, consolidation report, PIG report,
and OCPX projection.

Allowed effects are read-only observation, consolidation created, foundation
snapshot created, release manifest created, v0.27 readiness created, memory
candidate handoff packet created, and state candidate created.

## Acceptance Criteria

The release is acceptable when the consolidation report builds, Foundation v1
release manifest is releasable or releasable with warnings, dangerous boundary
counts are zero, the v0.27 handoff packet is refs-only, no memory/execution/
adapter/raw-output behavior is introduced, and `ready_for_v0_28=false`.

## Withdrawal Conditions

Withdraw readiness if memory candidate extraction, memory promotion, persistent
memory write, persona mutation, command/provider/local/file/ask execution,
final response emission, rerun/retry/repair, autonomous loop, external/vendor
adapter, pm4py/ocpa runtime dependency, PIG memory/policy/execution, raw
transcript/provider/secret persistence/output, Schumpeter split, OpenCode/
OpenClaw/Hermes runtime dependency, or LLM judge is introduced.

## Validity Horizon

Valid until `v0.27` Memory Candidate & Continuity begins or Workbench
Foundation policy changes.
