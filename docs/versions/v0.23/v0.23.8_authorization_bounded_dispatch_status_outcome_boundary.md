## Restore Scope

v0.23.8 introduces boundary-only authorization, bounded dispatch, status,
output, and outcome descriptors. Restore boundary request/report models,
authorization check/scope/consumption policy, dispatch mode policy, command
descriptor, provider interface boundary, status/output/outcome descriptors,
findings, OCEL mapping, CLI views, and PIG/OCPX projection.

## Restore Boundary

- Authorization boundary is not authorization consumption.
- Bounded dispatch boundary is not dispatch.
- Status boundary is not live status tracking.
- Output boundary is not output fetch.
- Outcome boundary is not real external outcome record.
- Next version: v0.23.9 Internal Dominion Consolidation / Release Readiness.
- v0.24.x = Internal Provider / Local Runtime Provider
- v0.25.x = General Agent Usability & Tool Routing
- v0.29.x+ = External Skill / External Provider Adapter Development
- `safe_to_dispatch=false` and `bounded_dispatch_allowed_now=false` in v0.23.8.
- CLI and reports must not expose credential values, raw secrets, or private
  full paths.

## Expected Model And Policy State

The restored boundary report should preserve:

- authorization exists/single-use/unconsumed/unexpired/scoped checks
- `authorization_valid_for_live_dispatch=false`
- `authorization_consumption_allowed_now=false`
- dispatch mode policy with provider API, runtime touch, actual dispatch,
  shell, local command, network, credential materialization, and LLM judge all
  disabled
- command descriptor with `command_payload_materialized=false`
- credential values not materialized
- provider-specific payload not created
- status tracking not started
- output fetch not started
- real external outcome not recorded

Report-level safety flags must remain false for safe-to-dispatch and bounded
dispatch allowed now.

## CLI And Trace Expectations

CLI output should show report id, boundary id, authorization id, report status,
authorization consumed=false, safe_to_dispatch=false,
bounded_dispatch_allowed_now=false, actual_dispatch_performed=false,
provider_api_call_performed=false, external_runtime_touched=false,
external_run_started=false, live status/output flags false, and sanitized
evidence refs.

OCEL should record boundary-created events, not provider-invoked or
external-control-dispatched events.

## Restore Procedure

1. Confirm `dispatch_boundary.py` exists.
2. Confirm dispatch boundary tests and boundary tests exist.
3. Build a boundary report from declared gate authorization and control plan
   inputs.
4. Confirm authorization is checked but not consumed.
5. Confirm dispatch/status/output/outcome descriptors are boundary-only.
6. Confirm report status is ready, warning, needs-more-input, no-action,
   failed, or blocked according to deterministic findings.
7. Run primary tests.
8. Run forbidden searches for actual dispatch, provider calls, runtime touch,
   authorization consumption, live status/output, real outcome recording,
   credentials, local runtime, and LLM judge.

## Evidence Strength

Evidence strength is high for boundary flags and tests. It is not evidence that
any live provider dispatch is safe or available; v0.23.8 intentionally does not
perform such dispatch.

## Withdrawal Conditions

Withdraw this addendum if authorization is consumed, a provider API is called,
an external runtime is touched, an actual dispatch is attempted, an external run
is started, live status/output is fetched, or real external outcome is recorded.

## Validity Horizon

Valid until provider-specific implementation begins in later tracks.
