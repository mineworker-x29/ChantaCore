## Restore Boundary

- Control request creation is not control execution.
- Control request is not dispatch.
- Control request is not control plan.
- Action candidate is not external runtime touch.
- Action candidate is not preflight.
- Action candidate is not authorization.
- No-action and needs-more-input are valid outcomes.
- Next version: v0.23.4 Control Plan & Target Binding.
- Action candidate creation is not external dispatch.
- No provider API call, runtime touch, authorization consumption, credential
  output, or LLM judge is allowed.

## Restore Detail

Restore v0.23.3 as the bridge from human/system intent into Dominion candidate
state. A control request expresses desired control. An action candidate
describes a possible external action. Neither is executable in v0.23.3.

Required restored surfaces:

- control request model
- external action candidate model
- request creation service
- candidate creation service
- deterministic validation and status fields
- source/evidence references
- OCEL mapping for request and candidate creation
- PIG/OCPX projection entries for candidate-only control grammar
- CLI views for request/candidate creation or inspection

The restored state must maintain a hard distinction between "candidate created"
and "action executed". Candidate payloads must not become provider-specific
runtime payloads.

## Expected Model And Policy State

The restored request/candidate layer should preserve:

- requested action verb as normalized descriptor text
- target references as ids, not live runtime handles
- input binding references as descriptors, not materialized secrets
- output policy references as policy ids, not output fetches
- risk/value notes as deterministic fields
- status values that can express partial, blocked, unsupported, or candidate-only

Forbidden states include actual dispatch, external control dispatch, provider
API calls, runtime touch, credential materialization, and shell/local command
execution.

## CLI And Trace Expectations

CLI output should print request and candidate ids, status, normalized action
verb, sanitized target refs, and non-secret evidence refs. OCEL should record
candidate creation events, not dispatch or provider invocation events.

## Restore Procedure

1. Confirm `control.py` exists.
2. Confirm control request/candidate tests exist.
3. Create or load a sample request using fixture-like data.
4. Confirm the resulting action candidate is descriptor-only.
5. Confirm no provider-specific payload is created.
6. Run primary tests.
7. Run forbidden searches for dispatch, provider calls, runtime touch, and
   credential materialization.

## Evidence Strength

Evidence strength is high for local source/test boundaries. Exact request
contents are scenario-specific and unknown without supplied fixture data.

## Withdrawal Conditions

Withdraw this addendum if candidates become executable commands, provider
payloads, or live dispatch requests in v0.23.x.

## Validity Horizon

Valid while v0.23.x remains non-executing Internal Dominion Foundation.
