## Restore Boundary

- Inventory is observation, not runtime ping.
- Inventory is not dispatch.
- Inventory is not provider API discovery.
- Inventory is not runtime touch.
- Inventory is not credential materialization.
- Next version: v0.23.2 Capability Observation & Digestion for Dominion.
- Inventory must not touch live external runtimes.
- Inventory must not call provider APIs.
- Inventory must not read credentials or expose private full paths.
- Inventory must not implement Local Runtime Provider.

Run the primary tests and confirm inventory CLI output remains sanitized.

## Restore Detail

Restore v0.23.1 as a declared inventory layer. The inventory records what
runtime, agent, and system surfaces are known to Internal Dominion without
probing them live.

Required restored surfaces:

- runtime inventory descriptors
- agent inventory descriptors
- system inventory descriptors
- inventory report and status calculation
- inventory source/evidence references
- OCEL object and event mappings for inventory visibility
- PIG/OCPX projection entries that say inventory is observational
- CLI inventory views that do not read credentials or touch runtimes

Inventory status may be complete, partial, missing, warning, or blocked
according to implementation rules, but the status must be derived from declared
data and deterministic checks. It must not depend on a network call, shell call,
provider call, or runtime ping.

## Expected Model And Policy State

The restored inventory model should preserve these ideas:

- inventory is a read model for Dominion planning
- provider and runtime references are descriptors, not live handles
- capability and runtime ids may be used as references, not invocation targets
- missing inventory creates a report condition, not an execution attempt

Forbidden states include:

- live runtime probe
- provider API call
- credential materialization
- private full path output
- local command execution
- Local Runtime Provider implementation

## CLI And Trace Expectations

The inventory CLI should print only sanitized identifiers and status fields.
OCEL visibility should make inventory objects and inventory-created events
inspectable without exposing credentials or private machine details.

## Restore Procedure

1. Confirm `inventory.py` exists and imports without starting runtime work.
2. Confirm inventory tests exist.
3. Build the inventory report through the service layer.
4. Confirm all runtime-touch/provider-call flags remain false or absent.
5. Run inventory tests.
6. Run forbidden searches for provider calls, runtime touch, shell execution,
   and credential exposure.

## Evidence Strength

Evidence strength is high for the file/test surface and policy boundary. Exact
inventory data contents are environment-dependent or synthetic and are therefore
unknown unless supplied by fixtures.

## Withdrawal Conditions

Withdraw this addendum if inventory starts performing live runtime reachability,
provider invocation, network inspection, local command execution, or credential
materialization.

## Validity Horizon

Valid while v0.23.x remains Internal Dominion Foundation and v0.24 provider
runtime work has not redefined inventory semantics.
