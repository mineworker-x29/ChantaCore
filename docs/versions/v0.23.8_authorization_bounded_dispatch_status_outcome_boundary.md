# v0.23.8 Authorization / Bounded Dispatch / Status / Outcome Boundary

Korean name: 권한·제한 Dispatch·상태·결과 경계

Track: Internal Dominion Foundation

## Version Definition

v0.23.8 creates provider-neutral Dominion boundary objects after v0.23.7 Human Review & Dominion Gate. It checks that a gate authorization exists, is scoped, is single-use, unexpired, and unconsumed, then creates descriptor-only dispatch/status/output/outcome boundaries.

This version is boundary-only. It does not consume authorization, dispatch, start external runs, call provider APIs, touch external runtimes, track live status, fetch live outputs, or record real external outcomes.

## Objective

The objective is to make the post-gate boundary visible in OCEL before any provider-specific implementation exists. A passing boundary report may mark the plan eligible for v0.23.9 consolidation visibility, but it never makes the plan safe to dispatch.

## Service Design

The boundary service reads the v0.23.7 gate report, gate authorization, v0.23.6 runtime preflight report, and v0.23.4 control plan as read-only source models. It then builds authorization boundary check, authorization scope descriptor, authorization consumption policy, dispatch mode policy, descriptor-only command boundary, provider dispatch interface boundary, bounded dispatch boundary, status boundary, output boundary, outcome boundary, findings, and report status.

## Boundary Status Rules

ready_for_consolidation requires an open gate, valid unconsumed single-use authorization, matching scope, boundary_only dispatch mode, descriptor-only command, provider interface boundary, status/output/outcome boundary descriptors, and no forbidden effect.

needs_more_input is used for missing authorization, partial scope, incomplete declared provider/status/output/outcome information, or incomplete command descriptors.

no_action is used when the gate is rejected, no_action, needs_more_input, or no longer relevant.

failed is used for invalid authorization, expired authorization, scope mismatch, provider-specific logic in core, or premature future-track implementation.

blocked is used for consumed authorization, attempted authorization consumption, provider API call, external runtime touch, actual dispatch attempt, external run start, credential materialization/output, raw secret output, or real external outcome recording.
