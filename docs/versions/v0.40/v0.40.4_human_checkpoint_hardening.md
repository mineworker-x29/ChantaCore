# v0.40.4 Human Checkpoint Hardening & Scope-Bound Approval Contract

## 1. Purpose

v0.40.4 hardens human checkpoint semantics. A human approval is bound to a
specific loop, iteration, rehearsal, artifact set, freshness policy, and allowed
action class.

v0.40.4 does not open standalone Default Personal agent runtime.

A human checkpoint approval is scope-bound and does not grant runtime authority.

Approval cannot authorize live workspace mutation, model invocation, prompt submission, subagent invocation, autonomous continuation, Dominion runtime, or production certification.

## 2. Baseline from v0.40.0, v0.40.1, v0.40.2, and v0.40.3

v0.40.0 defined the mission-loop boundary and dry-run simulation.
v0.40.1 added sandbox rehearsal and standalone runtime gap accounting.
v0.40.2 added manual two-iteration rehearsal with checkpoint enforcement.
v0.40.3 added negative runtime gate regression and denied-action coverage.

v0.40.4 builds on those layers without replacing them.

## 3. Why v0.40.4 Adds Human Checkpoint Hardening

Earlier checkpoint metadata proved that a second iteration requires human
approval. v0.40.4 narrows that approval so it cannot be interpreted as broad
future authority.

## 4. What v0.40.4 Opens

- Scope-bound approval metadata.
- Artifact binding metadata.
- Freshness and staleness metadata.
- Revocation and expiry metadata.
- Authority boundary records.
- Restore context snapshot, restore packet, and restore document manifest.
- v0.40.5 Provider / Prompt Boundary handoff.

## 5. What v0.40.4 Does Not Open

- Standalone Default Personal ChatService, Orchestrator, AgentLoop, CLI, or TUI.
- Model/provider invocation.
- Prompt submission.
- Subagent invocation.
- External coding agent execution.
- Live workspace apply.
- Autonomous loop, automatic repair, retry loop, or unbounded multi-cycle loop.
- Shell, arbitrary command, dependency install, network, or credential access.
- Dominion runtime.
- Production certification.

## 6. Scope-Bound Approval Overview

Approval scope records bind approval to `loop_id`, `mission_id`,
`iteration_index`, `checkpoint_request_id`, `checkpoint_decision_id`,
`rehearsal_id`, `artifact_refs`, `approved_action_class`, and freshness policy.

The only positive action class is `manual_second_iteration_rehearsal`.

## 7. Checkpoint Freshness Policy

Freshness policy marks a checkpoint stale when loop, iteration, artifact set,
policy version, or revocation status changes.

## 8. Artifact Binding Requirement

Approval must be tied to reviewed artifacts: checkpoint request, checkpoint
decision, loop, iteration state, rehearsal result, safety report, negative gate
report, and optional restore packet.

## 9. Revocation and Expiry Metadata

v0.40.4 adds records for revoked and expired checkpoint decisions. A revoked or
expired decision is not valid for constructing a manual second-iteration
candidate.

## 10. Authority Boundary

The authority boundary may allow only a manual rehearsal candidate when
validation succeeds. Runtime execution, live workspace mutation, provider calls,
prompt submission, subagent invocation, autonomous continuation, retry,
Dominion, and certification remain false.

## 11. Restore Document and Restore Packet

v0.40.4 includes a restore document and metadata packet to let future GPT/Codex
sessions recover the current state without accidentally opening unsafe runtime
authority.

## 12. Safety Reports and Readiness Flags

Readiness may mark checkpoint hardening, artifact binding, freshness,
revocation/expiry metadata, restore document, and v0.40.5 handoff ready.

All unsafe flags remain false.

## 13. v0.41 Smoke Run Acceleration Signal

The signal is non-authoritative. It may suggest whether v0.41 smoke-run planning
can be accelerated, but it does not start standalone runtime.

## 14. Tests

Focused tests live in:

```powershell
py -m pytest tests\test_v0404_human_checkpoint_hardening_restore.py
```

Regression tests include v0.40.3, v0.40.2, v0.40.1, v0.40.0, and v0.39.9.

## 15. Known Limitations

v0.40.4 does not implement provider boundary deepening, prompt dispatch
candidates, provider output quarantine, standalone runtime, or user-facing
runtime surfaces.

## 16. Withdrawal Conditions

Withdraw v0.40.4 if broad approval is accepted, stale approval is valid,
artifact mismatch is valid, negative-gate-incompatible approval is valid,
runtime authority is granted, restore document is missing, or any unsafe
runtime capability opens.
