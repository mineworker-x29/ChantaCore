# v0.40.2 Manual Two-Iteration Rehearsal & Human Checkpoint Enforcement

Release name: v0.40.2 Manual Two-Iteration Rehearsal & Human Checkpoint Enforcement.

Track name: Standalone-Agent Preparation Track: Controlled MissionLoop Boundary + Sandbox Rehearsal + Manual Two-Iteration Checkpoint Gate.

## 1. Purpose

v0.40.2 proves that a second sandbox rehearsal iteration cannot proceed from the first sandbox rehearsal unless explicit human checkpoint decision metadata exists.

It may construct and run a bounded iteration 1 sandbox rehearsal only in test-controlled conditions with:

- explicit checkpoint decision metadata,
- max two-iteration cap,
- manual-only continuation,
- no runtime authority grant,
- no autonomous continuation.

## 2. Baseline from v0.40.0 and v0.40.1

v0.40.0 provides the controlled mission-loop boundary metadata: MissionLoopEnvelope, IterationState, StopConditionContract, LoopBudgetGate, HumanCheckpointGate, LoopDecisionRecord, denied runtime action metadata, dry-run packet, and readiness report.

v0.40.1 provides the sandbox rehearsal layer over the v0.39.3 sandbox-only apply primitive and the v0.39.4 controlled retest primitive. It also records that standalone Default Personal runtime remains closed.

v0.40.2 connects those two layers into a manually gated two-iteration rehearsal.

## 3. Why v0.40.2 Adds Manual Two-Iteration Rehearsal

v0.40.1 verified a single bounded sandbox rehearsal. v0.40.2 tests the first dangerous edge of a loop: moving from iteration 0 to iteration 1.

The core question is not whether ChantaCore can auto-continue. It cannot. The core question is whether a second candidate can be blocked unless human checkpoint metadata explicitly allows a manual bounded rehearsal.

## 6. Iteration 0 -> Checkpoint -> Iteration 1 Candidate Flow

The flow is:

1. Run iteration 0 through the v0.40.1 sandbox rehearsal helper.
2. Emit ManualIterationCheckpointRequest after iteration 0.
3. Require ManualIterationCheckpointDecision before iteration 1 eligibility is true.
4. Evaluate SecondIterationEligibilityDecision.
5. Run iteration 1 sandbox rehearsal only when the decision is approve_second_iteration_rehearsal.
6. Record result, audit, safety, readiness, no-autonomy guarantee, and handoff metadata.

## 7. Second Iteration Eligibility Gate

The gate behavior is:

- missing checkpoint decision: blocked with human_checkpoint_missing.
- stop: blocked.
- do_nothing: blocked.
- reject: blocked.
- request_more_evidence: blocked.
- approve_second_iteration_rehearsal: eligible only as manual bounded rehearsal.

Eligibility is not execution permission.

## 8. No Autonomous Continuation Guarantee

NoAutonomousContinuationGuarantee records:

- autonomous_continuation_allowed false,
- automatic_second_iteration_allowed false,
- unbounded_loop_allowed false,
- retry_loop_allowed false,
- human_checkpoint_required true.

## 9. Max Two-Iteration Cap

ManualTwoIterationRehearsalInput and ManualTwoIterationRehearsalPlan reject max_iterations greater than 2.

ManualTwoIterationRehearsalResult records max_iteration_cap_enforced true.

## 13. Known Limitations

- v0.40.2 is not a user-facing runtime.
- v0.40.2 does not open Default Personal standalone operation.
- v0.40.2 does not execute model calls, prompt submission, subagent invocation, live workspace mutation, or autonomous repair.
- v0.40.2 does not replace v0.40.1 sandbox rehearsal; it composes it.

## 14. Withdrawal Conditions

- iteration 1 can proceed without checkpoint decision metadata,
- approval grants runtime authority,
- max_iterations greater than 2,
- autonomous continuation,
- retry loop,
- automatic repair loop,

- subagent invocation,
