# v0.40.0 Controlled Multi-Iteration Mission Loop Boundary Foundation

Track name: Controlled Multi-Iteration Mission Loop with Execution-Test Ladder and Subagent Verification Boundary.

## 1. Purpose

v0.40.0 creates the boundary foundation for a controlled multi-iteration mission loop. It can describe, simulate, deny, and guide. It cannot execute, invoke, submit, delegate, mutate, persist, or certify.

The release adds metadata artifacts for loop envelopes, iteration state, stop conditions, budget gates, human checkpoints, loop decisions, denied runtime actions, provider/prompt/subagent boundaries, dry-run packets, and readiness reporting.

## 2. Baseline From v0.39.9

v0.39.9 sealed the v0.39 track as Human-approved Sandbox Repair Apply & Re-test Loop v1 with PI-native Self-Prompting Mission Loop Boundary.

v0.39.x can support bounded human-approved sandbox apply, controlled post-apply re-test, before/after comparison, process-state reconstruction metadata, self-prompt draft metadata, subagent prompt draft metadata, and CLI preview surface metadata.

v0.39.x still does not support live apply, prompt submission to a model, model provider invocation, actual subagent invocation, external coding agent execution, autonomous loop runtime, retry loop, multi-cycle repair loop, Dominion runtime, or production certification.

v0.40.0 consumes that handoff as context only. It does not reimplement v0.39 artifacts.

## 5. Core Artifacts

`MissionLoopEnvelope` is the top-level loop metadata envelope. It defaults to iteration index `0`, max iteration count `1`, human checkpoint required, dry-run only, no autonomous continuation, no sandbox rehearsal, and no production certification.

`IterationState` records metadata refs for a loop iteration. The status `eligible_for_next_iteration` is not execution permission.

`StopConditionContract` defines required stop signals. Unsafe runtime requests must lead to block or stop decisions. `do_nothing` remains a valid loop decision.

`LoopBudgetGate` records iteration, prompt draft, verifier request draft, and estimated token budgets. Unknown token budget is not unlimited and requires checkpoint handling.

`HumanCheckpointGate` is mandatory for future continuation. Its default decision is `stop`, and approval does not grant runtime authority.

`LoopDecisionRecord` records draft-only loop decisions: `continue_as_draft`, `request_human_checkpoint`, `request_verifier_draft`, `stop`, `do_nothing`, or `block`.

`DeniedRuntimeActionMetadata` records blocked runtime requests and suggests safe alternatives such as `dry_run`, `request_human_checkpoint`, `create_draft`, `simulate`, `do_nothing`, or `stop`.

`ProviderBoundaryGate`, `PromptSubmissionBoundary`, and `VerifierSubagentBoundary` are metadata gates only. They do not invoke providers, submit prompts, or invoke subagents.

`SimulatedMultiIterationLoopPacket` is dry-run metadata. It does not execute, mutate files, invoke a model, or invoke a subagent.

`V040ReadinessReport` reports that the v0.40.0 boundary foundation is ready while all unsafe runtime flags remain false.

## 6. Dry-Run Simulation Boundary

The dry-run simulation can create a packet that shows an initial iteration and a second-iteration checkpoint marker. It stops before continuation and requests human checkpoint metadata.

Dry-run simulation is not autonomous runtime. It is not sandbox rehearsal. It does not apply patches, run tests, call models, submit prompts, invoke subagents, write traces, or certify production.

## 7. Human Checkpoint Requirement

Human checkpoint remains mandatory. In v0.40.0, second iteration cannot be treated as available without checkpoint metadata.

The checkpoint does not grant broad runtime authority. It can support review and next design decisions, but it cannot by itself authorize provider invocation, prompt submission, subagent invocation, live apply, automatic repair, or production certification.

## 8. Provider / Prompt / Subagent Boundary

Provider boundary metadata records provider/model/prompt refs and keeps dispatch mode as `metadata_only`.

Prompt submission boundary metadata keeps `submission_allowed=False` and `submitted_to_model=False`.

Verifier subagent boundary metadata keeps `invocation_allowed=False`, `subagent_invoked=False`, and `parent_context_shared=False`.

These boundaries are preparation for future gates. They are not runtime calls.

## 9. Denied Runtime Action Metadata

Denied runtime action metadata records unsafe requested actions and maps them to safe alternatives. Examples include denying model provider invocation in favor of `create_draft`, denying automatic repair in favor of `stop`, and denying live workspace apply in favor of `dry_run`.

Denied metadata is not remediation execution.

## 10. Default Personal Usage Surface

The Default Personal user guide explains how Chanta Minero can read v0.40.0 dry-run outputs. It clarifies that Default Personal is not a permission bypass mode.

The guide covers dry-run packet reading, loop decisions, denied runtime action metadata, human checkpoint handling, do-nothing decisions, and v0.40.1 handoff criteria.

## 13. Known Limitations

Unknown token budget remains conservative and requires checkpoint handling.
