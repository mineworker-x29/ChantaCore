# v0.40.4 Scope-Bound Approval Contract

## 1. Contract Purpose

This contract prevents a human checkpoint approval from becoming broad runtime
authority. Approval is specific, bounded, revocable, expirable, and compatible
with negative runtime gates.

## 2. Scope Fields

Required scope fields:

- `loop_id`
- `mission_id`
- `iteration_index`
- `checkpoint_request_id`
- `checkpoint_decision_id`
- `approved_rehearsal_id`
- `approved_action_class`
- `artifact_refs`
- freshness policy reference

## 3. Allowed Action Classes

- `manual_second_iteration_rehearsal`
- `request_more_evidence`
- `stop`
- `do_nothing`
- `reject`
- `restore_context_only`

## 4. Forbidden Action Classes

- `model_provider_invocation`
- `prompt_submission`
- `subagent_invocation`
- `live_workspace_apply`
- `autonomous_loop`
- `retry_loop`
- `dominion_runtime`
- `production_certification`

## 5. Artifact Binding

Approval must bind the reviewed checkpoint decision to the evidence artifacts
used to make the decision.

## 6. Freshness and Staleness

Approval becomes stale when the loop, iteration, artifact set, policy version,
or revocation state changes.

## 7. Revocation

A revocation record can mark a checkpoint decision as no longer usable.

## 8. Expiry

An expiry record can mark a checkpoint decision as past its approved freshness
window.

## 9. Authority Boundary

The authority boundary allows only manual rehearsal candidate construction when
validation passes. It does not allow runtime execution.

## 10. Validation Matrix

| condition | result |
| --- | --- |
| Valid scoped manual rehearsal approval | manual candidate may be constructed |
| Broad future authority | rejected |
| Forbidden action class | rejected |
| Missing artifact binding | rejected |
| Stale checkpoint | rejected |
| Negative gate incompatible approval | rejected |
| Runtime authority request | rejected |

## 11. Examples

Good example:

```text
Approve only manual_second_iteration_rehearsal for loop_id=L, iteration_index=1, checkpoint_request_id=R, artifact_refs=[A,B,C].
```

## 12. Anti-Examples

Bad examples:

```text
Approve all future iterations.
Allow model call.
Allow live apply.
Production certified.
Dominion ready.
```

## 13. Safety Invariants

- Approval does not grant runtime authority.
- Approval does not grant live workspace authority.
- Approval does not grant model invocation authority.
- Approval does not grant prompt submission authority.
- Approval does not grant subagent invocation authority.
- Approval does not grant autonomous continuation authority.
- Standalone Default Personal runtime remains closed.
