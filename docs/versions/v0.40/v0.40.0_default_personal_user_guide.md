# v0.40.0 Default Personal User Guide

This guide is for Chanta Minero using Default Personal mode.

## 1. What v0.40.0 Means In Default Personal

v0.40.0 means ChantaCore can describe a controlled mission loop, simulate loop state as dry-run metadata, deny unsafe runtime requests, and show what a human checkpoint should review.

It does not mean ChantaCore can run autonomously.

## 2. Default Personal Is Not Permission Bypass

Default Personal is not an override mode. It does not bypass model/provider gates, prompt submission gates, subagent gates, live workspace gates, shell/process gates, or production certification gates.

If a boundary says `False`, Default Personal should treat that as a real stop sign.

## 3. What A General User Can Do In v0.40.0

In v0.40.0, a user can:

- Ask for a mission loop envelope summary.
- Ask what iteration index is represented.
- Ask whether a second iteration needs checkpoint metadata.
- Read a dry-run simulation packet.
- Read a loop decision record.
- Read denied runtime action metadata.
- Ask why a model call, prompt submission, subagent call, automatic repair, retry loop, or live apply is denied.
- Choose `do_nothing` when the safest next step is to stop.
- Prepare questions for v0.40.1 sandbox rehearsal design.

## 4. What A General User Still Cannot Do

In v0.40.0, a user cannot:

- Start actual autonomous loop runtime.
- Execute sandbox apply through v0.40.0.
- Execute controlled re-test through v0.40.0.
- Submit prompts to a model.
- Invoke a model/provider.
- Invoke a subagent.
- Invoke an external coding agent.
- Start automatic repair.
- Start retry or multi-cycle repair loops.
- Apply to a live workspace.
- Run shell, raw process, or arbitrary commands.
- Install dependencies or use network access.
- Persist traces.
- Write OCEL/OCPX files.
- Execute PIG recommendations.
- Use Dominion runtime.
- Certify production readiness.

## 5. How To Read Dry-Run Loop Simulation

A dry-run loop simulation tells you what would be considered, not what was executed.

Read these fields first:

- `loop_envelope.current_iteration_index`
- `loop_envelope.max_iteration_count`
- `loop_envelope.human_checkpoint_required`
- `loop_envelope.dry_run_only`
- `iteration_states`
- `loop_decision.decision_kind`
- `denied_actions`
- `executed`
- `mutated_files`
- `invoked_model`
- `invoked_subagent`

Expected safe values:

- `executed=False`
- `mutated_files=False`
- `invoked_model=False`
- `invoked_subagent=False`

## 6. How To Read LoopDecisionRecord

`LoopDecisionRecord` tells you the safe decision chosen by the boundary layer.

Valid decision kinds:

- `continue_as_draft`
- `request_human_checkpoint`
- `request_verifier_draft`
- `stop`
- `do_nothing`
- `block`

Important: `continue_as_draft` is not execution. It means draft metadata may continue, not runtime action.

## 7. How To Read DeniedRuntimeActionMetadata

`DeniedRuntimeActionMetadata` tells you what was denied and what safer alternative exists.

Important fields:

- `requested_action_type`
- `denial_reason`
- `safety_boundary`
- `suggested_safe_alternative`

Safe alternatives include:

- `dry_run`
- `request_human_checkpoint`
- `create_draft`
- `simulate`
- `do_nothing`
- `stop`

Example: if `requested_action_type=model_provider_invocation`, the safe alternative may be `create_draft`.

## 8. What To Do When HumanCheckpointGate Appears

When `HumanCheckpointGate` appears, the user should review:

- current iteration index
- max iteration count
- stop condition
- budget status
- denied runtime action list
- current loop decision
- whether `do_nothing` is safer

The checkpoint default decision is `stop`.

Checkpoint approval does not grant broad runtime authority. It does not authorize model calls, prompt submission, subagent calls, live apply, automatic repair, or production certification.

## 9. When `do_nothing` Is Valid

Choose `do_nothing` when:

- the next step has unclear evidence,
- budget is unknown,
- a stop condition is active,
- a runtime action request was denied,
- the human reviewer does not want another draft,
- continuing would create confusion between dry-run and execution.

`do_nothing` is not failure. In this boundary, it is a valid safety decision.

## 10. Criteria For Moving To v0.40.1 SandboxRehearsalRunner

Move to v0.40.1 only when:

- v0.40.0 focused tests pass,
- v0.39.9 regression tests pass,
- dry-run packet never executes or mutates,
- second iteration still requires human checkpoint metadata,
- provider/prompt/subagent boundaries remain metadata only,
- unsafe runtime readiness flags remain false,
- the user can read the Default Personal guide without mistaking dry-run for execution.

v0.40.1 should introduce SandboxRehearsalRunner only inside an explicit sandbox rehearsal boundary. It must still block live workspace apply and autonomous repair.

## User Example: Can We Do The Next Iteration?

User question:

> This state says there is another iteration. Can I continue?

Expected answer shape:

- current iteration index: from `MissionLoopEnvelope.current_iteration_index`
- max iteration: from `MissionLoopEnvelope.max_iteration_count`
- stop condition: from `StopConditionContract.active_stop_signals`
- budget status: from `LoopBudgetGate.budget_status`
- checkpoint requirement: from `HumanCheckpointGate.required`
- decision kind: from `LoopDecisionRecord.decision_kind`
- unsafe runtime flags: must remain false

If checkpoint metadata is missing, the expected decision is `request_human_checkpoint`, `stop`, or `do_nothing`.

## User Example: Why Can We Not Call A Model Now?

User question:

> Why can we not call a model right away?

Expected answer:

- `ProviderBoundaryGate.invocation_allowed=False`
- `PromptSubmissionBoundary.submitted_to_model=False`
- v0.40.0 is dry-run boundary only
- safe alternative is `create_draft` or `dry_run`

No model/provider call is allowed in v0.40.0.
