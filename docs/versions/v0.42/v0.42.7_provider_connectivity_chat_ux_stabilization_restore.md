## Run Help Fix

`chanta-cli run --help` now exits 0 and shows usage text with `--profile`, `--home`, `--provider`, `--timeout`, and prompt argument details.

Help does not require a prompt or home, does not call providers, does not submit prompts, and does not mutate session or trace state.

## Configured Provider Traced Path

Configured provider run attempts stay on the v0.41.5+ traced lifecycle path:

* `run_started`
* `user_input_received`
* `prompt_assembled`
* `provider_text_call_started`
* `provider_text_call_completed` or `provider_text_call_failed`
* `session_turns_appended` only when response text is available
* `assistant_response_recorded` only when response text is available
* `run_completed` or `run_failed`

Provider timeout trace metadata includes `error_class: provider_timeout` where available.

## Provider Connectivity Probe Contract

`chanta-cli provider connectivity` performs a bounded no-completion metadata probe.

The probe may call only:

* `GET <base_url>/models`

The probe must not call:

* `/chat/completions`
* `/completions`
* tool endpoints
* function-call surfaces

The probe is loopback/local by default and reports endpoint reachability, model-list availability, configured model presence, available model ids, timeout seconds, error class, and next action. Remote probe remains blocked unless explicitly allowed by future policy.

## Timeout and Error Classification

Configured-provider failures now render:

* `status: failed`
* `error_class: provider_timeout` or another structured class
* base URL
* model
* timeout seconds
* provider/prompt flags
* shell/subagent/production false flags
* next actions for LM Studio/local server checks, `/models` verification, timeout increase, and smaller model fallback

## Provider Status Semantics

Provider status keeps old fields for compatibility and adds clearer labels:

* `mock_provider_available`
* `active_provider_config_present`
* `configured_provider_run_ready`
* `configured_provider_connectivity`
* `configured_provider_next_action`

Mock provider availability is distinct from configured provider readiness.

## Chat UX Stabilization

Chat turn execution suppresses the child run command stdout and renders the assistant response once through the chat shell.

After each user message, chat state updates:

* `turn_count`
* `latest_run_id`
* `run_ids`
* latest assistant preview

Interactive `/status` uses the current in-memory chat state. `/provider` remains read-only and does not call providers or submit prompts.
