## Default Home Resolver Summary

v0.42.1 adds a bounded resolver for personal runtime home paths. The resolver chooses a home path and reports source/status, but it does not create directories by itself.

## Home Resolution Policy

Policy:

* explicit home is supported
* `CHANTACORE_HOME` override is supported
* platform default is supported
* silent auto-create is not allowed
* quickstart auto-create is allowed through existing bounded init
* init auto-create remains allowed

## Home Resolution Order

Resolution order:

1. explicit `--home`
2. `CHANTACORE_HOME`
3. platform default

If the resolved home is missing, read-only and run-like commands do not silently create it. They instruct the user to run quickstart, pass `--home`, or set `CHANTACORE_HOME`.

## CHANTACORE_HOME Environment Override

`CHANTACORE_HOME` lets the user set a default personal runtime home without repeating `--home`.

Example:

```powershell
$env:CHANTACORE_HOME="$env:TEMP\chantacore-v0421-env"
chanta-cli home show
```

## Platform Default Home

On Windows, the platform default is `%LOCALAPPDATA%\\ChantaCore` when `LOCALAPPDATA` is available.

If platform default cannot be resolved, the resolver reports an unresolved state rather than silently creating an unsafe path.

## Home Show Contract

`chanta-cli home show` is read-only.

It reports:

* resolved home path
* resolution source
* resolution status
* whether the home exists
* whether the default-personal profile exists
* whether the path is safe to use

It does not mutate files, call a provider, submit a prompt, execute shell, invoke subagents, or certify production.

## Home Status Contract

`chanta-cli home status` is read-only.

It reports:

* profile existence
* profile config existence
* sessions directory existence
* traces directory existence
* provider config existence
* recommended next action

It does not mutate files, call a provider, submit a prompt, execute shell, invoke subagents, or certify production.

## Command Home Adoption Contract

The following existing safe commands may omit `--home` and use resolver support after a home exists:

* `chanta-cli profile status`
* `chanta-cli provider doctor`
* `chanta-cli prompt preview`
* `chanta-cli session new`
* `chanta-cli session list`
* `chanta-cli run`
* `chanta-cli trace recent`
* `chanta-cli trace summary`
* `chanta-cli run-report last`
* `chanta-cli safety check-command`

These commands do not create a missing home/profile when `--home` is omitted. Quickstart and init are the creation paths.

## Quickstart Summary

`chanta-cli quickstart` resolves home, validates it, creates the default-personal profile if missing through existing bounded init behavior, summarizes profile status, runs provider doctor with no completion, and shows next actions.

## Quickstart Setup-only Contract

Setup-only quickstart:

* may create the bounded default-personal profile under resolved home
* is idempotent and does not overwrite existing profile files
* runs provider doctor with `--no-completion` semantics only
* does not call a real provider
* does not submit a prompt
* does not execute shell
* does not invoke subagents
* does not certify production

## Quickstart Mock-run Contract

`chanta-cli quickstart --with-mock-run` additionally runs the existing mock-provider run path.

It may append session turns and trace events through the existing bounded v0.41.6 stores. It uses mock provider only, marks provider text as untrusted, and keeps real provider invocation false.
