`chanta-cli provider connectivity` reached the local `/v1/models` endpoint and found the configured model. It did not call completion, submit a prompt, execute shell, invoke a subagent, or claim production certification.

## Configured Provider Empty Response Bug

The previous parser read only the simplest content path and treated empty string as a successful provider response. v0.42.8 makes empty final assistant content a failed run for configured provider calls.

Invalid condition for new configured runs:

`assistant_response_preview="" AND status="completed"`

Possible causes include PowerShell formatting, empty `message.content`, reasoning-only output, non-standard fields, chat template issues, or too-small token limits.

## Runtime Identity Stabilization

Configured provider payloads include a system prompt that separates:

* provider model identity: Qwen, LM Studio, OpenAI-compatible local model, or another model
* runtime identity: ChantaCore default-personal runtime
* assistant identity: ChantaCore assistant under the default-personal profile

Expected identity answer policy:

`저는 ChantaCore default-personal runtime에서 동작하는 대화형 assistant입니다. 현재 응답 생성에는 Qwen 계열 local provider가 사용되고 있습니다.`

## Provider Response Parser Contract

The parser extracts final answer content from:

* `choices[0].message.content`
* `choices[0].text`
* `response.output_text`
* optionally `choices[0].delta.content` only when policy allows stream fallback

The parser detects reasoning fields:

* `choices[0].message.reasoning_content`
* `choices[0].message.reasoning`

Reasoning content is diagnostic by default. It is not final assistant text unless a future policy explicitly allows that.

## Empty Response Handling Contract

If final assistant content is empty, missing, or whitespace:

* run status is failed
* `error_class` is `provider_empty_response`, `response_parse_empty`, or `provider_reasoning_without_final`
* no normal assistant turn is appended
* no successful `assistant_response_recorded` event is emitted
* CLI prints troubleshooting guidance
* run-report exposes response parse fields

## Run-report Response Fields

`run-report last` includes:

* `response_parse_status`
* `response_error_class`
* `response_extracted_from_field`
* `response_content_length`
* `response_finish_reason`
* `provider_model`
* `runtime_identity_included`
* `provider_identity_is_implementation_detail`
* `empty_response_detected`
* `next_action`

Older runs may have null response parse fields.

## Response Parse Trace Events

Success path:

* `provider_text_response_parsed`
* `assistant_response_recorded`
* `run_completed`

Empty or malformed path:

* `provider_text_response_empty` or `provider_text_response_parse_failed`
* `run_failed`
* no normal successful `assistant_response_recorded`

All events keep shell, subagent, and production certification flags false.

## Session Append Policy

Successful provider response:

* append user turn
* append non-empty assistant turn

Empty provider response:

* do not append empty assistant turn as normal success
* keep failure evidence in trace/report
* do not pollute normal future session context with empty assistant content

## Command Surface Stabilization

Direct run remains:

`chanta-cli run --profile default-personal --provider mock "hi"`

Inspection commands are routed before direct run parsing:

* `chanta-cli run show last`
* `chanta-cli run history`
* `chanta-cli session show last`

`chanta-cli run show last` must not produce a confusing argparse "unrecognized arguments" failure.

## Reasoning Model Troubleshooting Guide

Symptoms:

* completion tokens exist but `message.content` is empty
* reasoning content exists without final answer content
* PowerShell top-level rendering hides nested fields

Actions:

* inspect raw provider response JSON
* increase `max_tokens`
* ask for final answer only
* check LM Studio chat template
* try a smaller model
* run `chanta-cli provider connectivity`
* run a direct `/chat/completions` test
* do not treat empty response as completed success
