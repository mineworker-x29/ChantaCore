## Provider Setup UX Summary

v0.42.2 makes provider state inspectable and configurable without calling a real provider. Provider setup writes metadata only.

## Provider Preset Registry

Presets:

* mock
* local OpenAI-compatible
* LM Studio compatible metadata

All presets keep setup provider call false, status provider call false, doctor completion false, tool calling false, and function calling false.

## Provider Config Contract

Provider config path:

`<home>/profiles/default-personal/profile/PROVIDER.json`

Policy:

* bounded to resolved home
* create if missing
* no silent overwrite
* idempotent if identical
* conflict if different
* stores env var names only
* stores no raw API key values

## Provider Setup Mock Contract

`chanta-cli provider setup mock` writes bounded mock provider metadata. It does not call a provider and does not submit a prompt.

## Provider Setup Local OpenAI-compatible Contract

`chanta-cli provider setup local-openai --base-url http://localhost:1234/v1 --model local-model` writes text-only local provider metadata. `--api-key-env OPENAI_API_KEY` may store the env var name only.

Setup does not call the endpoint.

## Provider Status Contract

`chanta-cli provider status` reports provider config presence, provider kind, mode, base_url, model, env var name, env var presence, mock readiness, configured-provider readiness, and next action.

It does not call a provider.

## Provider Config Show Contract

`chanta-cli provider config show` shows redacted provider config metadata only. Secret values are never printed.

## Provider Secret Policy

Provider setup stores only env var names such as `OPENAI_API_KEY`. Provider status may check presence of that env var but does not print the value.

## Provider Run Readiness Contract

Run readiness separates:

* mock provider readiness
* configured provider readiness
* provider doctor no-completion availability

Completion remains allowed only through the existing `chanta-cli run` path.

## Provider Doctor UX Interpretation

Provider doctor remains no-completion. A passing doctor report does not mean completion was sent and does not by itself prove configured-provider run readiness.

## Provider Troubleshooting

Troubleshooting covers:

* provider config missing
* base_url missing
* model missing
* OPENAI_API_KEY not set
* LM Studio server not running
* using mock provider intentionally
* provider doctor passes but run fails
* configured provider flow not required for mock testing

## Provider User Guide

Recommended commands:

```powershell
chanta-cli provider status
chanta-cli provider setup mock
chanta-cli provider setup local-openai --base-url http://localhost:1234/v1 --model local-model --dry-run
chanta-cli provider setup local-openai --base-url http://localhost:1234/v1 --model local-model
chanta-cli provider config show
chanta-cli provider doctor --profile default-personal --no-completion
chanta-cli run --profile default-personal --provider mock "Summarize what ChantaCore is in three bullets."

```
