## Project Context for New Codex Session

ChantaCore is a personal OCEL/Process-Intelligence-native agent runtime project. It is not merely another coding agent. Its purpose is to build a personal runtime where actions, sessions, provider calls, denials, traces, and runtime states can be reconstructed as structured process evidence.

Design direction:

* profile-first personal runtime
* CLI-first local operation
* provider text-only invocation under strict scope
* append-only session and trace stores
* denial-first safety posture
* Process Intelligence traceability
* no unsafe capability unless explicitly gated in a later version

v0.42 responds to the fact that the user can no longer review every code path directly. The strongest review perspective is now user experience, operational behavior, Process Intelligence trace semantics, and whether runs, sessions, provider calls, denials, and reports can be understood as a process.

## Default Home UX Decision

Target version: v0.42.1.

Current state: the user flow often requires explicit `--home`.

Desired state: home resolution order should be:

1. explicit `--home`
2. `CHANTACORE_HOME`
3. platform default local app data

Windows default path:

```text
%LOCALAPPDATA%\ChantaCore
```

Decision:

* explicit `--home` remains supported
* environment override should be supported
* platform default should be supported
* auto-create should happen only through quickstart/init
* commands should not silently create a home from every command

## Provider UX Decision

Target version: v0.42.2.

Provider modes to support in UX:

* mock
* local OpenAI-compatible
* configured provider

Decision:

* provider setup/status UX should be added later
* provider doctor remains no-completion
* completion is allowed only in `run`
* provider doctor completion remains closed
* tool calling remains closed
* function calling remains closed
* secret redaction remains required

## Trace UX Decision

Target version: v0.42.3.

Current state: JSON trace remains useful but is hard for repeated human review.

Decision:

* JSON trace stays available
* human-readable timeline is required
* run history is required
* provider call event count vs transaction count must be clarified

Process Intelligence note: `provider_call_count=2` in the v0.41.6 evidence means provider call event count, not necessarily provider transaction count.
