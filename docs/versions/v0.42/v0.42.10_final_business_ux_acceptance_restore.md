## Final Business UX Acceptance Scope
v0.42.10 is an acceptance and gate version. It defines manual and automated acceptance criteria for default run, debug run, configured provider, mock provider, chat, provider status/connectivity, command guide, runtime identity, empty response handling, diagnostics, Process Intelligence review availability, and safety boundaries.

## Configured Provider Acceptance
Configured provider acceptance requires provider config, metadata connectivity, an actual completion attempt, clean default output, ChantaCore runtime identity, empty-response non-success handling, and debug output availability. If live provider completion is not tested, full pass is disallowed.

## Mock Provider Acceptance
Mock provider acceptance requires mock run, mock chat, no duplicated response, and debug availability. Mock provider remains the deterministic fallback for local acceptance.

## Default Run UX Acceptance
Default run output must hide raw trace/runtime banners, show the assistant response or a plain-language failure card, and may show a compact run/session footer.

## Debug UX Acceptance
Debug output must show run id, session id, provider/debug context, response parse status, response error class, trace event count or equivalent, and high-risk safety flags.

## Chat UX Acceptance
Chat must show a clean banner, grouped help, readable status and provider views, no duplicated response, safe internal commands, and operator review commands such as history, trace, run last, and handoff.

## Provider UX Acceptance
Provider status and connectivity must be readable, show model and endpoint when configured, show mock availability, show configured readiness, provide clear next action, hide secrets, and keep connectivity as a no-completion metadata probe.

## Command Guide Acceptance
`chanta-cli commands` must include Start, Talk, Provider, Review, Skills, Diagnostics, and Safety sections without internal Python class or artifact names as primary UX.

## Runtime Identity Acceptance
When asked "넌 누구야?", the target answer is ChantaCore default-personal work assistant. Provider model identity is implementation detail. Base model identity is not the primary identity.

## Empty Response Acceptance
Empty final assistant content must not be completed success. The user-facing message must be plain-language and actionable. Run-report parse fields remain required.

## Diagnostic Acceptance
Report bundle and copy-paste output must remain available with redaction. Diagnostic context should include provider, run, trace, feedback, and safety information.

## Process Intelligence Review Availability
Trace timeline, run-report, run history, report bundle, feedback note, and process evidence reconstruction remain available for operator and Process Intelligence review.

## Safety Boundary Acceptance
Provider doctor completion, provider tool calling, function calling, shell execution, file edit, patch apply, arbitrary file read, broad scan, repo search, subagents, general AgentLoop, Dominion, and production certification remain closed.

## Final UX Findings
The main non-code finding is live configured-provider dependence on the user's local server/model/template. This does not require new capability; it requires manual acceptance on the real configured home.
