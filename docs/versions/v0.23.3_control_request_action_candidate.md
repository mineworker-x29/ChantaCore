# v0.23.3 Control Request & Action Candidate

Version: v0.23.3
Version name: Control Request & Action Candidate
Korean name: 제어 요청·외부 행동 후보
Track: Internal Dominion Foundation
Layer: internal_dominion

## Objective

v0.23.3 implements the candidate-only Control Request & Action Candidate stage for Internal Dominion. It reads v0.23.1 runtime inventory and v0.23.2 capability candidates, then creates `DominionControlRequest` and `ExternalActionCandidate` objects.

## Relationship

v0.23.0 defines the Internal Dominion contract. v0.23.1 declares runtime/agent/system inventory. v0.23.2 digests capability descriptors into external capability candidates. v0.23.3 creates request/action candidates only.

## Policy Reminder

v0.23.x is Internal Dominion Foundation. v0.23.x is not Self-Execution Safety. Self-Execution Safety is reclassified to v0.24.x Local Runtime Provider. Vendor systems are future external provider skills/provider adapters. GrowthKernel is a future consumer / future optimizer, not an active runtime dependency.

## Principles

- Control request is not dispatch.
- Control request is not control plan.
- Action candidate is not external runtime touch.
- Action candidate is not preflight.
- Action candidate is not authorization.
- Action candidate is not execution.
- No-action and needs-more-input are valid outcomes.

## Included Scope

Data models include `DominionControlRequestCreateRequest`, `ControlTargetRef`, `DominionActionIntentDescriptor`, `DominionActionInputDraft`, `DominionControlConstraint`, `DominionPreliminaryActionRisk`, `DominionControlRequest`, `DominionActionCandidateFinding`, `ExternalActionCandidate`, `DominionNoActionCandidate`, `DominionNeedsMoreInputCandidate`, and `DominionControlRequestCandidateReport`.

Services include read-only source loading, target resolution, intent classification, input draft creation, constraints, preliminary risk assessment, candidate findings, external action candidate construction, and control request candidate reports.

## Non-Goals

This release does not create control plans, target bindings, runtime preflights, provider API calls, external runtime touches, external dispatches, external runs, capability execution, RPA/agent runtime API calls, local command execution, network fetch, MCP connection, plugin loading, credential/token reading, credential/token storage/output, human gates, authorizations, run status tracking, output fetch, outcome records, GrowthKernel runtime dependency, or vendor-specific adapter implementation.

## Candidate Creation Rules

Safe requests create `ExternalActionCandidate` with `candidate_status=candidate_only`. Missing capability, ambiguous target, missing action, or missing required input can create `DominionNeedsMoreInputCandidate`. Destructive or unjustified actions can create `DominionNoActionCandidate`. Credential value exposure blocks the report.

All action candidates require future control plan, target binding, static safety, preflight, Dominion gate, authorization, status tracking, and outcome record. None of those later steps are executed in v0.23.3.

## PIG/OCPX

PIG uses `version=v0.23.3`, `layer=internal_dominion`, and `subject=control_request_action_candidate`. Its safety boundary records no control plan, no preflight, no human gate, no authorization, no dispatch, no provider API call, no external runtime touch, no credential exposure, and no raw secret output.

OCPX state is `dominion_external_action_candidate_created` and targets `DominionControlRequestState`, `ExternalActionCandidateState`, `ControlTargetState`, `DominionActionIntentState`, and `DominionPreliminaryRiskState`.

## Acceptance Criteria

Control request can be created. External action candidate can be created. No-action and needs-more-input candidates can be created. Action candidates remain candidate-only. No control plan, target binding, preflight, gate, authorization, dispatch, provider API call, external runtime touch, credential storage, or credential output occurs.

## Withdrawal Conditions

Withdraw if control plan, target binding, preflight, gate, authorization, dispatch, provider API call, runtime touch, credential output, GrowthKernel active dependency, or vendor-specific core adapter logic is introduced.

## Validity Horizon

Valid until capability candidates, inventory, requested target, or Dominion policy changes.
