# v0.36.2 Dry-run Patch Apply Simulation

Parent track: Human-approved Patch Apply Sandbox.

v0.36.2 introduces in-memory dry-run patch apply simulation artifacts. It builds on v0.36.1 Apply Candidate & Human Approval Contract and v0.36.0 Patch Apply Sandbox Boundary, and it reuses v0.35.4 diff proposal metadata and v0.35.2 context/source evidence metadata as inert inputs.

v0.36.2 may parse bounded unified diff text, align hunks to supplied in-memory source images, detect conflicts, produce simulated file deltas, and produce bounded after-content previews. It does not create a sandbox workspace, write sandbox files, write live workspace files, perform sandbox patch apply, apply patches, write/edit files, call apply_patch, call git apply, execute tests, execute shell/subprocess/commands, execute/import OpenCode/Hermes/OpenClaw, invoke external coding agents, implement Dominion runtime, or implement infinite loops.

## Relationship To Prior Stages

v0.36.1 supplies `ApplyCandidateEnvelope`, `HumanApprovalContract`, and `ApplyEligibilityDecision` metadata. v0.36.2 may consume those artifacts by ID or object reference to determine whether a dry-run simulation request should proceed. Eligibility for future dry-run from v0.36.1 is not patch application permission.

v0.36.0 supplies the sandbox boundary. v0.36.2 preserves that boundary: dry-run simulation is allowed, but sandbox workspace creation and sandbox write remain closed.

v0.35.4 supplies `DiffProposalEnvelope`, `UnifiedDiffProposal`, and `StructuredPatchProposal` metadata. v0.36.2 may parse bounded unified diff artifact text in memory. Hunk alignment is not git apply, and parsed diff text is not a patch application request.

v0.35.2 supplies `PatchContextSnapshot` and `PatchContextEvidenceBundle` style source context. v0.36.2 uses only supplied `DryRunSourceImage` text. It does not read files directly.

## Definitions

Dry-run patch apply simulation is a bounded in-memory calculation that asks: if this proposed hunk were aligned against this supplied source image, what virtual after-content preview would result, and what conflicts would block future sandbox handoff?

Dry-run simulation is not patch application. In-memory simulated delta is not file write. Virtual after-content preview is not code edit. Hunk alignment is not git apply. Conflict report is not remediation execution. Successful dry-run does not grant sandbox apply permission.

In v0.36, agentic operation means bounded function/task execution metadata and future-gated single-cycle operation. v0.36.2 dry-run simulation is a bounded function over supplied strings. It is not an independent autonomous agent, not an external agent orchestration loop, not Codex-to-Claude-Code control, and not Dominion runtime.

Digestion-first / Dominion-fallback remains metadata only. Dominion awareness is not Dominion runtime.

## Taxonomies

Dry-run mode taxonomy includes unified_diff_dry_run, structured_patch_dry_run, combined_diff_dry_run, metadata_only_dry_run, blocked, no_op, and unknown.

Source taxonomy includes v0.36.1 apply candidate envelope, v0.36.1 human approval contract, v0.36.1 eligibility decision, v0.35.4 diff proposal envelope, v0.35.4 unified diff proposal, v0.35.4 structured patch proposal, v0.35.2 patch context snapshot, v0.35.2 evidence bundle, in-memory source image, test fixture, and unknown.

Status and readiness taxonomy distinguish input_validated, simulation_ready, simulation_completed, simulation_completed_with_conflicts, safe_failed, dry_run_contract_ready, dry_run_input_ready, hunk_alignment_ready, conflict_report_ready, simulated_delta_ready, and v0.36.3/v0.36.4 design handoff.

Decision and risk taxonomy distinguishes dry-run allowances from deny/block/review/future-gate outcomes. Risks include invalid candidate, missing or invalid human approval, missing/malformed diff, missing source image, stale source context, hunk alignment failure, patch conflict, scope mismatch, secret exposure, live workspace write confusion, sandbox write confusion, patch apply confusion, apply_patch, git apply, shell/test/install, external agent execution, and Dominion runtime.

Hunk alignment taxonomy includes aligned_exact, aligned_with_offset, aligned_with_fuzz, not_found, ambiguous, stale_context, blocked, skipped, and unknown. Alignment is simulation metadata only.

Conflict taxonomy includes missing_target_source, malformed_hunk, context_not_found, ambiguous_context, stale_source, overlapping_hunks, blocked_target, scope_violation, secret_target, binary_target, unsupported_operation, and unknown. Conflicts prevent future sandbox handoff unless explicitly review-gated later.

File delta taxonomy includes simulated_addition, simulated_deletion, simulated_replacement, simulated_file_create, simulated_file_delete, simulated_metadata_change, unsupported_delta, blocked_delta, no_change, and unknown. File create/delete remains simulated and must not touch the filesystem.
