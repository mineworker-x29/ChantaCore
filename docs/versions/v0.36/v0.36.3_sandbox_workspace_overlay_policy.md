# v0.36.3 Sandbox Workspace / Overlay Policy

Parent track: Human-approved Patch Apply Sandbox.

v0.36.3 builds on:
- v0.36.2 Dry-run Patch Apply Simulation.
- v0.36.1 Apply Candidate & Human Approval Contract.
- v0.36.0 Human-approved Patch Apply Sandbox Boundary Foundation.
- v0.35.4 Diff Proposal Envelope as the earlier source of proposal artifacts.

## Boundary

v0.36.3 creates sandbox workspace and overlay policy metadata only. It may create in-memory sandbox manifest artifacts, file-map entries, overlay entries, path refs, root policy, overlay policy, live workspace write block metadata, validation reports, and readiness reports.

It does not create sandbox directories. It does not materialize sandbox workspace. It does not write sandbox files. It does not write live workspace files. It does not perform sandbox patch apply. It does not apply patches. It does not write/edit files. It does not call apply_patch. It does not call git apply. It does not execute tests. It does not execute shell/subprocess/commands. It does not execute/import OpenCode/Hermes/OpenClaw. It does not invoke external coding agents. It does not implement Dominion runtime.

Sandbox workspace policy is not sandbox workspace creation. Overlay entry is not overlay materialization. Sandbox manifest is not filesystem write. Sandbox file map is not filesystem mutation. Future materialization input is not sandbox apply. Live workspace write block is not live workspace access.

## Concepts

Sandbox workspace policy defines the metadata contract for a future sandbox root and overlay plan. It describes what a later stage may materialize only after a separate future gate.

Overlay policy defines virtual overlay metadata. In v0.36.3, overlay materialization mode is `metadata_only` or `not_materialized`; actual materialization is outside this release.

Sandbox manifest is an in-memory metadata artifact that binds dry-run simulated deltas to validated sandbox path refs. It is not a directory, file, log, JSONL, OCEL packet, or persistent trace.

Sandbox file map entry maps a virtual target path to a simulated dry-run delta. It may be eligible for future materialization input, but `ready_for_write` and `ready_for_apply` remain false.

Sandbox overlay entry carries bounded simulated after-content preview metadata. It is never materialized in v0.36.3, and `materialized`, `ready_for_write`, and `ready_for_apply` remain false.

Live workspace write block records that live workspace writes, workspace writes, code edits, and patch application remain blocked.

Bounded agentic function/task interpretation remains metadata-only. v0.36.3 prepares one bounded future operation surface; it is not independent autonomous runtime, not a multi-cycle agent loop, not external agent orchestration, and not Dominion runtime.

Status/readiness:
- Status includes `policy_created`, `manifest_created`, `manifest_validated`, `manifest_validated_with_gaps`, `future_materialization_ready`, `blocked`, `review_required`, `future_gated`, `safe_failed`.
- Readiness includes `sandbox_policy_ready`, `overlay_policy_ready`, `manifest_ready`, `file_map_ready`, `live_write_block_ready`, and design handoff levels for v0.36.4/v0.36.5.
- Manifest readiness is not write readiness. Design handoff is not execution.

Decisions and risks:
- Allow decisions permit metadata only: manifest metadata, overlay policy metadata, file-map metadata, or future materialization input.
- Risks include live workspace write risk, sandbox write confusion, sandbox escape, symlink escape, outside-root write, traversal, absolute path, reference-root write, secret/credential/binary target, unresolved dry-run conflict, patch-apply confusion, shell, external agent, and Dominion risks.

Path taxonomy:
- Allowed metadata roles include sandbox root, manifest path, overlay root, virtual target/source file, and future materialization target.
- Live workspace root, reference root, secret path, credential path, binary path, external path, and unknown role are blocked for future materialization.
- Path status blocks absolute paths, traversal, outside root, symlink risk, secret/credential/binary targets, reference roots, and external roots.
- Escape kinds other than `no_escape_detected` block.

Overlay posture:
- Write mode defaults to `virtual_overlay_metadata`; `no_write` remains valid.
- Materialization mode defaults to `metadata_only`.
- Future materialization modes are handoff metadata only. No actual materialization is performed in v0.36.3.

## Safe Defaults

Default policies:
- Require relative targets.
- Block absolute targets.
- Block path traversal.
- Block symlink escape risk.
- Block outside-root paths.
- Block live workspace roots.
- Block reference roots.
- Block secret-like, credential-like, and binary-like targets.
- Allow manifest metadata, file-map metadata, overlay metadata, and future materialization input.
- Disallow directory creation, sandbox file write, live file write, patch application, apply_patch, and git apply.

Default readiness:
- `ready_for_sandbox_workspace_policy` may be true.
- `ready_for_sandbox_root_policy` may be true.
- `ready_for_overlay_policy` may be true.
- `ready_for_sandbox_workspace_manifest` may be true.
- `ready_for_sandbox_file_map` may be true.
- `ready_for_live_workspace_write_block` may be true.
- `ready_for_future_sandbox_materialization_input` may be true.
- `ready_for_v0364_sandbox_patch_apply_engine` may be true only as design-stage handoff.
- `ready_for_v0365_sandbox_post_apply_validation` may be true only as design-stage handoff.

Always false in v0.36.3:
- `ready_for_sandbox_workspace_materialization`
- `ready_for_sandbox_workspace_write`
- `ready_for_sandbox_patch_apply`
- `ready_for_live_workspace_write`

## Flow

`DryRunApplySimulationResult + ApplyCandidateEnvelope -> SandboxWorkspaceInput -> SandboxPathRefs -> SandboxFileMapEntries -> SandboxOverlayEntries -> SandboxWorkspaceManifest -> SandboxWorkspacePlan -> V0363ReadinessReport -> v0.36.4 Sandbox Patch Apply Engine`

If a dry-run result has blocking conflicts, the manifest must not be ready for future materialization input. If path validation fails, the manifest must not be ready for future materialization input. If a target points to `references/`, a live root, an absolute path, a traversal path, a secret/credential path, or a binary-like target, future materialization input must be blocked.

## Acceptance Criteria

- `SandboxRootPolicy`, `SandboxOverlayPolicy`, `SandboxWorkspacePolicy`, `SandboxWorkspaceInput`, `SandboxPathRef`, `SandboxFileMapEntry`, `SandboxOverlayEntry`, `SandboxWorkspaceManifest`, `SandboxWorkspacePlan`, and `SandboxLiveWriteBlock` exist.
- Safe relative paths are accepted.
- Absolute, traversal, reference, live, secret, credential, and binary-like paths are blocked.
- Manifest may be ready for future materialization input only when dry-run succeeded and no blocking conflicts or blocked paths exist.
- Manifest is never materialized.
- File-map and overlay entries never become write/apply ready.
- Live workspace write remains blocked.
- Focused tests pass.

## Withdrawal Conditions

Withdraw the v0.36.3 readiness claim if any of the following is observed:
- Sandbox directory creation is implemented.
- Sandbox workspace materialization is implemented.
- Sandbox or live file write is implemented.
- Sandbox patch apply or patch application is implemented.
- `apply_patch` or `git apply` is called at runtime.
- Tests, shell commands, dependency install, reference execution/import, external agent invocation, provider/network/credential access, Dominion runtime, infinite loop, persistent trace write, UI runtime, or authority grant is implemented.
- Any unsafe readiness flag listed above can be true.

## Handoff
