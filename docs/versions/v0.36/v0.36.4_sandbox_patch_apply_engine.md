## Boundary

v0.36.4 introduces the Sandbox Patch Apply Engine only. It may create sandbox directories under a validated sandbox root. It may write sandbox files under a validated sandbox root. It may perform sandbox-only patch apply by materializing dry-run/overlay after-content into sandbox files.

Sandbox patch apply is not live patch apply. Sandbox file write is not live workspace write. Internal sandbox apply is not `git apply` or `apply_patch`. Sandbox apply success is not test success. Sandbox apply success is not live apply permission. Sandbox apply result is not production readiness.

v0.36.4 does not write live workspace files. It does not edit live code. It does not perform unrestricted patch application. It does not call `apply_patch`. It does not call `git apply`. It does not execute tests. It does not execute shell/subprocess/commands. It does not install dependencies. It does not execute/import OpenCode/Hermes/OpenClaw. It does not invoke external coding agents. It does not implement Dominion runtime, autonomous runtime, or multi-cycle agentic loops.

## Allowed Sandbox Behavior

The only permitted filesystem effects are:
- Creating sandbox directories under an explicitly provided, validated sandbox root.
- Writing sandbox files under that same validated sandbox root.
- Writing content supplied by `SandboxWorkspaceManifest`, `SandboxOverlayEntry`, or dry-run simulated after-content metadata.

All writes must validate containment immediately before the directory creation or file write. Target paths must be relative. Traversal, absolute targets, reference roots, live roots, secret-like paths, credential-like paths, binary-like paths, and symlink escapes are blocked before writing.

## Models

`SandboxPatchApplyPolicy` permits sandbox directory creation, sandbox file write, and sandbox patch apply by default. It blocks live workspace write, unrestricted patch application, workspace write, code edit, `apply_patch`, `git_apply`, test execution, shell, dependency install, external agent execution, and Dominion runtime.

`SandboxPatchApplyInput` is a sandbox-only apply request. It must include a sandbox root ref and prohibited runtime actions.

`SandboxMaterializationPlan` contains directory and file plans. It may be ready for sandbox materialization, but never live write or execution.

`SandboxDirectoryMaterializationPlan` describes sandbox directory creation under root only.

`SandboxFileMaterializationPlan` describes a sandbox file write from overlay/dry-run after-content metadata. It may be ready for sandbox write but never live write.

`SandboxFileWriteOperation` carries content and target path for a sandbox-only write. `live_write` is always false.

`SandboxFileWriteRecord` records sandbox-only write status, bytes written, sandbox root ref, and normalized sandbox path. It must never report live write or outside-sandbox write.

`SandboxPatchEngineOperation` records internal engine strategy and file write operations. It must always report `used_apply_patch=False`, `used_git_apply=False`, `used_shell=False`, and `live_write=False`.

`SandboxPatchEngineFileResult` records per-file sandbox write result. It is never ready for live apply.

`SandboxPatchApplyResult` may be successful, but only as sandbox apply. It is not live apply, not test success, and not live apply permission.

`SandboxPatchApplyNoLiveWriteGuarantee` preserves no-live-write guarantees. `no_sandbox_write` may be false when sandbox writes occurred under policy; all live/external/runtime prohibitions remain true.

## Flow

`SandboxWorkspaceManifest + SandboxWorkspacePlan + DryRunApplySimulationResult + ApplyCandidateEnvelope -> SandboxPatchApplyInput -> MaterializationPlan -> SandboxFileWriteOperations -> SandboxPatchApplyResult -> V0364ReadinessReport -> v0.36.5 Sandbox Post-Apply Validation`

The engine writes only after:
1. Policy allows sandbox materialization/write/apply.
2. Sandbox root is explicitly provided and safe.
3. Target path is relative and normalized.
4. Target path resolves under sandbox root.
5. Target is not live/reference/secret/credential/binary-like.
6. Symlink resolution does not escape the sandbox root.

- Sandbox-only directory creation works under validated sandbox root.
- Sandbox-only file write works under validated sandbox root.
- `SandboxPatchApplyResult` can be successful.
- No live workspace write.
- No outside sandbox write.
- No references root write.
- No `apply_patch`.
- No `git apply`.
- No shell/subprocess/command.
- No test execution runtime.
- No dependency install.
- No external agent or Dominion runtime.
- Focused tests pass.

## Withdrawal Conditions

Withdraw v0.36.4 readiness if:
- A helper writes outside sandbox root.
- A helper writes live workspace files.
- A helper edits live code.
- `apply_patch`, `git apply`, shell, subprocess, tests, dependency install, provider/network, credential reads, external agent invocation, or Dominion runtime is introduced.
- Successful sandbox apply is represented as test success or live apply permission.
- Any unsafe readiness flag can be true.
