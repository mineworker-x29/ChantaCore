# v0.27.5 Durable Memory Record & Registry

Version: `v0.27.5`
Name: Durable Memory Record & Registry
Korean name: Durable Memory Record·Registry
Track: Memory Candidate & Continuity
Previous: `v0.27.4` Memory Promotion Gate
Next: `v0.27.6` Session Continuity Context Builder

## Objective

`v0.27.5` introduces the gated durable memory record and registry layer. It reads
promotion gate decisions from `v0.27.4`, evidence and scores from `v0.27.3`,
candidate refs from `v0.27.2`, and source refs from `v0.27.1`.

Durable memory record creation requires a prior promotion decision. A promote
decision alone is not durable memory. Persistent memory write requires a write
gate, including release hygiene and runtime data hygiene gates. If the hygiene
status is missing or failed, `v0.27.5` may create write policies, registry
policies, write gates, write decisions, blocked-write records, dry-run records,
record previews, audit trails, integrity reports, and readiness reports, but it
must not perform persistent memory write.

## Core Principles

- Durable memory record requires a prior promotion decision.
- Promote decision alone is not durable memory.
- Durable memory registry is not persona.
- Durable memory registry is not behavior policy.
- Persistent memory write requires a write gate.
- No write is a valid outcome.
- Blocked write is a valid outcome.
- Dry-run registry materialization is allowed when the write gate is not satisfied.
- Durable memory must be scoped, evidenced, versioned, auditable, revocable, forgettable, and OCEL-visible.
- PIG guidance is not memory authority.
- Raw transcript and raw provider output must not be written as memory.
- Durable memory records must not trigger provider invocation, command execution, continuity injection, or safety bypass.

## Relationship To Earlier v0.27 Versions

`v0.27.4` records promotion gate decisions only. `v0.27.5` consumes those
decision records and durable-readiness previews. It does not treat a promote
decision as durable memory by itself.

`v0.27.3` provides evidence bundles, score refs, privacy assessments, and
contradiction checks. `v0.27.5` indexes those refs but does not store raw
transcripts, raw provider outputs, secrets, credentials, or private paths.

`v0.27.0` remains the contract boundary. The release hygiene prerequisite from
`v0.26.10` is required before persistent memory write can be marked as passed.

## Data Models

The release defines durable write policies, registry policies, write requests,
write gates, write decisions, blocked-write records, dry-run records, durable
record previews, durable memory records, registries, registry entries, scope,
provenance, evidence indexes, lifecycle policies, version metadata, status
records, privacy boundaries, conflict markers, forget/revoke bindings, audit
trails, integrity reports, findings, and registry reports.

Durable memory records are refs-only. They include:

- candidate ref
- promotion decision record ref
- scope
- provenance
- evidence index
- lifecycle policy
- version metadata
- privacy boundary
- conflict marker
- forget/revoke binding

## Service Design

The registry service evaluates prerequisites through refs and summaries only.
The write gate checks promotion decision, evidence, score, source refs, scope,
lifecycle or expiry, forget/revoke path, audit, provenance, evidence index,
privacy boundary, contradiction review, release hygiene, runtime data hygiene,
and raw-memory blockers.

If all durable refs pass but hygiene gates are missing, the service emits a
dry-run decision and dry-run record. If mandatory refs are missing, it emits a
blocked write record. A durable record and registry entry are created only when
the write gate and hygiene gates pass.

## OCEL Mapping

Object types include durable write policy, registry policy, write request, write
gate, write decision, blocked/dry-run records, durable record preview, durable
memory record, durable registry, registry entry, record scope, provenance,
evidence index, lifecycle policy, version, status, privacy boundary, conflict
marker, forget/revoke binding, audit trail, integrity report, finding, report,
promotion gate report, promotion decision record, PIG report, OCPX projection,
and execution envelope.

Allowed effects are read-only observation, write-gate evaluation, write-decision
recording, dry-run record creation, blocked-write record creation, gated durable
record creation, gated registry entry creation, gated registry update, gated
persistent memory write, provenance creation, evidence-index creation, lifecycle
policy creation, audit creation, and state-candidate creation.

Conditional effects are durable memory record creation, registry update, and
persistent memory write. They are allowed only when the write gate passes; for
persistent memory write, release and runtime data hygiene gates must also pass.

Forbidden effects include write without gate, write without promotion decision,
write without evidence/source/scope/forget-revoke/audit, persistent write without
hygiene, continuity context creation, continuity injection, persona mutation,
behavior policy mutation, raw transcript/provider/secret/credential memory, PIG
authority or execution, provider invocation, command execution, safety bypass,
external adapters, Schumpeter split, and sole LLM authority.

asserted by the CLI.

## PIG And OCPX

The PIG report subject is `durable_memory_record_registry`, with conditional
durable record, registry update, and persistent memory flags. Session continuity
injection, persona mutation, behavior policy mutation, raw memory, PIG execution,
provider invocation, command execution, safety bypass, external adapters,
Schumpeter split, and LLM judge authority remain false.

The OCPX state is `durable_memory_record_registry_created`, with target read
models for write gate, write decision, durable record, registry, registry entry,
provenance, evidence index, lifecycle policy, audit, and `V027ReadinessState`.

## Non-Goals

`v0.27.5` does not build session continuity context, inject continuity, mutate
persona or behavior policy, persist raw transcripts/provider outputs/secrets as
memory, treat PIG as memory authority, invoke providers, execute commands, bypass
safety gates, implement external adapters, introduce Schumpeter split, or use an
LLM judge as sole durable memory authority.

## Acceptance Criteria

- Durable write and registry policies can be created.
- Write gate can be evaluated.
- Write decisions can be recorded.
- Blocked-write and dry-run records can be created.
- Durable record previews can be created.
- Durable records and registry entries are created only when write gate and hygiene gates pass.
- Scope, provenance, evidence index, lifecycle, version, status, privacy, conflict, and forget/revoke binding artifacts can be created.
- Registry integrity report can be created.
- Persistent write is blocked when release/runtime hygiene gates are missing.
- Raw transcript/provider output/secret/credential memory is not written.
- Persona and behavior policy are not mutated.
- Continuity is not injected.
- Providers and commands are not invoked.
- Safety gates are not bypassed.
- PIG memory/policy/execution is not introduced.
- External adapters and Schumpeter split are not introduced.
- LLM judge is not sole durable memory authority.

## Withdrawal Conditions

Withdraw readiness if a durable record or registry entry is created without a
promotion decision, evidence, source refs, scope, lifecycle or expiry,
forget/revoke path, audit, or write gate. Withdraw persistent-write readiness if
release or runtime data hygiene gates are missing. Withdraw the release if raw
memory, persona/policy mutation, continuity injection, PIG authority, provider or
command execution, safety bypass, external adapter, Schumpeter split, or sole LLM
authority appears.

## Validity Horizon

Valid until `v0.27.6` Session Continuity Context Builder begins or durable memory
registry policy changes.
