# v0.23.0 OCEL-native Internal Dominion Contract

Korean name: OCEL-native 내부 지배 계약

Track: Internal Dominion Foundation

## Definition

v0.23.0 introduces Internal Dominion as the third internal skill category after Observation and Digestion.

Internal Dominion is not arbitrary control. It is not a vendor adapter. It is not execution by default. It is not self-execution. Internal Dominion is a vendor-neutral, OCEL-native, gated control grammar for governing external runtimes, agents, tools, workflows, RPA systems, APIs, local runtimes, and enterprise systems through visible inventory, observation, digestion, control planning, authorization, status tracking, and outcome records.

This release is contract-only. It does not dispatch, touch external runtimes, call provider APIs, execute local commands, call network resources, connect MCP, load plugins, store credentials, or integrate GrowthKernel as a runtime dependency.

## Policy Change

v0.23.x is no longer Self-Execution Safety. v0.23.x is Internal Dominion Foundation.

Self-Execution Safety is reclassified to v0.24.x Local Runtime Provider. Bounded command execution and local runtime execution planning belong to that future provider track, not to v0.23.0.

A360, Automation Anywhere, Brity, UiPath, Power Automate, and other vendor systems are future external provider skills or provider adapters. They are not Internal Dominion core logic.

GrowthKernel is a future consumer / future optimizer. GrowthKernel is not an active runtime dependency, not a release blocker, and not required for ChantaCore or Schumpeter to own OCEL trace, outcome, and workbench state.

Schumpeter is the company deployment name for ChantaCore Company Edition. Schumpeter Legacy is a previous prototype / asset source, not a runtime dependency.

## Internal Skill Taxonomy

The internal skill taxonomy is:

- observation
- digestion
- dominion

Dominion extends Observation and Digestion by specifying the grammar that any future external-control path must satisfy before a provider adapter may act. Observation sees. Digestion interprets. Dominion governs possible control through gates. It does not bypass gates and does not itself become a vendor adapter.

## Provider Interface Boundary

Provider types are local runtime, RPA runtime, agent runtime, workflow engine, browser automation, enterprise API, database or ETL, and custom system.

Required provider methods are `discover_runtimes`, `list_capabilities`, `describe_capability`, `validate_action`, `preflight`, `dispatch`, `get_status`, `fetch_output`, `cancel_or_stop`, and `map_outcome`.

The provider interface is a contract only. Providers cannot bypass gate, cannot store credentials in output, must return OCEL references, must support status tracking, and must support outcome mapping. v0.23.0 implements no provider call.

## Gate Contract

Dominion requires runtime inventory, capability observation, capability digestion, control request, action candidate, control plan, static safety check, preflight before dispatch, human gates for mutating and production actions, single-use authorization, idempotency key, rate-limit policy, cancel/stop plan, status tracking, and outcome record.

Dominion denies control when credential exposure risk exists, provider bypasses the gate, runtime is not allowlisted, or capability is not allowlisted.

## Effect Policy

Allowed v0.23.0 effect types:

- `read_only_observation`
- `state_candidate_created`

Future effect types include external runtime observation, action candidates, control plans, gate creation, runtime touch, control dispatch, run start, status observation, output capture, and outcome recording.

Forbidden v0.23.0 effects include `external_runtime_touched`, `external_control_dispatched`, `external_run_started`, `network_accessed`, `plugin_loaded`, `mcp_connected`, `shell_executed`, `workspace_file_changed_by_command`, and `credential_exposed`.

## Service Design

- `InternalDominionRegistryService` lists subjects, seed skill IDs, skill contracts, and builds the contract.
- `DominionMigrationAuditService` scans public code/docs/tests for superseded self-execution wording, GrowthKernel active dependency wording, vendor hardcoding, and provider bypass risk.
- `DominionMigrationRemediationService` converts migration findings into fixed remediation records without mutating files.
- `DominionConformanceService` checks contract-only status, dispatch disablement, provider API disablement, GrowthKernel non-dependency, vendor-neutral definition, and OCEL-native mapping.
- `InternalDominionReportService` builds the contract report, PIG report, OCPX projection, and CLI renderings.

## Migration / Audit / Remediation

Existing v0.23 Self-Execution Safety planning is superseded. Public docs and registry wording should describe v0.23.x as Internal Dominion Foundation and v0.24.x as Local Runtime Provider.

GrowthKernel wording must not say required dependency, bridge required, active dependency, runtime dependency, or optimizer required. It must say future consumer / future optimizer.

Vendor names may appear in public docs and tests only as future external provider adapter examples. They must not appear as core/internal Dominion runtime logic.

Provider bypass language such as dispatch without gate, provider direct run, control without authorization, or external control dispatch before authorization must be detected as a migration risk and remain blocked in v0.23.0.
