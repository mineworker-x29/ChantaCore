# v0.20.5 Self-Project Structure Awareness

Korean name: 자기 프로젝트 구조 자각

Track: OCEL-native Self-Awareness

## Version Definition

v0.20.5 introduces deterministic, metadata-only project surface structure awareness for public ChantaCore.

Self-awareness is not self-modification.

Project structure candidates are not canonical memory.

## Objective

ChantaCore can create a project surface snapshot from workspace inventory metadata. The snapshot records workspace-relative tree nodes, file kind distribution, and heuristic surface candidates such as README, docs root, source root, test root, config file, dependency manifest, entrypoint file, package root, script file, and CI config.

## Relationship To Earlier Versions

- v0.20.0 introduced the self_awareness layer contract.
- v0.20.1 introduced workspace path verification and metadata-only inventory.
- v0.20.2 introduced bounded text/code perception and redaction.
- v0.20.3 introduced bounded literal workspace search.
- v0.20.4 introduced deterministic markdown/python structure summarization and summary candidates.

v0.20.5 reuses the v0.20.1 path policy and inventory service. It does not need file content for tree construction.

## Deterministic Surface Mapping

This version is deterministic project surface mapping, not architecture inference. It classifies visible project surfaces by path, name, suffix, size, depth, and directory/file metadata. It does not infer runtime architecture, resolve imports, build dependency graphs, create call graphs, or summarize project semantics.

## Included Scope

- `skill:self_awareness_project_structure`
- Metadata-only project tree snapshot
- File kind distribution
- Surface candidate detection
- Candidate-only project structure output
- OCEL/PIG/OCPX visibility
- Read-only gate/envelope invocation

## Non-Goals

- LLM architecture inference
- Semantic project summary
- Embedding or vector project summary
- Dependency graph
- Import resolution
- Call graph
- Runtime introspection
- Test execution
- Package install, build, or lint
- Shell execution
- Network/web fetch
- MCP connection
- Plugin loading
- External harness execution
- Sidecar/event bus
- File write, edit, or patch application
- Memory, persona, or overlay mutation
- Canonical promotion of project structure candidates

## Data Models

- `SelfProjectStructureRequest`
- `SelfProjectStructurePolicyDecision`
- `ProjectTreeNode`
- `ProjectSurfaceCandidate`
- `ProjectFileKindDistribution`
- `SelfProjectStructureCandidate`

`ProjectTreeNode` contains metadata only: workspace-relative path, type, depth, suffix, file kind, size, child count, hidden flag, and excluded flag. It does not contain file content.

`SelfProjectStructureCandidate` defaults to `review_status=candidate_only`, `canonical_promotion_enabled=false`, and `promoted=false`.

## Service Design

- `SelfProjectStructurePolicyService` reuses workspace path policy and blocks traversal, outside workspace, private boundary, missing path, and non-directory targets.
- `ProjectTreeSnapshotService` reuses metadata-only workspace inventory.
- `ProjectSurfaceCandidateDetector` detects candidates by deterministic path/name/suffix heuristics.
- `ProjectFileKindDistributionService` counts file kinds, suffixes, files, directories, and top-level path buckets.
- `SelfProjectStructureAwarenessSkillService` exposes the read-only skill service.

## Surface Candidate Detection Policy

- README: `README`, `README.md`, `README.rst`, `README.txt`
- Docs root: `docs`, `doc`, `documentation`
- Source root: `app`, `src`, `lib`, `packages`, `services`
- Test root: `tests`, `test`, `specs`, `spec`
- Dependency manifest: `pyproject.toml`, `package.json`, `requirements.txt`, `poetry.lock`, `pnpm-lock.yaml`, `yarn.lock`, `Cargo.toml`, `go.mod`, `pom.xml`, `build.gradle`
- Config candidate: `.yaml`, `.yml`, `.toml`, `.ini`, `.cfg`, `.json`, `.jsonc`, `.env.example`
- Entrypoint candidate: `main.py`, `app.py`, `cli.py`, `__main__.py`, `index.ts`, `index.js`, `server.py`
- CI config: `.github/workflows`, `.gitlab-ci.yml`, `azure-pipelines.yml`

Config and test surfaces are helper candidates only. Their public skills remain contract-only and non-executable in v0.20.5.

## File Kind Distribution Policy

Distribution is computed from inventory metadata. It records total files, total directories, suffix counts, file-kind counts, and top-level path counts. It does not read config values, code bodies, markdown bodies, or secret files.

## Candidate Semantics

Project structure candidates are evidence-bearing review candidates. They are not canonical memory. They are not promoted automatically. They do not mutate workspace, memory, persona, overlay, external systems, or capability state.

## CLI Commands

- `python -m chanta_core.cli self-awareness project structure`
- `python -m chanta_core.cli self-awareness project structure --path .`
- `python -m chanta_core.cli self-awareness project structure --max-depth 4 --max-entries 1000`
- `python -m chanta_core.cli self-awareness project structure --no-summary-candidates`

CLI output shows workspace-relative paths, candidate id, candidate status, promotion flags, bounded tree preview, distribution summary, surface candidates, and `content_printed=false`.

## PIG/OCPX Report Update

PIG reports:

- `project_structure=implemented_surface_snapshot`
- `config_surface=candidate_helper_only`
- `test_surface=candidate_helper_only`
- `llm_architecture_inference_enabled=false`
- `dependency_graph_enabled=false`
- `runtime_introspection_enabled=false`
- `canonical_promotion_enabled=false`

OCPX projection state is `project_surface_structure_awareness`. Candidate types include `project_structure_candidate` and `project_surface_candidate`.

## Tests

Tests cover candidate creation, workspace-relative paths, file distribution, surface detection, limits, blocked paths, hidden/noisy exclusions, metadata-only tree building, candidate-only semantics, read-only safety flags, CLI, PIG/OCPX, and regressions for earlier self-awareness layers.

## Acceptance Criteria

- Project surface structure candidate builds from metadata-only inventory.
- Output is workspace-relative.
- Surface candidates are heuristic and deterministic.
- Config/test surfaces are helper candidates only and public skills remain non-executable.
- No dependency graph, import resolution, runtime introspection, or LLM architecture inference is implemented.
- No file content is exposed.
- Project structure candidate is candidate-only and not promoted.
- No write/shell/network/MCP/plugin/external harness execution occurs.
- No memory/persona/overlay mutation occurs.
- OCEL/PIG/OCPX visibility exists.
- v0.20.0 through v0.20.4 tests still pass.

## Withdrawal Conditions

Withdraw v0.20.5 if implementation reads file content during tree building, exposes raw private/secret content, emits full private paths, creates dependency graphs or import resolution, invokes LLM architecture reasoning, mutates workspace or memory/persona/overlay, or enables shell/network/MCP/plugin/external harness execution.
