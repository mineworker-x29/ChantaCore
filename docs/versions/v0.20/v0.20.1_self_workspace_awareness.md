# ChantaCore v0.20.1 - Self-Workspace Awareness

## Version Definition

- Version: `0.20.1`
- Version name: `Self-Workspace Awareness`
- Korean name: `자기 워크스페이스 자각`
- Track: `OCEL-native Self-Awareness`
- Public boundary: generic ChantaCore workspace/path/inventory concepts only.

## Objective

v0.20.1 implements the first executable read-only part of the Self-Awareness Layer. ChantaCore can now contemplate its own workspace boundary and workspace structure through metadata-only path verification and inventory reports.

This is self-awareness, not generic file listing. The runtime is not gaining a broad file tool; it is exposing a constrained evidence surface that describes whether paths are inside the workspace and what metadata-only structure is visible under that boundary.

Self-awareness is not self-modification.

## Relationship To v0.20.0

v0.20.0 introduced the `self_awareness` layer contract, seed skill IDs, conformance service, report skeletons, and OCEL mapping surface. v0.20.1 preserves that contract and implements only:

- `skill:self_awareness_workspace_inventory`
- `skill:self_awareness_path_verify`

All other self-awareness seed skills remain `contract_only` and non-executable.

## Included Scope

- Workspace root aliases.
- Workspace-relative path verification.
- Traversal and outside-workspace blocking.
- Configured private-boundary blocking.
- Metadata-only directory/file inventory.
- Depth and entry count limits.
- Hidden and noisy path exclusion by default.
- Read-only execution gate compatibility.
- OCEL/PIG/OCPX visibility for workspace awareness coverage.

## Non-Goals

v0.20.1 does not implement file content reading, code search, symbol parsing, AST parsing, content hashing, project semantic inspection, shell execution, network fetch, MCP connection, plugin loading, external harness execution, sidecar/event bus behavior, workspace mutation, memory mutation, persona mutation, overlay mutation, or plan/todo materialization.

File content reading is not part of v0.20.1.

## Data Models

- `WorkspaceRootRef`: root alias, display name, root path, canonical path, source, primary flag, and private-boundary flag.
- `WorkspacePathResolution`: input path, normalized path, canonical path, root id, workspace membership, blocked status, finding type, and reason.
- `WorkspaceInventoryRequest`: root id, relative path, depth/entry limits, hidden/ignored inclusion flags, and file/directory inclusion flags.
- `WorkspaceInventoryEntry`: workspace-relative path, entry type, suffix, size metadata, modified time metadata, depth, hidden flag, and exclusion fields.
- `WorkspaceInventoryReport`: report id, requested path, resolved path, entries, counts, truncation status, exclusions, blocked count, warnings, and evidence refs.

Inventory entries contain metadata only. They do not contain content previews, file bodies, parsed code, hashes, or search hits.

## Service Design

- `SelfWorkspacePathPolicyService`
  - `list_workspace_roots()`
  - `resolve_path(input_path, root_id=None)`
  - `assert_readonly_metadata_allowed(resolution)`
- `SelfWorkspaceInventoryService`
  - `build_inventory(request)`
- `SelfWorkspaceAwarenessSkillService`
  - `verify_path(input_path, root_id=None)`
  - `inventory_workspace(request)`

The skill invocation path uses the existing explicit read-only execution gate and explicit skill invocation service. Direct service calls remain useful for deterministic tests, but the CLI-facing skill route goes through the gate.

## Path Policy

Paths are normalized against an explicit workspace root. Path traversal, absolute-path access, outside-workspace resolution, configured private boundaries, and symlink traversal are blocked with structured results.

The policy never creates permission grants and never mutates policy state.

## Inventory Semantics

Inventory uses metadata APIs only. It may inspect directory entries and stat metadata. It must not read file contents, compute content hashes, parse source code, or search inside files.

Default noisy exclusions:

- `.git`
- `.venv`
- `__pycache__`
- `node_modules`
- `.pytest_cache`
- `.mypy_cache`
- `.ruff_cache`
- `dist`
- `build`

Hidden paths are excluded by default unless `include_hidden=true`. Paths in output are workspace-relative.

## OCEL Mapping

Additional object types include `workspace_root`, `workspace_directory`, `workspace_inventory_report`, and `path_policy_decision`.

Additional event types include path verification requested/completed/blocked and inventory requested/completed/blocked/truncated events.

Additional relation types include `belongs_to_workspace`, `resolves_to_path`, `contains_path`, `contains_file`, `contains_directory`, `blocked_by_policy`, `produces_inventory_report`, `recorded_in_envelope`, and `derived_from_request`.

OCEL remains the canonical process substrate. JSONL is not introduced as a canonical store.

## CLI Commands

- `python -m chanta_core.cli self-awareness workspace roots`
- `python -m chanta_core.cli self-awareness workspace verify-path .`
- `python -m chanta_core.cli self-awareness workspace inventory .`
- `python -m chanta_core.cli self-awareness workspace inventory . --max-depth 2 --max-entries 200`

CLI output uses workspace aliases and workspace-relative paths. It must not print private full paths or file contents.

## PIG/OCPX Report Update

The PIG report includes workspace awareness coverage, safety counts, diagnostics, and a recommendation to proceed to v0.20.2 only if conformance passes.

The OCPX projection marks path verification and workspace inventory as implemented. Text read, search, symbol scan, and project semantic inspection remain not implemented.

## Tests

Acceptance tests cover root listing, path verification, traversal blocking, outside-workspace blocking, private-boundary blocking, metadata-only inventory, depth and entry limits, hidden/noisy exclusion, no content reading, no subprocess/shell use, read-only effect type, unchanged mutation/execution safety flags, CLI coverage, PIG/OCPX coverage, and v0.20.0 contract preservation.

## Acceptance Criteria

- Workspace path verification works through the read-only gate.
- Workspace inventory works through the read-only gate.
- No file contents are read.
- No workspace, memory, persona, or overlay mutation occurs.
- No shell, network, MCP, plugin, or external harness execution occurs.
- Traversal and outside-workspace paths are blocked.
- Inventory output is workspace-relative and metadata-only.
- OCEL/PIG/OCPX visibility exists.
- v0.20.0 self-awareness contract tests still pass.

## Withdrawal Conditions

Withdraw v0.20.1 acceptance if inventory reads file contents, hashes file contents, parses code, searches file bodies, mutates workspace or canonical state, mutates memory/persona/overlay, uses shell/network/MCP/plugin/external harness execution, exposes private material, or bypasses the read-only gate for user-facing execution.
