# v0.20.8 Self-Awareness Workbench

Korean name: 자각 워크벤치

Track: OCEL-native Self-Awareness

## Version Definition

v0.20.8 defines the Self-Awareness Workbench, a read-only operator surface for
Public ChantaCore self-awareness outputs.

Self-awareness is not self-modification.

Workbench is not execution.

Workbench is not approval.

Workbench is not promotion.

## Objective

The workbench builds a sanitized snapshot over self-awareness registry coverage,
safety boundary counts, candidate queues, verification queues, recent envelope
summaries, PIG status, OCPX status, and workbench findings.

## Relationship To v0.20.0 Through v0.20.7

- v0.20.0 introduced the self-awareness layer contract.
- v0.20.1 introduced workspace inventory and path verification.
- v0.20.2 introduced bounded text/code perception.
- v0.20.3 introduced bounded literal workspace search.
- v0.20.4 introduced deterministic structure summarization.
- v0.20.5 introduced project surface structure awareness.
- v0.20.6 introduced self-surface verification.
- v0.20.7 introduced candidate-only self-directed intention bundles.
- v0.20.8 adds a read-only operator read model over those outputs.

## Included Scope

- Registry view.
- Version coverage rows.
- Safety boundary counters.
- Candidate queue counts and sanitized refs.
- Verification queue counts and sanitized refs.
- Recent execution envelope summaries.
- PIG and OCPX status summaries.
- Workbench findings and status.

## Non-Goals

v0.20.8 does not implement candidate approval, candidate promotion, task queue
creation, plan execution, todo materialization, file write/edit/apply_patch,
shell/test/lint execution, network/web fetch, MCP connection, plugin loading,
external harness execution, memory/persona/overlay mutation, LLM judge, LLM
summary, or runtime introspection execution.

## Data Models

- `SelfAwarenessWorkbenchRequest`
- `SelfAwarenessCoverageRow`
- `SelfAwarenessSafetyBoundaryStatus`
- `SelfAwarenessCandidateQueueView`
- `SelfAwarenessVerificationQueueView`
- `SelfAwarenessEnvelopeView`
- `SelfAwarenessWorkbenchFinding`
- `SelfAwarenessWorkbenchSnapshot`

The snapshot defaults are `read_only=true` and `mutation_performed=false`.

## Service Design

- `SelfAwarenessCoverageService` builds registry-derived coverage rows.
- `SelfAwarenessSafetyBoundaryService` summarizes conformance safety counters.
- `SelfAwarenessCandidateQueueService` builds sanitized candidate count/ref views.
- `SelfAwarenessVerificationQueueService` builds sanitized verification report views.
- `SelfAwarenessEnvelopeAuditService` builds sanitized envelope summaries.
- `SelfAwarenessWorkbenchService` builds the final snapshot.

## Status Rules

- Critical or error findings produce `violation`.
- Warning findings produce `warning`.
- Only info/ok findings produce `ok`.
- Dangerous capability count greater than zero produces `violation`.
- Write/shell/network/MCP/plugin/external harness enabled counts greater than
  zero produce `violation`.
- Memory/persona/overlay mutation counts greater than zero produce `violation`.
- Promoted or materialized candidate counts greater than zero produce `violation`.

## Safety Boundary Counters

The workbench exposes write, shell, network, MCP, plugin, external harness,
memory, persona, overlay, canonical promotion, materialization, read-only
invocation, dangerous execution, and dangerous capability counts.

## Queue Views

Candidate and verification queues are count/ref views. They do not print raw
candidate bodies, raw file content, raw secrets, private material, or full
private paths.

## Envelope Audit View

Recent envelope rows include envelope id, skill id, event type, effect types,
blocked state, created time, sanitized summary, and sanitized object refs only.

## OCEL Mapping

v0.20.8 extends the self-awareness mapping with workbench snapshot, coverage
row, safety boundary status, candidate queue view, verification queue view,
envelope view, workbench finding, PIG report, and OCPX projection object types.
It adds workbench requested/snapshot/viewed/warning/violation event types and
view/summarization/finding relation types. OCEL remains the canonical substrate;
JSONL is not introduced as a canonical store.

## PIG/OCPX Report Update

PIG includes `workbench=implemented_read_only_operator_surface`,
`read_only=true`, `mutation_performed=false`, `approval_enabled=false`,
`promotion_enabled=false`, and `materialization_enabled=false`.

OCPX uses state `self_awareness_workbench_read_model`, includes workbench read
model types, and keeps `read_only_observation` visible.

## Tests

Tests cover snapshot creation, read-only defaults, coverage from v0.20.0 through
v0.20.7, safety counters, candidate and verification queue counts, sanitized
envelope refs, status rules, CLI sections, PIG/OCPX visibility, and regression
boundaries.

## Acceptance Criteria

- Workbench snapshot builds.
- Workbench is read-only.
- Workbench does not approve, promote, materialize, or execute anything.
- Workbench shows coverage, safety counts, candidates, verification reports,
  recent envelopes, and PIG/OCPX status.
- Workbench output is sanitized.
- No raw secrets, private full paths, or raw file content are shown.
- No write/shell/network/MCP/plugin/external harness execution occurs.
- No memory/persona/overlay mutation occurs.
- OCEL/PIG/OCPX visibility exists.
- v0.20.0 through v0.20.7 tests still pass.

## Withdrawal Conditions

This version should be withdrawn if the workbench approves, promotes,
materializes, executes, mutates workspace or memory/persona/overlay state,
invokes LLM judge/summary behavior, loads plugins, connects to MCP, performs
network fetches, prints raw file content, prints raw secrets, or exposes private
full paths.
