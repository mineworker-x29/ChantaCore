# v0.20.3 Self-Code Search Awareness

Korean name: 자기 코드 검색 자각

Track: OCEL-native Self-Awareness

## Version Definition

v0.20.3 introduces bounded literal workspace search for the public ChantaCore Self-Awareness Layer.
Self-awareness is not self-modification.

This version lets ChantaCore inspect its own workspace through policy-gated, bounded, redacted, internal literal text search. It builds on v0.20.0 contracts, v0.20.1 workspace path and inventory policy, and v0.20.2 bounded text perception and redaction.

## Objective

The objective is search awareness, not shell grep, not semantic search, not summary, not symbol analysis, and not mutation. The runtime can answer whether a literal string appears in allowed text-like workspace files, while preserving read-only boundaries and returning workspace-relative evidence.

## Included Scope

- `skill:self_awareness_workspace_search`
- Literal match mode only
- Case-insensitive matching by default
- Optional case-sensitive matching
- Workspace-relative search root
- Include and exclude glob filters
- Bounded candidate count, match count, per-file bytes, total bytes, and context lines
- Redacted snippets only
- Structured skipped or blocked candidate findings
- PIG and OCPX visibility

## Non-Goals

- External command-line text search
- Regex search
- Semantic retrieval
- AST parsing
- Python symbol parsing
- Markdown summary
- Deterministic or LLM summary
- File write/edit/delete
- Shell, network, MCP, plugin, or external harness execution
- Memory, persona, or overlay mutation
- Plan or todo materialization

## Data Models

- `SelfWorkspaceSearchRequest`
- `SelfWorkspaceSearchPolicyDecision`
- `SelfSearchFileCandidate`
- `SelfSearchMatchSnippet`
- `SelfSearchMatch`
- `SelfWorkspaceSearchResult`

The request carries the query, root id, relative path, glob filters, case sensitivity, literal match mode, context lines, and all search budgets. The result carries candidate counts, scanned/skipped/blocked file counts, matches, truncation state, skipped candidates, evidence refs, blocked status, and warnings.

## Service Design

- `SelfWorkspaceSearchPolicyService` validates query, match mode, and path boundary.
- `SelfSearchCandidateService` reuses v0.20.1 inventory and v0.20.2 text read policy to collect allowed text candidates.
- `SelfWorkspaceLiteralSearchService` reuses v0.20.2 bounded text perception to read only approved slices before literal matching.
- `SelfCodeSearchAwarenessSkillService` coordinates policy, candidate collection, search, redaction, and result construction.

## Search Policy

The policy blocks empty queries, overlong queries, unsupported match modes, traversal paths, outside-workspace paths, and private boundaries. v0.20.3 supports only `literal` mode. Hidden files are excluded by default. Noisy directories are excluded through existing inventory policy. Secret-like and binary files are skipped or blocked by the text read policy.

## Candidate Selection Policy

Candidates come from metadata inventory, not an unbounded workspace scan. Include and exclude globs are applied before reading. Candidate collection stops at the effective `max_files` limit and records truncation if the limit or inventory budget is reached.

## Read Budget Policy

Search applies:

- `max_files`
- `max_matches`
- `max_matches_per_file`
- `max_bytes_per_file`
- `max_total_bytes`
- `context_lines`

When a budget is reached, the result sets `truncated=true` and records `truncated_reason`.

## Redaction Policy

Search snippets are built from redacted v0.20.2 text slices. Token, password, API key, secret-like text, private key headers, bearer-style tokens, GitHub-style tokens, high-entropy credential-like text, and private path literals are redacted before output. Blocked files never return content.

## CLI Commands

- `python -m chanta_core.cli self-awareness search "query"`
- `python -m chanta_core.cli self-awareness search "query" --path src --include "*.py"`
- `python -m chanta_core.cli self-awareness search "query" --case-sensitive`
- `python -m chanta_core.cli self-awareness search "query" --context-lines 3`

CLI output shows query, mode, files scanned/skipped/blocked, match count, truncation, effect type, and redacted snippets. It does not print private full paths.

## Tests

Tests cover literal match behavior, case sensitivity, context lines, all search budgets, truncation, no-match results, blocked policy cases, binary/secret candidate handling, redacted snippets, read-only safety flags, CLI output, PIG/OCPX coverage, and v0.20.0-v0.20.2 regression.

## Acceptance Criteria

- Workspace literal search works only for allowed text-like files inside workspace.
- Search is bounded by file, match, per-file byte, total byte, and context budgets.
- Binary, secret-like, private, outside, and traversal paths are skipped or blocked.
- Snippets are redacted.
- No shell text-search utility or process-spawned search is used.
- No semantic retrieval, summary, symbol parsing, or AST parsing is implemented.
- No write/shell/network/MCP/plugin/external harness execution occurs.
- No memory/persona/overlay mutation occurs.
- OCEL/PIG/OCPX visibility exists.
- v0.20.0, v0.20.1, and v0.20.2 tests still pass.

## Withdrawal Conditions

Withdraw this version if search escapes workspace policy, returns unredacted secret-like snippets, uses external execution, introduces mutation, reads without bounds, or makes JSONL the canonical process substrate.
