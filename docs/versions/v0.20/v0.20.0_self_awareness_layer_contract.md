# ChantaCore v0.20.0 - Self-Awareness Layer Contract

## Version Definition

- Version: `0.20.0`
- Target name: `Self-Awareness Layer Contract`
- Korean name: `자각 계층 계약`
- Primary public package: `src/chanta_core/self_awareness`
- Canonical process substrate: OCEL
- State: `contract_only`

## Objective

v0.20.0 introduces the public Self-Awareness Layer contract foundation. The layer describes read-only self-contemplation of workspace, codebase, runtime boundary, capability registry, evidence surfaces, and non-mutating intention candidates.

This is a contract release, not an execution release.

## Why Basic/Foundation Became Self-Awareness

Earlier planning language used Basic/Foundation as a possible name for a small internal tool pack. That name was too generic. The v0.20.0 scope is not generic foundation tooling; it is the first contract layer for a runtime to inspect and describe its own public boundaries. The layer is therefore renamed and reframed as Self-Awareness.

## Self-Awareness Is Not Self-Modification

Self-awareness is not self-modification.

Self-awareness is read-only by default. It can produce evidence, summaries, verification results, reports, and candidates. It must not mutate memory, persona, overlay, code, workspace, external systems, or capability state.

## Non-Goals

- No actual workspace file read implementation
- No actual search or symbol scan implementation
- No write/edit/apply-patch capability
- No shell, test, or lint execution
- No network or web fetch
- No MCP connection
- No plugin loading
- No external harness execution
- No sidecar or event bus
- No memory, persona, or overlay auto-promotion
- No canonical JSONL store

## Data Models

All seed contracts must use `layer="self_awareness"`. In v0.20.0, `execution_enabled=false`, `canonical_mutation_enabled=false`, and `read_only=true` are required defaults.

Risk defaults must keep the following false:

- `mutates_workspace`
- `mutates_memory`
- `mutates_persona`
- `mutates_overlay`
- `uses_shell`
- `uses_network`
- `uses_mcp`
- `loads_plugin`
- `executes_external_harness`
- `dangerous_capability`

Gate defaults require evidence references and an execution envelope, but the envelope is a contract boundary only; it does not permit execution.

## Service Design

`SelfAwarenessRegistryService` lists seed contracts, resolves a contract by skill ID, lists seed IDs, and summarizes layer safety counts.

`SelfAwarenessConformanceService` checks every seed contract for read-only defaults, non-execution, no canonical mutation, and no dangerous capability.

`SelfAwarenessReportService` builds a PIG report and OCPX projection from the registry and conformance report.

## OCEL Mapping

Object types include self-awareness layer, skill contract, capability, workspace, workspace path, workspace file, code symbol, project structure, runtime boundary, capability registry, verification result, summary candidate, plan candidate, todo candidate, and execution envelope.

Event types include layer registration, contract registration, contract check, conformance report creation, registry view, PIG report creation, and OCPX projection creation.

Relation types include layer membership, declared capability, gate/risk/observability linkage, OCEL object/event/relation mapping, conformance check linkage, workbench visibility, PIG visibility, and OCPX visibility.

OCEL remains the canonical process substrate. JSONL is not introduced as a canonical store.

## PIG/OCPX Report

The PIG report shows the self-awareness layer definition, contract list, safety boundary counts, diagnostics, and a recommendation to proceed to `v0.20.1 Self-Workspace Awareness` only if conformance passed.

The OCPX projection shows object, event, and relation coverage with `state="contract_only"`.

## Tests

Required tests verify:

- all seed skill IDs are registered;
- every seed contract uses `layer="self_awareness"`;
- every seed contract has execution and canonical mutation disabled;
- risk profiles are read-only and do not use write, shell, network, MCP, plugin, or external harness execution;
- dangerous capability count is zero;
- default conformance passes;
- CLI registry and conformance output show the self-awareness layer and safe counts;
- PIG and OCPX reports build;
- forbidden private or dangerous implementation strings do not appear in new public self-awareness code except as documented boundary language.

## Acceptance Criteria

- `python -m pytest tests/test_self_awareness_contract.py`
- `python -m pytest tests/test_self_awareness_boundaries.py`
- existing observation/digestion tests still pass
- CLI commands return safe read-only output
- dangerous capability count is `0`
- write, shell, network, MCP, plugin, external harness, memory, persona, and overlay mutation counts are `0`

## Withdrawal Conditions

Withdraw v0.20.0 acceptance if any seed contract enables execution, enables canonical mutation, declares a dangerous capability, implements write/shell/network/MCP/plugin/external harness behavior, changes external adapter execution status, or introduces private material into public code, docs, tests, CLI output, or reports.

Withdraw the canonical-store claim if any implementation treats JSONL as the canonical process substrate instead of OCEL.

## Next Version

`v0.20.1 Self-Workspace Awareness` may add reviewed read-only workspace awareness only if v0.20.0 conformance passes and the boundary remains non-mutating.
