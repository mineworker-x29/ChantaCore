# v0.20.7 Self-Directed Intention Candidate

Korean name: 자기 지향 의도 후보

Track: OCEL-native Self-Awareness

## Version Definition

v0.20.7 adds candidate-only self-directed intention generation to the public
ChantaCore self-awareness layer. It can create plan, todo, no-action, and
needs-more-input candidates from supplied self-awareness sources.

Self-awareness is not self-modification.
Self-directed intention is not execution.
Plan candidate is not plan execution.
Todo candidate is not materialized todo.
No-action is a first-class candidate.

## Relationship To v0.20.0 Through v0.20.6

v0.20.0 introduced the `self_awareness` layer contract. v0.20.1 added
workspace path verification and metadata-only workspace inventory. v0.20.2
added bounded text/code perception. v0.20.3 added bounded literal workspace
search. v0.20.4 added deterministic markdown/python structure summarization.
v0.20.5 added deterministic project surface structure awareness. v0.20.6 added
self-surface verification. v0.20.7 uses those outputs only as reviewed source
references for candidate generation.

## Objective

Create a deterministic candidate bundle:

- `SelfPlanCandidate`
- `SelfTodoCandidate`
- `SelfNoActionCandidate`
- `SelfNeedsMoreInputCandidate`
- `SelfDirectedIntentionCandidateBundle`

The bundle is a review surface. It does not execute steps, create tasks, write
files, update memory, mutate persona/overlay state, or promote canonical state.

## Included Scope

- Public, generic self-awareness intention candidate concepts.
- Deterministic rule-based source collection and candidate builders.
- Explicit read-only gate invocation for:
  - `skill:self_awareness_plan_candidate`
  - `skill:self_awareness_todo_candidate`
- CLI rendering for `self-awareness intention candidates`.
- OCEL/PIG/OCPX visibility.

## Non-Goals

v0.20.7 does not implement plan execution, todo file creation, task queue
creation, scheduler registration, autonomous mission loops, file edits, shell
or test execution, network fetch, MCP connection, plugin loading, external
harness execution, sidecar/event bus behavior, memory/persona/overlay mutation,
candidate auto-promotion, LLM planner use, or materialization of todo/plan
candidates.

## Service Design

- `SelfDirectedIntentionPolicyService.decide(request)` creates read-only,
  no-write, no-shell, no-network, no-MCP, no-plugin, and candidate-only
  constraints.
- `IntentionSourceCollector.collect(request)` converts goal text, source
  candidate ids, and source report ids into bounded source refs.
- `PlanCandidateBuilder.build(...)` creates non-executable plan step candidates.
- `TodoCandidateBuilder.build(...)` creates non-materialized todo candidates.
- `NoActionCandidateBuilder.build(...)` always preserves the no-action option
  when requested.
- `NeedsMoreInputCandidateBuilder.build(...)` creates missing-input candidates
  when source evidence is insufficient.
- `SelfDirectedIntentionService.create_candidates(request)` composes the bundle.
- `SelfDirectedIntentionSkillService` exposes plan/todo candidate bundle
  creation to explicit read-only skill invocation.

## Candidate Generation Policy

Candidate generation is deterministic and rule-based. It does not call an LLM
planner or model judge. Candidate steps are suggestions for review only.
Potential write, shell, network, MCP, or plugin work can appear only as blocked
future candidate steps and never as executable steps.

## No-Action Policy

No-action is not failure. It is a first-class review candidate with
`recommended_review_decision=no_action`. It may appear even when plan or todo
candidates also exist.

## Needs-More-Input Policy

Needs-more-input candidates include `missing_inputs` and
`recommended_review_decision=needs_more_input`. They do not send prompts, ask
the user automatically, or trigger any follow-up action.

## Risk Policy

Risk assessments default to `required_review=true`,
`safe_to_execute=false`, and `safe_to_materialize=false`. Blocked capabilities
include write, shell, network, MCP, plugin, external harness, task queue,
scheduler, memory mutation, persona mutation, overlay mutation, and canonical
promotion.

## OCEL Mapping

Object types include directed intention request, candidate bundle, plan/todo
candidates, plan step/todo item candidates, no-action and needs-more-input
candidates, source refs, constraints, risk assessments, surface verification
reports, project structure candidates, summary candidates, search results, and
execution envelopes.

Event types include requested, policy checked, sources collected, plan/todo
candidate created, no-action candidate created, needs-more-input candidate
created, bundle created, and blocked.

Relation types include derived-from, uses verification/project/summary/search
sources, constrained-by, assesses-risk, produces candidate outputs, belongs to
bundle, and recorded-in-envelope.

OCEL remains the canonical process substrate. JSONL is not introduced as a
canonical store.

## CLI Commands

Supported CLI form:

```powershell
python -m chanta_core.cli self-awareness intention candidates
python -m chanta_core.cli self-awareness intention candidates --goal "inspect project structure and propose next safe steps"
python -m chanta_core.cli self-awareness intention candidates --source-candidate self_project_structure_candidate:example
python -m chanta_core.cli self-awareness intention candidates --source-report self_surface_verification_report:example
python -m chanta_core.cli self-awareness intention candidates --include-no-action
python -m chanta_core.cli self-awareness intention candidates --strictness strict
```

CLI output shows bundle id, candidate counts, `candidate_only`, false execution
and materialization flags, false promotion flags, and that no actual execution
occurred. It must not print private full paths or raw secrets.

## PIG/OCPX Report Update

PIG report coverage includes:

- `directed_intention_candidate=implemented_candidate_only`
- `execution_enabled=false`
- `materialization_enabled=false`

- `canonical_promotion_enabled=false`

`needs_more_input_candidate`, and `directed_intention_candidate_bundle`.

## Tests

Tests cover plan/todo/no-action/needs-more-input generation, candidate-only
semantics, non-execution, non-materialization, blocked future capabilities, no
task queue or TODO file creation, no memory/persona/overlay mutation, no LLM
planner use, CLI output, OCEL mapping, PIG/OCPX coverage, and regression for
earlier self-awareness layers.

## Acceptance Criteria

- Plan/todo/no-action/needs-more-input candidates are generated.
- No candidate is executed.
- No candidate is materialized.
- No task queue entry or TODO file is created.
- No LLM planner is used.
- No write/shell/network/MCP/plugin/external harness execution occurs.
- No memory/persona/overlay mutation occurs.
- All candidates are candidate-only and require review.
- OCEL/PIG/OCPX visibility exists.
- v0.20.0 through v0.20.6 tests still pass.

## Withdrawal Conditions

Withdraw v0.20.7 acceptance if any implementation executes candidates,
materializes todo/plan state, creates task queue entries, writes `TODO.md`,
uses an LLM planner, performs write/shell/network/MCP/plugin/external harness
execution, mutates memory/persona/overlay state, enables canonical promotion, or
exposes private user-specific material in public code, docs, tests, CLI output,
or reports.
