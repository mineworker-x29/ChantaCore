# v0.34.1 Provider Profile & Invocation Policy

Parent track: Controlled Model Invocation Boundary.

Related foundations:
- v0.34.0 Controlled Model Invocation Boundary Foundation.
- v0.33.9 Internal General Agent Runtime MVP v1.

v0.34.1 defines provider profile metadata and invocation policy artifacts only. A provider profile is a descriptor for a possible future provider role. It is not provider invocation, provider runtime, credential access, network access, endpoint construction, or execution readiness.

Explicit distinctions:
- Provider profile is not provider invocation.
- Provider registry is not provider runtime.
- Provider invocation policy is not invocation permission.
- Credential posture is not credential access.
- Network posture is not network access.
- Existing chat_service boundary ref is not a provider call.

## Boundary Meaning

Provider profile means metadata about a profile kind, boundary kind, invocation mode, credential posture, network posture, limits, prompt data policy, response data policy, source refs, validation findings, and resolution decisions.

Provider registry means a metadata registry of descriptors. It is not a runtime registry and does not call providers.

Provider invocation policy means a constraint object for future controlled invocation design. It is not invocation permission in v0.34.1.

Credential posture means policy metadata. It is not credential access. v0.34.1 does not read credentials, does not read `.env`, does not read secrets, does not read OS environment variables for credentials, and does not create a credential loader.

Network posture means policy metadata. It is not network access. v0.34.1 does not use requests/httpx/urllib/aiohttp/socket, does not create a network adapter, and does not store endpoint URL values.

Existing chat_service/provider boundary refs are string/id refs only. Existing provider boundary invocation remains false. v0.34.1 does not create a provider adapter and does not invoke an existing boundary.

Mock and supplied-output provider profiles are non-network metadata profiles. Local and remote provider profiles are future-gated metadata only.

## Policy Surface

Provider profile kinds include mock_provider, supplied_output_provider, existing_chat_service_provider, existing_provider_boundary, local provider future gates, remote provider future gates, disabled_provider, and unknown.

Credential policy blocks env read, secret file read, inline secrets, credential value storage, and credential logging.

Network policy blocks direct network, endpoint value storage, arbitrary endpoints, localhost direct calls, and remote direct calls. `endpoint_url_value` remains `None`.

Invocation limit policy records future constraints only. Retries, streaming, tool calls, function calls, and recursive calls remain false in v0.34.1.

Prompt data policy blocks raw prompt persistence, secrets in prompts, credentials in prompts, file content in prompts, and unbounded prompts. Redaction is required.

Response data policy blocks raw response persistence, untrusted response as action, response tool calls, response patches, response commands, and unbounded responses. Redaction and quarantine are required.

## Validation And Resolution Flow

Flow:

`V0340 Boundary / ExistingProviderBoundaryRef / manual profile specs -> ProviderProfileDescriptor -> ProviderProfileRegistry -> ProviderInvocationPolicy -> ValidationReport -> ResolutionDecision -> V0341ReadinessReport -> v0.34.2 Model Request Envelope`

Validation checks metadata only. A validation report is not runtime certification. Resolution selects metadata only and keeps invocation, provider invocation, network access, credential access, and secret read false.

## Reference Corpus Policy

references/OpenCode, references/Hermes, and references/OpenClaw remain safe reference-context assets only. v0.34.1 does not execute them, import them, install their dependencies, run their tests or package scripts, or use them as provider adapter code.

## Handoff

`ready_for_v0342_model_request_envelope`, `ready_for_v0343_model_response_envelope`, and `ready_for_v0344_existing_provider_boundary_adapter` may be true only as design-stage handoff signals. They do not mean model invocation, provider invocation, existing boundary invocation, network access, credential access, or execution readiness.

## Hard Prohibitions

v0.34.1 does not call providers. It does not invoke provider SDKs. It does not import provider SDKs. It does not use network. It does not read credentials. It does not read `.env`. It does not read secrets. It does not create a provider adapter. It does not create a credential loader. It does not create a network adapter. It does not execute agent steps. It does not execute tools. It does not execute shell/subprocess/commands. It does not write/edit/patch. It does not persist traces. It does not implement UI runtime. It does not grant authority.

## Acceptance Criteria

v0.34.1 is acceptable only if provider profile descriptors, registry metadata, credential policy, network policy, invocation limits, prompt/response data policies, validation, resolution, run preview, no-invocation guarantee, and readiness report exist while every unsafe runtime readiness flag remains false.

## Withdrawal Conditions

Withdraw v0.34.1 readiness if any provider call, provider SDK import, existing boundary invocation, network call, credential/env/secret read, endpoint value storage, provider adapter, credential loader, network adapter, agent/tool/shell execution, write/edit/patch, reference execution/import/install, persistent trace write, UI runtime, external control, authority grant, or unsafe readiness flag is introduced.
