# v0.34.4 Existing Provider Boundary Adapter

Parent track: Controlled Model Invocation Boundary.

Related foundations:
- v0.34.3 Model Response Envelope & Sanitizer.
- v0.34.2 Model Request Envelope.
- v0.34.1 Provider Profile & Invocation Policy.
- v0.34.0 Controlled Model Invocation Boundary.

v0.34.4 defines a controlled adapter around an existing ChantaCore provider/chat_service boundary. In this implementation, the callable path is dependency-injected only. No automatic project-local provider boundary discovery or wrapper was added.

## Explicit Distinctions

- ExistingProviderBoundaryAdapter is not a new provider SDK adapter.
- Existing boundary invocation is not direct provider invocation.
- Injected callable is not SDK integration and not network integration.
- ExistingProviderBoundaryCallableRef is metadata and is not credential access.
- Provider output is untrusted model output, not action.
- Response envelope bridge is not action execution.
- Controlled existing-boundary invocation may occur only after adapter policy and invocation decision allow it.

## Adapter Scope

The adapter can:
- validate ModelRequestEnvelope metadata
- validate adapter policy, callable refs, and descriptor metadata
- decide whether a controlled existing-boundary call is allowed
- call an explicitly supplied boundary callable after policy approval
- safe-fail if the callable raises
- return no-op/blocked results when no callable is supplied or policy blocks
- bridge returned text into v0.34.3 Model Response Envelope/Sanitizer helpers

The adapter does not discover provider endpoints, load credentials, import provider SDKs, or construct network clients.

## Taxonomies

Adapter kinds include injected_callable_adapter, existing_chat_service_boundary_adapter, existing_provider_boundary_adapter, mock_boundary_adapter, supplied_output_adapter, disabled_adapter, future_gate_adapter, blocked_adapter, and unknown.

Invocation modes include no_invocation, dry_run, mock_only, supplied_output_only, injected_existing_boundary, project_local_existing_boundary, blocked, future_gate, and unknown. Only injected_existing_boundary and project_local_existing_boundary are controlled existing-boundary modes, and this patch implements callable injection only.

Decision kinds distinguish allow_mock_response, allow_supplied_response, allow_injected_existing_boundary_call, allow_project_local_existing_boundary_call, deny, block, no_op, require_review, future_gate_required, and unknown.

Risk kinds include direct provider SDK, arbitrary network, credential exposure, secret read, endpoint construction, existing boundary bypass, raw prompt/response persistence, unbounded prompt/response, provider output as action, recursive model call, autonomous loop, tool/function call, command execution, workspace write, reference execution, and external harness execution.

## Policy Gate Requirements

The default policy may allow mock, supplied output, and injected existing-boundary calls. Project-local boundary calls are represented but remain off by default.

The policy always blocks:
- direct provider SDK
- direct network
- credential read
- secret read
- endpoint construction
- recursive call
- tool calls
- function calls
- action execution
- raw prompt persistence
- raw response persistence

Policy is not invocation by itself. Invocation requires an invocation input, a valid adapter descriptor, an allow decision, and an explicitly supplied callable.

## Callable Injection Policy

The only implemented call path is a callable argument supplied to `invoke_existing_provider_boundary_adapter`. The adapter calls that function only after `decide_existing_provider_boundary_invocation` returns an allow decision for an existing-boundary mode.

If the callable is absent, the adapter returns no_op safely. If policy blocks, the adapter returns a blocked result and does not call the callable. If the callable raises, the adapter returns safe_fail without retrying or performing unsafe fallback work.

## Project-Local Boundary Policy

Project-local existing provider boundary references are represented as future wiring. A future implementation may connect a ChantaCore-local chat_service/provider boundary only if it is already part of the project and still passes this adapter policy gate. v0.34.4 does not add such a wrapper.

## Response Bridge

Returned callable text is treated as untrusted provider output. It is bridged through v0.34.3 response envelope/sanitizer helpers. The output is not executed, is not a tool call, is not a patch, is not a command, and cannot override adapter policy.

Raw prompt and raw response persistence remain disabled.

## Reference Corpus Policy

references/OpenCode, references/Hermes, and references/OpenClaw remain safe reference-context assets only. v0.34.4 does not execute them, import them, install their dependencies, run their tests or package scripts, or wrap them as providers.

Provider boundary adapter code must not call or wrap OpenCode, Hermes, or OpenClaw as a provider.

## Flow

`ModelRequestEnvelope / ProviderProfileDescriptor / ExistingProviderBoundaryRef / injected callable -> AdapterPolicy -> InvocationDecision -> ExistingBoundaryCall -> ModelResponseEnvelope Sanitizer -> InvocationResult -> V0344ReadinessReport -> v0.34.5 Model Output Action Quarantine`

v0.34.6 Agent Step Runner Model Integration remains a future design-stage handoff.

## Hard Prohibitions

v0.34.4 may call only an injected or existing project-local provider boundary after policy approval. This patch implements only injected callable wiring.

v0.34.4 does not import provider SDKs. It does not implement a new provider SDK adapter. It does not create a credential loader. It does not create a network adapter. It does not directly use network libraries. It does not read credentials. It does not read `.env`. It does not read secrets. It does not read env vars for credentials. It does not construct endpoints.

v0.34.4 does not execute provider output. It does not execute agent steps. It does not execute tools. It does not execute shell/subprocess/commands. It does not write/edit/patch. It does not persist raw prompt/response. It does not persist traces. It does not implement UI runtime. It does not grant authority.

Action execution remains disabled until future quarantine/integration stages.

## Acceptance Criteria

v0.34.4 is acceptable only if the adapter can use an injected callable after policy approval, missing callable returns no-op or blocked safely, callable exceptions safe-fail, returned output is sanitized through v0.34.3, and every direct SDK/network/credential/action/write/persistence/execution flag remains false.

## Withdrawal Conditions

Withdraw v0.34.4 readiness if any direct provider SDK import, direct SDK invocation, direct network access, credential/env/secret read, endpoint construction, new provider SDK adapter, credential loader, network adapter, general agent execution, autonomous loop, tool/function/action execution, shell/subprocess/command execution, workspace write/edit/patch, reference execution/import/install, dependency install, raw prompt/response persistence, persistent trace write, UI runtime, external control, authority grant, or unsafe readiness flag is introduced.

## Handoff

v0.34.4 hands off only to v0.34.5 Model Output Action Quarantine and v0.34.6 Agent Step Runner Model Integration as design-stage work. It does not enable action execution or general agent execution.
