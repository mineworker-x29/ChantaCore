# v0.43.11.7 Chat Transcript Visual Hierarchy Restore

## Restore Purpose

v0.43.11.7 is a visual hotfix for the Schumpeter full-screen TUI transcript. It improves message readability without opening v0.44 runtime capabilities.

## User-Observed Message Differentiation Problem

The provider and Schumpeter identity path were observed to work, but the transcript still looked like a plain log. User messages, assistant messages, status output, and artifacts had similar visual weight.

## v0.43.11.7 Goal

Make transcript entries immediately distinguishable as user, assistant, status, artifact, diagnostic, or unavailable/error cards while preserving scrollable chat and visible input.

## Message Card Visual Contract

Every transcript item renders through a message card contract with:

- a stable kind;
- a speaker or surface label;
- card spacing between turns;
- bounded line wrapping;
- no raw metadata by default;
- zero runtime side effects.

## User Message Card

User messages use the `You` label, `.message-card.user`, and a distinct user accent. They do not show debug metadata.

## Assistant Message Card

Assistant messages use the `Schumpeter` label, `.message-card.assistant`, and the product accent. Long Korean text wraps inside the card width.

## Status Card

Status output uses `.message-card.status` and the `Status` label. Provider/model/safety fields may appear as human-readable status content, not raw metadata footers.

## Artifact Card

Artifact output uses `.message-card.artifact` and the `Summary` label for summary-style output. Raw fields such as `type: summary`, `grounding:`, and `source:` are hidden by default.

## Diagnostic Card

Diagnostic output uses `.message-card.diagnostic` and the `Diagnostic` label. Debug metadata remains limited to explicit debug surfaces.

## Error Card

Unavailable capability output uses `.message-card.error` and the `Unavailable` label. It should be clear but not alarming.

## Textual CSS / Theme Changes

The Textual theme now declares:

- `.message-card`
- `.message-card.user`
- `.message-card.assistant`
- `.message-card.status`
- `.message-card.artifact`
- `.message-card.diagnostic`
- `.message-card.error`
- `.speaker-label`
- `.message-body`

The pure snapshot renderer uses the same card contract for deterministic tests.

## Manual Acceptance

Run:

```powershell

chanta-cli start
```

Inside the TUI:

```text

/provider

/what-happened
저장소 상태 확인해줘

```

Pass if user, assistant, status, artifact, diagnostic, and unavailable cards are visually distinct, input remains visible, and no raw metadata appears.

## Safety Boundary

This hotfix is rendering-only. Rendering must not call providers, submit prompts, run shell or git, read workspace or repository files, call tools/functions, invoke subagents, mutate memory, write CORE_MEMORY, or claim production certification.

## Withdrawal Conditions

Withdraw this hotfix if:

- user and assistant messages still look like one plain log;
- status, artifact, diagnostic, and error cards are indistinguishable;
- Korean text overflows horizontally;
- input becomes hidden;
- raw metadata appears by default;
- rendering introduces runtime side effects.

## v0.44 Gate Recommendation

Do not proceed to v0.44 from this visual hotfix alone. Recheck the TUI transcript manually first. If the card hierarchy passes and provider context remains stable, the next step is either v0.43.11.8 visual polish or a separate v0.44 gate review.
