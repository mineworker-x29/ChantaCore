# v0.43.11.5 TUI Provider Context Alignment Restore

## Restore Purpose

v0.43.11.5 aligns the Schumpeter full-screen TUI provider context with the
existing `chanta-cli run` context. The hotfix prevents the TUI from reporting
`provider_not_configured` when the configured provider works through the CLI.

## User-Observed Provider Context Mismatch

The user verified that configured provider execution works outside the TUI:

- `chanta-cli provider connectivity` succeeds.
- `chanta-cli run --profile default-personal --provider configured --timeout 120 "1+1은?"` returns a valid answer.
- run-report shows a parsed provider response.

Inside the TUI, the same kind of general chat returned provider unavailable /
not configured while the sidebar still displayed configured provider status.

## External Provider Health Evidence

The external evidence supports this diagnosis:

- configured model found
- completion path works through `chanta-cli run`
- `provider_invoked` and `prompt_submitted` are true in run-report
- shell, subagent, memory mutation, and production certification remain false

## Root Cause Hypothesis

The TUI was using a different provider context than the CLI run path. In
particular, earlier TUI setup could fall back to a worktree-local
`.chantacore-personal` home, while CLI run/home resolution can use
`CHANTACORE_HOME` or platform default home such as `%LOCALAPPDATA%\ChantaCore`.

This hypothesis is valid while the TUI context is created from work-session
state instead of the shared v0.42 home/provider resolver. Withdraw it if a later
trace shows identical home/profile/provider values but provider invocation still
fails differently.

## v0.43.11.5 Goal

The TUI must use the same home/profile/provider resolver and existing run
service as `chanta-cli run`.

## Provider Context Contract

`V043115TUIProviderContext` records:

- profile id
- resolved home path
- provider mode
- configured-provider availability
- provider label
- configured model
- provider status source
- provider resolution error
- timeout
- same-resolver flag
- provider/prompt flags
- production certification flag

The context uses the v0.42 home resolver and provider status report surface.

## RuntimeAdapter Dispatch Alignment

`V04310RuntimeAdapter` now creates provider context first, then creates its work
session state from that resolved context. General chat dispatch uses the context
home/profile/provider/timeout when constructing `RunCommandInput`.

## Provider Status Consistency

`/provider`, the session panel, and the PI monitor use the resolved provider
context instead of hardcoded provider text.

If status says configured and invocation fails, the error explains that the
provider was available in status but failed during invocation.

## Existing Run Path Binding

The TUI still calls the existing internal run service. It does not create a
new OpenAI, LM Studio, HTTP, socket, or direct completion client.

## Error Rendering

Unavailable provider output is concise:

- Provider unavailable
- `/provider`
- `chanta-cli provider connectivity`

No traceback or raw run-report JSON is shown by default.

## Withdrawal Conditions

Withdraw this hotfix if:

- TUI still says provider unavailable/not configured while CLI run works.
- sidebar/status and dispatch disagree.
- TUI adds a direct provider HTTP client.
- help/status/sidebar/palette call provider.
- high-risk capabilities open.
- production certification is claimed.
