# v0.43.10.1 Structured TUI Interaction Loop Repair Restore

## Restore Purpose

This document restores the v0.43.10.1 hotfix context for the Schumpeter structured TUI interaction loop.

## One-Screen Restore Summary

v0.43.10.1 repairs the Schumpeter structured TUI interaction loop. It prevents the full static shell from being reprinted after every user input, separates snapshot rendering from interactive turn rendering, preserves transcript state, and keeps all high-risk capabilities closed.

## User-Observed Failure

`chanta-cli start --tui` showed the full Schumpeter header, project panel, session panel, PI monitor, chat notice, input block, and status line again after normal inputs. The result behaved like repeated static snapshots instead of a usable structured TUI transcript.

## Root Cause Hypothesis

The v0.43.10 plain and prompt_toolkit fallback loops used the full snapshot renderer after each submitted input. That is valid for snapshot mode, but wrong for append-style interactive mode.

## v0.43.10.1 Goal

Repair the interaction loop without opening new capabilities. Interactive mode must print static chrome once, then append only user/result turns.

## Interactive Loop Repair

The loop now keeps one app state, dispatches submitted text through `RuntimeAdapter`, applies turn results to transcript state, and renders each turn with a dedicated turn renderer.

## Transcript State Repair

`V04310TranscriptState` and `V04310InteractiveLoopState` make transcript ownership explicit. Transcript messages preserve user, assistant, artifact, diagnostic, status, error, and system notice order.

## Plain Fallback Rendering Policy

Plain fallback prints the static structured header once. After that, each turn prints only:

```text
You> input
Status>/Schumpeter>/Artifact>/Diagnostic> result
```

## Full-Screen Redraw vs Plain Append Policy

Snapshot/full frame rendering may be used for one-shot snapshots or future full-screen replacement redraws. Plain append mode must not append the full frame after every turn.

## Snapshot vs Interactive Separation

Snapshot mode uses the frame renderer and exits. Interactive mode uses the turn dispatcher and turn renderer after startup.

## Turn Dispatch Contract

`/status` maps to status, normal text maps to assistant, `/summary` maps to artifact, `/what-happened` maps to diagnostic, and `/exit` sets the exit flag. Turn dispatch never requests full static chrome reprint.

## Renderer Boundary Preservation

Components and renderers remain pure display code. Rendering, status, sidebar, palette, and snapshot paths do not call provider completion, shell, git, repo search, workspace read, or memory mutation.

## No-Repeat-Chrome Policy

The no-repeat policy requires static chrome once in plain mode, no full frame append after each turn, snapshot renderer only for snapshot/full-screen replacement, and turn renderer for plain interactive turns.

## Golden Interaction Sequence

Golden input sequence:

```text
/status
오늘 작업 계획 정리해줘
/summary 오늘 Schumpeter TUI를 테스트하고 있어.
/what-happened
/exit
```

Expected static counts: Schumpeter header <= 1, Project <= 1, Session <= 1, PI Monitor <= 1, startup notice <= 1.

## Safety Boundary

Still closed: workspace read, arbitrary file read, repo search, workspace search, shell/git execution, file edit/apply, provider tool calling, function calling, subagents, general AgentLoop expansion, autonomous coding, memory mutation from rendering, CORE_MEMORY write, and production certification.
