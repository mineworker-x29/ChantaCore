# v0.43.11.4 TUI Provider-Backed General Chat Restore

## Restore Purpose

v0.43.11.4 repairs the Schumpeter full-screen TUI general conversation path.
Before this hotfix, provider-eligible normal text entered in the TUI could fall
through to a local fallback sentence instead of using the existing configured
provider run path.

## User-Observed Provider Dispatch Gap

The provider path worked outside the TUI:

- `chanta-cli provider connectivity` reported the configured local provider as reachable.
- `chanta-cli run --profile default-personal --provider configured --timeout 120 "1+1은?"` returned a real assistant response.
- `chanta-cli run-report last --profile default-personal` showed provider invocation and parsed response metadata.

Inside the TUI, the same kind of normal input returned local fallback guidance.
That showed the TUI general chat route was not wired to the existing run service.

## External Provider Verification Evidence

The user-confirmed evidence was:

- configured provider reachable
- configured model found
- `chanta-cli run` returned assistant text
- run-report showed `provider_invoked` and `prompt_submitted`
- shell, subagent, and production certification remained closed

This document treats that as external manual evidence, not as a live automated
provider dependency.

## v0.43.11.4 Goal

Provider-eligible normal TUI input now routes through `RuntimeAdapter` to the
existing single-turn configured-provider run service. The TUI does not construct
a second provider client.

## General Chat Routing Contract

Routing order:

1. Slash commands use the canonical command dispatcher.
2. Deterministic local intents remain local:
   - identity
   - capability
   - repository boundary/status
   - runtime/help/status/about surfaces
3. Other normal text routes to provider-backed `general_chat`.

The old fallback text must not be used for provider-eligible general chat.

## RuntimeAdapter Provider Dispatch

`V04310RuntimeAdapter.submit_user_input()` now sends general chat to the existing
`default_personal_run.execute_run_command()` service through an injectable run
service boundary. Tests use a fake run service; production defaults to the
existing configured run path.

The adapter preserves:

- `run_id`
- `session_id`
- provider flags
- response parse status
- provider model when available
- assistant response preview

These fields are for tests/debug/report surfaces and are hidden from default
chat rendering.

## Provider Mode Selection

The adapter uses the current work session provider mode. The Schumpeter TUI
default remains configured/auto-oriented through the existing runtime state.
Provider calls happen only on submitted provider-eligible user text.

## Renderer Behavior

Provider responses render as normal assistant messages. They do not render as
artifacts, diagnostics, raw run-report JSON, or debug metadata.

## Error Handling

Provider failures render a concise error card:

- short explanation
- `/provider`
- `chanta-cli provider connectivity`

Tracebacks and raw provider metadata are hidden by default.

## Trace / Run Report Integration

The adapter emits the same bounded run events used by run-report reconstruction.
Automated tests verify `create_last_run_report()` sees a provider invocation and
assistant response preview after a fake provider-backed TUI turn.

## Safety Boundary

Still closed:

- workspace read
- repo search
- shell/git execution
- file edit/apply
- provider tools/functions
- subagents
- memory mutation
- CORE_MEMORY write
- production certification

The TUI reuses the existing run service and does not create a direct provider
HTTP client.

## Withdrawal Conditions

Withdraw this hotfix if:

- TUI general chat still returns the old fallback sentence.
- The TUI creates a direct provider client.
- Provider responses render as artifact/status/debug text by default.
- Help/status/sidebar/palette invoke the provider.
- Any closed high-risk capability opens.
- production certification is claimed.
