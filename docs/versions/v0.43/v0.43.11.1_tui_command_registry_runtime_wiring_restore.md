# v0.43.11.1 TUI Command Registry Runtime Wiring Restore

## Restore Purpose

v0.43.11.1 repairs the functional wiring gap in the Schumpeter full-screen TUI. The fix synchronizes help, slash palette, completion, command dispatch, usage validation, and renderer classification around one canonical command registry.

## User-Observed Functional Wiring Gap

The v0.43.11 Textual surface opened successfully and looked like a product UI, but the command layer was incomplete. `/help` and palette did not expose the same command surface, bare workflow commands produced placeholder artifacts, and known natural-language diagnostic intents could fall through to generic conversation fallback.

## v0.43.11.1 Goal

The goal is to make the TUI command layer coherent without opening v0.44 capabilities. Commands must either have a safe handler or show a clean usage/unavailable response. Rendering, help, palette, status refresh, and snapshot paths remain side-effect-free.

## Canonical Command Registry

The canonical registry lives in:

`src/chanta_core/schumpeter_tui/command_registry.py`

Each command spec includes command text, aliases, category, Korean descriptions, usage, examples, provider policy label, side-effect label, availability, handler id, argument requirement, renderer kind, and high-risk capability flags.

Categories:

- Start / Basic
- Workflows
- Artifacts
- Notes
- Evidence
- Grounded Workflows
- Status / Diagnostics
- Pilot / Release
- TUI
- Safety

## Help / Palette Synchronization

`/help commands`, slash palette rendering, and completion use the same registry. The palette filters from the registry and inserts text only; it does not invoke handlers before final submission.

## Command Dispatch Table

TUI dispatch first resolves the canonical registry. It then applies argument validation and renderer classification before falling through to the existing safe work-session path. Registered commands no longer silently become generic fallback output.

Key routes:

- `/help` -> help modal/status surface
- `/status` -> concise status renderer
- `/status --debug` -> debug status renderer
- `/summary <text>` -> artifact renderer
- `/summary` -> usage error renderer
- `/todo <text>` -> artifact renderer
- `/todo` -> usage error renderer
- `/recall <query>` -> evidence/diagnostic guidance
- `/recall` -> usage error renderer
- `/what-happened` -> diagnostic renderer
- `/v044 readiness` -> status/diagnostic renderer
- `/exit` -> clean exit

## Argument Validation

Commands requiring arguments are blocked before they can produce placeholder artifacts. Bare commands show:

```text
Usage
/summary <내용>

예:
  /summary 오늘 테스트 결과를 요약해줘
```

This applies to workflow, revision, note search, evidence, use-evidence, and grounded workflow commands.

## Natural Language Routing Improvements

Known natural-language intents are intercepted before generic fallback:

- capability question: `너가 할 줄 아는게 뭐야`
- trace/log request: `ocel 로그 확인해줘`
- repository status request: `저장소 상태 확인해줘`

Trace/log and repository requests do not claim direct inspection. They guide the user to `/what-happened`, `/trace`, `/run-report`, `/status`, or the v0.44 Controlled Workspace Read gate.

## Renderer Mapping

Registry `renderer_kind` maps command results to:

- message
- artifact
- diagnostic
- status
- error
- modal

Default output hides raw metadata, provider internals, raw safety booleans, and debug flags.

## Withdrawal Conditions

Withdraw or roll back if help and palette diverge, bare workflow commands produce placeholder artifacts, known diagnostic intents fall to generic fallback, rendering opens provider/shell/repo/workspace/memory actions, or production certification is claimed.

## v0.43.11.2 or v0.44 Recommendation

If this wiring passes manual acceptance, the next reasonable step is either a small v0.43.11.2 visual/keyboard polish pass or a v0.44 gate review for controlled workspace read. v0.44 should not start until the TUI command surface is accepted.
