# v0.43.11.10 Forced Bottom Layout Rewrite Restore

## Restore Purpose
Restore the Schumpeter full-screen Textual TUI bottom layout so the input widget has guaranteed clearance above the status line.

## User-Observed Continued Input/Footer Failure
The previous clearance patch did not materially change the visible result. Korean input such as `넌 누구야` still appeared squeezed against the bottom status row.

## Why Previous Patch Failed
The failure condition is consistent with a style-only adjustment that did not fully rewrite the bottom widget tree. If input and status are not normal-flow siblings with fixed row allocation, Textual footer or docked status behavior can still visually compete with the input area.

## v0.43.11.10 Goal
Use a normal-flow vertical stack with explicit regions: main body, palette, input, and status. The required geometry is `input_bottom <= status_top - 1`.

## Bottom Layout Tree
The intended tree is:

```text
AppRoot
MainBody
PaletteRegion
InputRegion
StatusRegion
```

`MainBody` owns the sidebar and chat panel. `PaletteRegion`, `InputRegion`, and `StatusRegion` are siblings under `#app-root`.

## Footer Removal
Textual's built-in `Footer` is not mounted. The visible one-line status surface is a custom `#status-region` plus `#status-bar` inside the normal vertical layout.

## InputRegion / StatusRegion Geometry Contract
`#input-region` is fixed at height 5. `#main-input` is fixed at height 3 and must remain fully inside `#input-region`. `#status-region` is fixed at height 1. Tests assert that the input widget bottom is at least one row above the status region top.

## Palette Anchoring Contract
`#palette-region` is above `#input-region`, hidden by default, and switched to `.open` when active. It must not cover the input widget.

## Widget Region Tests
The v0.43.11.10 tests inspect actual Textual widget regions and enforce:

```python
input_widget.region.y + input_widget.region.height <= status_region.region.y - 1
```

## Snapshot Acceptance
Snapshots cover wide and compact layouts with Korean input and palette filtering. They assert the bottom order `Commands`, `InputRegion`, `StatusRegion` and a clearance line before status.

## Manual Acceptance
Run `chanta-cli start`, type `넌 누구야`, then press `/` and type `/s`. The input text and cursor should remain visible, the palette should appear above input, and the status line should stay below the input region.

## Safety Boundary
This is a layout-only change. It does not call a provider, submit prompts, run shell/git, read a repository or workspace, mutate memory, write CORE_MEMORY, or claim production certification.

## Withdrawal Conditions
Withdraw this patch if widget region tests fail, if `Footer()` is mounted again, if status uses bottom docking, if negative margins or absolute overlap are introduced, or if the input bottom no longer satisfies the clearance invariant.

## v0.44 Gate Recommendation
Do not move to v0.44 until v0.43.11.10 passes geometry tests and the user confirms that the input/footer overlap is gone in the live TUI.

## Copy-Paste Restore Prompt
Implement v0.43.11.10 as a hard layout repair: remove Textual Footer, keep `#app-root` vertical, make `#main-body` height `1fr`, place `#palette-region` above `#input-region`, place `#status-region` below it, and enforce `input_bottom <= status_top - 1` with Textual widget region tests.
