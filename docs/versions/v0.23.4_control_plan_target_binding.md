# v0.23.4 Control Plan & Target Binding

Version: v0.23.4
Version name: Control Plan & Target Binding
Korean name: 제어 계획·대상 바인딩
Track: Internal Dominion Foundation
Layer: internal_dominion

## Objective

v0.23.4 implements the plan-only Control Plan & Target Binding stage for Internal Dominion.

It reads the v0.23.3 `ExternalActionCandidate`, v0.23.2 capability candidates, and v0.23.1 runtime inventory as read-only source records, then creates `DominionControlPlan` and binding/policy descriptors.

This version does not perform static safety, preflight, gate creation, authorization, dispatch, run tracking, output fetch, or outcome recording.

## Relationship

v0.23.0 introduced the OCEL-native Internal Dominion Contract.
v0.23.1 introduced Runtime / Agent / System Inventory.
v0.23.2 introduced Capability Observation & Digestion for Dominion.
v0.23.3 introduced Control Request & Action Candidate.

v0.23.4 binds a candidate action to provider, runtime, capability, control surface, environment, input, output, status, idempotency, rate-limit, and cancel/stop policy descriptors. The next version is v0.23.5 Dominion Static Safety Check.

## Policy Reminder

v0.23.x is Internal Dominion Foundation. It is not Self-Execution Safety. Self-execution remains reclassified to v0.24.x Local Runtime Provider.

Vendor systems are future external provider skills or provider adapters. GrowthKernel remains a future consumer / future optimizer, not an active runtime dependency.

## Core Principles

- Control plan is not dispatch.
- Target binding is not runtime touch.
- Input binding is not credential materialization.
- Control plan is not preflight.
- Control plan is not authorization.
- Control plan is not execution.
- Every plan remains provider-neutral until a future provider adapter binding.

## Service Design

- `DominionControlPlanSourceService` loads action candidate, runtime inventory report, and capability report as read-only source state.
- `ProviderBindingService` binds provider metadata without provider-specific core logic.
- `RuntimeBindingService` binds runtime metadata without touching the runtime.
- `CapabilityBindingService` binds capability candidate and risk/boundary metadata.
- `ControlSurfaceBindingService` binds declared control surface metadata.
- `EnvironmentBindingService` binds environment and future gate requirements.
- `DominionInputBindingService` creates draft input binding and blocks credential values.
- `DominionOutputPolicyService` creates redacted output policy.
- `DominionStatusTrackingPolicyService` creates required future status policy metadata.
- `DominionIdempotencyPolicyService` creates idempotency metadata.
- `DominionRateLimitPolicyService` creates rate and concurrency metadata.
- `DominionCancelOrStopPlanService` creates non-executing cancel/stop descriptors.
- `DominionControlPlanConstraintService` creates required future-stage constraints.
- `DominionControlPlanFindingService` creates findings and migration continuity findings.
- `DominionControlPlanService` creates the plan report and PIG/OCPX views.

## Plan Status Rules

`planned` requires an action candidate, provider binding, runtime binding, capability binding, environment binding, non-blocked input binding, output policy, status policy, safe idempotency/rate/cancel policy, and no forbidden effect.

`needs_more_input` is used for missing action candidate, missing required input, or unknown environment.

`no_action` is reserved for a provider-adapter-required path when the request marks no safe provider path.

`failed` is used for early dispatch flags, provider-specific core logic, or incorrect later-stage completion flags.

`blocked` is used for credential values, raw secret output, external runtime touch, provider API call, dispatch attempt, or vendor hardcoding in the core path.
