# v0.22.1 Self-Modification Request & Patch Candidate

## Version Definition

- Version: `v0.22.1`
- Name: Self-Modification Request & Patch Candidate
- Korean name: `자기수정 요청·패치 후보`
- Track: Self-Modification Safety
- Layer: `self_modification_safety`
- Subject: `modification_request_patch_candidate`

## Objective

v0.22.1 introduces the first concrete OCEL-native self-modification safety step:
candidate-only creation of modification requests and patch candidates.

This version may create request/candidate/read-model state. It must not create
patch drafts, diff hunks, diff previews, file mutations, patch applications,
dry-run execution, test/lint/shell execution, rollback execution, LLM patch
generation, or LLM judgment.

## Relationship To Prior Foundations

- v0.22.0 defines the OCEL-native Self-Modification Safety Contract, patch
  policy, lifecycle policy, and gate contract.
- v0.21.9 consolidates OCEL-native Deep Self-Introspection Foundation v1.
- v0.20.9 consolidates OCEL-native Self-Awareness Foundation v1.

If any prerequisite is absent, v0.22.1 must report gaps or blockers rather than
faking readiness.

## Core Principles

- Self-modification request is not file mutation.
- Patch candidate is not patch draft.
- Patch candidate is not diff preview.
- Patch candidate is not patch apply.
- No-action and needs-more-input are valid outcomes.

## Data Models

Core request/candidate models live in
`src/chanta_core/self_modification_safety/candidate.py`.

Request creation input records the goal, workspace-relative target paths,
operator source refs, constraints, patch type preference, strictness, and max
target file limit.

Target refs store only sanitized workspace-relative metadata. They do not expose
private full paths or raw file content.

Patch candidates remain `candidate_only` with:

- `lifecycle_state=candidate_created`
- `diff_generated=False`
- `dry_run_checked=False`
- `review_approved=False`
- `apply_gate_opened=False`
- `applied=False`
- `file_write_enabled=False`
- `apply_patch_enabled=False`
- `materialized=False`
- `execution_enabled=False`
- `canonical_promotion_enabled=False`
- `promoted=False`

## Service Design

- `ModificationTargetResolver` uses the existing self-awareness workspace path
  policy and metadata-only path checks.
- `PatchIntentClassifier` uses deterministic keyword classification only.
- `PatchScopeBuilder` computes target count, scope status, and scope risks.
- `PatchConstraintBuilder` records the no-write/no-apply/deferred-diff and gate
  constraints inherited from v0.22.0.
- `PatchPreliminaryRiskService` produces a preliminary risk record with
  `safe_to_generate_diff=False` and `safe_to_apply=False`.
- `PatchCandidateFindingService` converts target, policy, boundary, and
  forbidden patch conditions into findings.
- `SelfModificationRequestCandidateService` creates the request and one of:
  patch candidate, no-action candidate, or needs-more-input candidate.

## Candidate Generation Rules

- Safe, workspace-relative, existing single-file requests may create
  `PatchCandidate`.
- Missing target input or missing target file may create
  `SelfModificationNeedsMoreInputCandidate`.
- Forbidden patch type, private boundary risk, secret file risk, or scope
  overflow may create `SelfModificationNoActionCandidate` or blocked request
  state.
- Generated file risk is visible as warning/finding.
- `requires_diff_generation` may be true as an intent property, but no diff is
  generated in v0.22.1.
- Patch candidates require diff preview, static safety check, dry-run, human
  review, apply gate, rollback plan, and post-apply verification before any
  future mutation-capable version may proceed.

## CLI Commands

Read-only/candidate-only commands:

```powershell
python -m chantacore.cli self-modification request create --goal "docs cleanup" --target README.md
python -m chantacore.cli self-modification candidate create --goal "docs cleanup" --target README.md
python -m chantacore.cli self-modification candidate list
python -m chantacore.cli self-modification candidate view --candidate-id <id>
python -m chantacore.cli self-modification request view --request-id <id>
python -m chantacore.cli self-modification candidate risks --candidate-id <id>
```

CLI output must show request/candidate ids, workspace-relative targets only,
`lifecycle_state=candidate_created` when a patch candidate is created,
`diff_generated=false`, `applied=false`, `file_write_enabled=false`,
`apply_patch_enabled=false`, the next required step `v0.22.2 Patch Draft / Diff
Preview`, and `No file mutation occurred.`

## Acceptance Criteria

- Modification request can be created.
- Patch candidate can be created.
- No-action and needs-more-input candidates can be created.
- Patch candidate remains `candidate_only`.
- No diff or patch draft is generated.
- No file write or patch application occurs.
- No dry-run, test, lint, or shell execution occurs.
- No LLM patch generator or LLM judge is used.
- OCEL/PIG/OCPX visibility exists.
- v0.22.0, v0.21.x, and v0.20.x tests remain passing.

## Withdrawal Conditions

Withdraw v0.22.1 readiness if it generates a diff hunk, creates a patch draft,
writes files, applies patches, runs dry-run/test/lint/shell execution, opens an
apply gate, marks a candidate applied, emits `workspace_file_changed`, exposes
private full paths, prints raw file content, prints raw secrets, activates LLM
patch generation, or uses an LLM judge.

## Validity Horizon

This restore judgment remains valid until v0.22.2 Patch Draft / Diff Preview
changes the candidate-to-diff boundary, or until the self-modification safety
contract, patch policy, lifecycle policy, workspace path policy, CLI, OCEL
mapping, PIG/OCPX report, or candidate owner changes.
