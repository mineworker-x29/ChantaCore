# v0.22.5 Human Review & Apply Gate

Korean name: 인간 검토·적용 게이트

Track: Self-Modification Safety

## Version Definition

v0.22.5 introduces OCEL-native human review decision records and apply gate state records for self-modification safety.

This version may create HumanReviewRequest, HumanReviewDecision, RollbackPlanDescriptor, ApplyGatePreconditionCheck, ApplyGateState, ApplyGateAuthorization, HumanReviewApplyGateReport, and no-action/needs-more-input candidates.

This version does not write files, call apply_patch, apply patches, emit workspace_file_changed, execute an external patch command, run shell/test/lint/import/compile/runtime checks, execute rollback, perform post-apply verification, mutate memory/persona/overlay state, mutate policy/gate/capability registry, inject context, mutate prompts, use an LLM judge, use LLM semantic approval, or perform autonomous self-edit.

## Core Principles

- Review decision is not execution.
- Review approval is not file mutation.
- Apply gate is not patch apply.
- Apply gate requires dry-run pass.
- Apply gate requires rollback plan.
- Apply gate requires explicit operator approval.
- Apply gate opening must be OCEL-visible.

## Service Design

HumanReviewSourceService loads patch candidate, patch draft, diff preview, static safety report, and dry-run report read-only.

HumanReviewRequestService creates sanitized review requests without raw patch content or raw file content.

HumanReviewDecisionService records operator or authorized-reviewer decisions. approve sets approved_for_apply_gate=true, but approved_for_direct_apply is always false.

RollbackPlanService creates a rollback descriptor and reverse-operation references. It does not execute rollback and does not output raw before content.

ApplyGatePreconditionService requires static safety, dry-run pass, dry-run review eligibility, review approval, rollback plan, clear private/secret boundaries, and unchanged target scope.

ApplyGateService opens the apply gate only when every precondition passes. An open gate creates ApplyGateAuthorization for v0.22.6 bounded patch apply. safe_to_apply remains false.

## Gate Opening Rules

Gate may open only if:

- static safety report exists and is passed or acceptable warning
- dry-run report exists and is passed or acceptable warning
- dry-run eligible_for_review=true
- review decision is approve
- approved_for_apply_gate=true
- rollback plan exists
- rollback_possible=true
- private and secret boundaries are clear
- target scope remains within scope
- no file write has occurred
- no patch has been applied
- no workspace_file_changed effect exists

Open gate output:

- gate_status=open
- apply_gate_opened=true
- eligible_for_bounded_apply=true
- safe_to_apply=false
- patch_applied=false
- file_write_performed=false
- workspace_file_changed_emitted=false
- authorized_next_version=v0.22.6

Actual bounded apply is v0.22.6 responsibility.
