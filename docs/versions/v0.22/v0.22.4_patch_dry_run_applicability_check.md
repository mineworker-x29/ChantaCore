# v0.22.4 Patch Dry-run / Applicability Check

Korean name: 패치 Dry-run·적용가능성 검사

Track: Self-Modification Safety

## Version Definition

v0.22.4 introduces the OCEL-native in-memory dry-run and applicability check step for self-modification patch drafts and diff previews.

This version may load patch candidate, patch draft, diff preview, and static safety report read models. It may collect bounded/sanitized target snapshots, check anchor applicability, simulate operation application in memory, detect conflicts, compute result hashes, and create a PatchDryRunReport.

This version does not write files, call apply_patch, apply patches to the workspace, execute an external patch command, run shell/test/lint/import/compile/runtime checks, approve human review, open an apply gate, execute rollback, perform post-apply verification, use an LLM judge, or mutate memory/persona/overlay state.

## Core Principles

- Patch dry-run is an in-memory applicability check.
- Patch dry-run is not file mutation.
- Patch dry-run pass is not review approval.
- Patch dry-run pass is not apply permission.
- Patch dry-run pass only permits moving to the human review stage.
- Patch dry-run result must be recorded before human review.

## Service Design

PatchDryRunSourceService loads patch candidate, patch draft, diff preview, and static safety report objects through existing owners. It collects target snapshots from bounded/sanitized context and does not output raw full target content.

PatchInMemoryApplicabilityEngine simulates operations over an in-memory buffer only. It may set applied_in_memory=true for the buffer, while file_write_performed, patch_applied, and workspace_file_changed_emitted remain false.

PatchAnchorApplicabilityService checks exact/eof/line-range style anchors and reports missing or ambiguous anchors.

PatchDryRunConflictDetector detects operation conflicts such as overlapping hunks.

PatchDryRunReportService computes dry_run_status, eligible_for_review, operation counts, conflict counts, and boundary flags. safe_to_apply remains false in every report.

## Status Rules

passed:

- static safety report exists and is passed or acceptable warning
- target snapshots are available
- anchors match exactly once or deterministically
- all operations would apply in memory
- no conflicts exist
- no private/secret boundary block exists
- no file write or workspace_file_changed effect occurs

warning:

- redacted/truncated context creates uncertainty
- line ending or encoding is uncertain
- low-confidence but unique anchor is present
- static safety warning exists but is not a blocker

failed:

- anchor missing
- anchor ambiguous
- old text mismatch
- context mismatch
- operation conflict
- overlapping hunks
- target changed since preview

blocked:

- missing static safety report
- static safety failed or blocked
- target snapshot unavailable under strict mode
- private/secret boundary block
- candidate/draft/preview already applied
- dry-run would require raw private content exposure
