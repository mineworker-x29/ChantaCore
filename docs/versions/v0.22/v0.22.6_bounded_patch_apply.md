# v0.22.6 Bounded Patch Apply

Korean name: 제한적 패치 적용

Track: Self-Modification Safety

## Version Definition

v0.22.6 is the first controlled mutation step in the OCEL-native self-modification safety track. It may write an approved workspace-relative text file only after apply gate authorization, pre-apply revalidation, and rollback plan availability have been checked.

Bounded apply is controlled mutation. Bounded apply is not autonomous self-editing. Patch apply requires explicit apply authorization. Apply authorization must be single-use. Pre-apply revalidation is mandatory. Rollback plan must exist before apply. Workspace file change must be OCEL-visible. Post-apply verification is still required.

## Relationship

v0.22.0 defines the safety contract. v0.22.1 creates request and patch candidate records. v0.22.2 creates patch draft and diff preview artifacts. v0.22.3 creates static safety reports. v0.22.4 creates in-memory dry-run/applicability reports. v0.22.5 creates human review and apply gate authorization. v0.22.6 consumes that authorization for a bounded write. v0.22.7 remains responsible for post-apply verification and outcome recording.

## Included Scope

- Validate `ApplyGateAuthorization`.
- Revalidate target hash and operation scope immediately before apply.
- Apply approved operations through `BoundedWorkspaceWriter`.
- Write only approved workspace-relative text file targets.
- Emit `workspace_file_changed` through `WorkspaceFileChangedEventService`.
- Consume single-use authorization after successful apply.
- Create `BoundedPatchApplyTransaction` and `BoundedPatchApplyReport`.

## Non-Goals

v0.22.6 does not apply without authorization, reuse consumed or expired authorization, modify unapproved targets or operations, run external patch commands, run shell/test/lint, run post-apply verification, execute rollback, use an LLM judge, rewrite patches with an LLM, mutate memory/persona/overlay, mutate policy/gate/capability registries, inject context, mutate prompts, or expose raw full file content.

## Service Design

- `BoundedPatchApplySourceService` loads apply gate, authorization, draft, preview, dry-run report, static report, and rollback plan read-only.
- `ApplyAuthorizationValidationService` blocks missing, closed, consumed, expired, mismatched, non-single-use, or unconfirmed authorization.
- `PreApplyRevalidationService` re-reads the current target, checks the current hash, checks target boundaries, and verifies operation applicability.
- `BoundedPatchApplyEngine` applies approved operations to an internal buffer and delegates the only real write to `BoundedWorkspaceWriter`.
- `BoundedWorkspaceWriter` enforces workspace-relative path resolution, checks the expected before hash immediately before writing, writes text via a temporary file replacement, and does not print raw content.
- `WorkspaceFileChangedEventService` creates the only `workspace_file_changed` event id after a successful bounded write.
- `ApplyAuthorizationConsumeService` marks the single-use authorization consumed after success.
- `BoundedPatchApplyReportService` creates the transaction/report and keeps post-apply verification pending.

## Status Rules

`applied` requires valid authorization, passed pre-apply revalidation, writer success, `workspace_file_changed` event emission, authorization consumption, and `post_apply_verification_required=true`.

`blocked` is used for missing/closed gate, missing/consumed/expired authorization, missing confirmation, missing/impossible rollback plan, target hash drift, private/secret boundary risk, or operation revalidation failure.

`failed` is used for writer failure before any file change. `partial_failed` remains a status concept, but default `max_files=1` prevents normal multi-file partial application.

## CLI Commands

- `python -m chanta_core.cli self-modification apply bounded --apply-gate-id <id> --authorization-id <id> --confirm "APPLY BOUNDED PATCH"`
- `python -m chanta_core.cli self-modification apply report --report-id <id>`
- `python -m chanta_core.cli self-modification apply transaction --transaction-id <id>`
- `python -m chanta_core.cli self-modification apply changes --report-id <id>`

CLI output shows apply status, transaction id, changed file count, applied operation count, before/after hashes when available, workspace file changed event id when available, authorization consumed, rollback plan availability, `post_apply_verification_required=true`, `post_apply_verified=false`, and next required step v0.22.7. CLI output must not print raw full file content, private full paths, or raw secrets.

## PIG/OCPX

PIG subject is `bounded_patch_apply` with `bounded_file_write_enabled=True`, `external_patch_tool_allowed=False`, `shell_executed=False`, `test_lint_executed=False`, `post_apply_verified=False`, and `llm_judge_enabled=False`.

OCPX state is `self_modification_bounded_patch_applied`. Target read models include `BoundedPatchApplyState`, `WorkspaceFileChangeState`, `PostApplyVerificationRequiredState`, and `RollbackEligibilityState`.

## Tests

Tests cover blocking without gate/authorization/confirmation/rollback, consumed authorization, target hash drift, expected-before-hash checks, successful approved workspace-relative write, before/after hash computation, `workspace_file_changed` event id emission, authorization consumption, report flags, no shell/test/lint, no LLM judge, PIG/OCPX, and v0.22 regression preservation.

## Acceptance Criteria

- Bounded apply can modify approved workspace file only with valid apply gate authorization.
- Apply blocks on missing, invalid, consumed, or expired authorization.
- Apply revalidates target hash and operation before mutation.
- Apply emits `workspace_file_changed` after successful bounded write.
- Authorization is consumed.
- Post-apply verification is required but not performed.
- No shell, test/lint, external patch command, rollback execution, post-apply verification, or LLM judge occurs.
- OCEL/PIG/OCPX visibility exists.

## Restore Procedure

To restore this version, recreate the bounded apply models, source/validation/revalidation/writer/event/consume/report services, OCEL mapping entries, CLI views, PIG/OCPX projections, and tests listed above. The decisive invariant is that real file writes exist only in `BoundedWorkspaceWriter`, and `workspace_file_changed` is emitted only after a successful bounded write.

## Withdrawal Conditions

Withdraw this version claim if bounded apply can run without open apply gate authorization, if consumed or expired authorization can be reused, if unapproved targets or operations can be mutated, if file writes occur outside `BoundedWorkspaceWriter`, if `workspace_file_changed` is emitted before a successful write, if shell/test/lint/external patch commands run, if rollback or post-apply verification executes, if raw full file content/private full paths/secrets are printed, or if an LLM judge/rewrite path is introduced.

## Validity Horizon

Valid until post-apply verification and outcome are recorded, or until target file, patch draft, dry-run report, apply gate authorization, rollback plan, writer policy, or OCEL mapping changes.
