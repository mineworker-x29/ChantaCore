# v0.22.8 Self-Modification Workbench

Korean name: 자기수정 워크벤치

Track: Self-Modification Safety

## Definition

v0.22.8 adds a read-only OCEL-native workbench over the full v0.22.0 through v0.22.7 self-modification safety pipeline.

The workbench is not modification. Workbench visibility does not authorize apply. The workbench does not consume authorization. The workbench does not execute rollback. The workbench does not run tests or shell.

## Relationship

- v0.22.0 defines the Self-Modification Safety Contract.
- v0.22.1 defines request and patch candidate records.
- v0.22.2 defines patch draft and diff preview records.
- v0.22.3 defines static safety reports.
- v0.22.4 defines dry-run/applicability reports.
- v0.22.5 defines human review, rollback descriptor, and apply gate records.
- v0.22.6 defines bounded patch apply and workspace change records.
- v0.22.7 defines post-apply verification and outcome records.
- v0.22.8 aggregates those artifacts into sanitized read-only workbench views.

## Included Scope

- Pipeline status and next required action.
- Subject status across the self-modification safety subjects.
- Authorization views without consuming authorization.
- Workspace change views without reading raw file content.
- Findings and timeline views.
- Safety boundary summary.
- OCEL coverage view.
- Readiness for v0.22.9 Self-Modification Safety Consolidation.

## Non-Goals

v0.22.8 does not write files, call `apply_patch`, apply patches, emit a new `workspace_file_changed` event, execute rollback, run post-apply verification, rewrite outcomes, create review decisions, mutate apply gates, consume authorization, run shell/test/lint/import checks, use network, connect MCP, load plugins, mutate memory/persona/overlay, mutate policy/gate/capability registry, inject context, mutate prompts, use an LLM judge, or run an autonomous repair loop.

## Service Design

- `SelfModificationWorkbenchSourceService` reads existing in-memory artifact owners.
- `SelfModificationSubjectStatusService` builds subject status summaries.
- `SelfModificationPipelineViewService` links request through outcome when available.
- `SelfModificationSafetyBoundaryService` counts safety-boundary risks.
- `SelfModificationAuthorizationViewService` displays authorization state without consuming it.
- `SelfModificationChangeViewService` displays sanitized workspace change metadata.
- `SelfModificationFindingViewService` aggregates findings by severity and stage.
- `SelfModificationTimelineService` builds deterministic timeline events.
- `SelfModificationOCELCoverageService` checks required mapping coverage.
- `SelfModificationReadinessService` computes v0.22.9 readiness.
- `SelfModificationWorkbenchService` builds the snapshot and PIG/OCPX reports.

## Status and Readiness Rules

Ready requires visible pipeline artifacts, no unauthorized write, no apply without gate, no consumed authorization reuse, no unverified applied transaction, no critical/error open finding, complete workspace change traces, and acceptable OCEL coverage.

Warning covers pending candidates, pending review/dry-run/outcome stages, non-critical findings, rollback recommendations pending operator review, and optional OCEL relation gaps.

Blocked covers unauthorized write, workspace change without transaction, apply without gate, consumed authorization reuse, missing post-apply verification after apply, missing outcome after verification, critical safety finding, and raw secret/private exposure risk.

## OCEL Mapping

Object types include workbench snapshot, subject status view, pipeline item, safety boundary summary, authorization view, change view, finding view, timeline event, OCEL coverage view, readiness view, and the existing v0.22 pipeline artifact types.

Event types include workbench requested, snapshot created, stage views, findings/timeline/safety boundary viewed, workbench warning created, and workbench blocker detected.

Relation types include views pipeline/artifacts/authorization/safety/timeline/findings/coverage, computes readiness, detects blockers, and derivation from v0.22 pipeline artifacts.

v0.22.8 workbench effect type is only `read_only_observation`. It does not emit `workspace_file_changed`, `outcome_recorded`, or `gate_state_created`.

## PIG/OCPX

PIG subject: `self_modification_workbench`

PIG records read-only mode, no mutation performed, no file write, no workspace change event emission, no rollback execution, no shell/test/lint execution, and LLM judge disabled.

OCPX state: `self_modification_workbench_snapshot_created`

Target read models include `SelfModificationWorkbenchState`, `SelfModificationPipelineState`, `SelfModificationSafetyBoundaryState`, and `SelfModificationReadinessState`.

## Acceptance Criteria

- Workbench snapshot builds.
- Full v0.22.0 through v0.22.7 pipeline is visible.
- Safety boundary summary detects unauthorized mutation patterns.
- Authorization, change, finding, timeline, coverage, and readiness views exist.
- Readiness for v0.22.9 is computed.
- Workbench output is sanitized.
- No apply, rollback, post-apply verification, outcome rewrite, authorization consume, shell/test/lint, or LLM judge occurs.
- OCEL/PIG/OCPX visibility exists.
- v0.22.7 through v0.22.0 and v0.21.x/v0.20.x tests remain valid.

## Withdrawal Conditions

Withdraw v0.22.8 acceptance if the workbench writes files, applies patches, emits a new workspace change event, executes rollback, triggers post-apply verification, rewrites outcomes, consumes authorization, creates a review decision, mutates policy/gate/capability registry, exposes private raw content, or uses an LLM judge.

## Validity Horizon

Valid until self-modification artifacts, OCEL mapping requirements, or v0.22.9 consolidation rules change.
