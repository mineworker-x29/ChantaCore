# v0.22.7 Post-Apply Verification & Outcome

Korean name: 적용 후 검증·결과 기록

Track: Self-Modification Safety

## Definition

v0.22.7 adds the OCEL-native read-only verification and outcome recording stage after v0.22.6 Bounded Patch Apply.

Post-apply verification is not another patch.
Verification failure does not trigger automatic rollback.
Outcome recording must be OCEL-visible.
Bounded apply is incomplete until outcome is recorded.

## Relationship

- v0.22.0 defines the Self-Modification Safety Contract.
- v0.22.1 creates self-modification requests and patch candidates.
- v0.22.2 creates patch drafts and diff previews.
- v0.22.3 creates static safety reports.
- v0.22.4 creates in-memory dry-run/applicability reports.
- v0.22.5 records human review and apply gate authorization.
- v0.22.6 performs bounded patch apply under authorization.
- v0.22.7 verifies the bounded apply result and records the outcome.
- v0.21.9 and v0.20.9 remain prerequisite OCEL-native self-awareness foundations.

## Included Scope

- Load `BoundedPatchApplyReport`, `BoundedPatchApplyTransaction`, `WorkspaceFileChangeRecord`, existing `workspace_file_changed` event references, rollback plan, and apply authorization.
- Collect current target snapshots read-only.
- Verify before, after, and current hashes.
- Verify OCEL trace completeness.
- Verify apply authorization was consumed.
- Verify rollback readiness.
- Verify changed scope against approved operation scope.
- Run bounded static safety regression checks without executing code.
- Create `PostApplyVerificationReport`.
- Create `ModificationOutcomeRecord`.
- Create needs-more-input and rollback-recommended candidates.

## Non-Goals

v0.22.7 does not write files, call `apply_patch`, apply additional patches, execute rollback, run shell, run tests, run lint, perform import/compile/runtime checks, use network, connect MCP, load plugins, mutate memory/persona/overlay, mutate policy/gate/capability registry, use an LLM judge, or emit a new `workspace_file_changed` event.

## Service Design

- `PostApplyVerificationSourceService` loads apply sources read-only.
- `PostApplyTargetSnapshotService` reads current targets through workspace path policy and emits only metadata/hashes.
- `PostApplyHashVerifier` compares recorded after-hash with current hash.
- `PostApplyOCELTraceVerifier` checks the existing workspace change trace.
- `PostApplyAuthorizationVerifier` checks consumed authorization state.
- `PostApplyRollbackReadinessVerifier` checks rollback descriptor readiness without execution.
- `PostApplyScopeVerifier` compares approved and changed targets/operations.
- `PostApplySafetyRegressionVerifier` performs static pattern checks without code execution.
- `PostApplyVerificationReportService` aggregates status.
- `ModificationOutcomeRecordService` creates an OCEL-visible outcome record without memory/persona/overlay mutation.

## Verification Rules

Passed requires an apply report, transaction, change record, existing workspace change event, current hash matching the recorded after-hash, consumed authorization, matched scope, available rollback plan, and no hard post-apply regression.

Warnings cover partial rollback readiness, optional trace gaps, or non-critical regression uncertainty.

Failed covers after-hash mismatch, missing workspace change event, unconsumed authorization, unexpected target or operation, scope mismatch, or hard regression.

Blocked covers missing apply report, missing transaction, missing change record, missing current file, private/secret boundary risk, or verification that would require private raw content exposure.

## Outcome Rules

- `passed` -> `applied_verified`
- `warning` -> `applied_with_warnings`
- `failed` with rollback possible -> `rollback_recommended`
- `failed` without rollback readiness -> `verification_failed`
- `blocked` -> `blocked` or `needs_more_input`

Outcome recording does not auto-rollback and does not promote memory, persona, overlay, or canonical state.

## PIG/OCPX

PIG subject: `post_apply_verification_outcome`

PIG records:
- additional file write not performed
- additional patch not applied
- rollback not executed
- shell/test/lint not executed
- LLM judge disabled
- memory promotion disabled

OCPX state: `self_modification_post_apply_verified_outcome_recorded`

Target read models include `PostApplyVerificationState`, `ModificationOutcomeState`, `RollbackRecommendationState`, and `SelfModificationCompletionState`.

## Acceptance Criteria

- Post-apply verification report can be created from bounded apply output.
- Hash, OCEL trace, authorization, rollback readiness, scope, and safety regression are checked.
- Outcome record is created.
- Verification failure may recommend rollback but does not execute rollback.
- No additional file write, `apply_patch`, new `workspace_file_changed` event, shell/test/lint, rollback execution, or LLM judge occurs.
- OCEL/PIG/OCPX visibility exists.
- v0.22.6 through v0.22.0 and v0.21.x/v0.20.x tests remain valid.

## Withdrawal Conditions

Withdraw v0.22.7 acceptance if it writes files, applies an additional patch, emits a new workspace change event, executes rollback, runs shell/test/lint/import checks, exposes raw private content, or uses an LLM judge.

## Validity Horizon

Valid until the target file changes again, rollback/outcome state changes, or a later version changes the post-apply verification contract.
