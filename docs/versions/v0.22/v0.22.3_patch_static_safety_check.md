# v0.22.3 Patch Static Safety Check

Korean name: 패치 정적 안전성 검사

Track: Self-Modification Safety

Release status: implemented as read-only/report-only static safety checks.

## Version Definition

v0.22.3 introduces the OCEL-native static safety checking step for patch drafts and diff previews.

Static safety check is not dry-run.
Static safety report is not approval.
Static safety pass is not clean application.
Static safety pass does not imply tests pass.
Static safety pass does not imply patch apply.

This version creates static safety rules, rule results, findings, category results, and reports. It may mark a patch draft eligible_for_dry_run, but it must never mark safe_to_apply=true.

## Relationship To Earlier Versions

v0.22.0 defines the Self-Modification Safety Contract, lifecycle policy, patch policy, and gate requirements.

v0.22.1 creates modification requests and patch candidates.

v0.22.2 creates patch drafts and diff previews as candidate-only preview artifacts.

v0.22.3 reads those artifacts and creates a static safety report. It does not mutate the draft, preview, candidate, workspace, policy, memory, persona, or overlay state.

v0.21.9 remains the Deep Self-Introspection Foundation v1 source. v0.20.9 remains the Self-Awareness Foundation v1 source.

## Data Models

PatchStaticSafetyCheckRequest selects a patch candidate, patch draft, and/or diff preview, plus which safety categories to include.

PatchStaticSafetyRule describes a static rule in one category: path, operation, scope, content_pattern, secret_private, lifecycle, or policy.

PatchStaticSafetyRuleResult records the deterministic outcome of one rule.

PatchStaticSafetyFinding is a sanitized finding derived from rule results.

PatchStaticSafetyCategoryResult aggregates rule and finding counts by category.

PatchStaticSafetyReport records all rule results, category results, findings, status, dry-run eligibility, and safety flags. safe_to_apply is always false in this version.

PatchStaticSafetyNoActionCandidate and PatchStaticSafetyNeedsMoreInputCandidate are candidate-only outcomes. They do not approve, apply, or execute anything.

## Service Design

PatchStaticSafetySourceService loads patch candidate, patch draft, diff preview, patch policy, and lifecycle policy read-only. It does not write files or execute dry-run.

PatchStaticRuleRegistry defines deterministic static safety rules.

PatchPathSafetyVerifier checks workspace-relative paths and private path risk.

PatchOperationSafetyVerifier checks allowed operation types.

PatchScopeSafetyVerifier checks hunk, added-line, and removed-line limits.

PatchContentPatternSafetyVerifier checks sanitized operation/diff preview text for dangerous patterns, permission grant patterns, and shell/network/MCP/plugin patterns.

PatchSecretPrivateSafetyVerifier checks secret, binary, and generated file risks.

PatchLifecycleSafetyVerifier checks preview_created state, not-applied state, write/apply flags, dry-run visibility, review visibility, and apply-gate visibility.

PatchPolicyConsistencyVerifier checks draft/preview presence and records that static safety is not safe-to-apply.

SelfModificationStaticSafetyService builds the report and read-only CLI/PIG/OCPX surfaces.

## Static Safety Rules

Rules cover:

- workspace-relative path
- private path risk
- secret file risk
- binary file risk
- generated file risk
- allowed operation type
- hunk count limit
- added/removed line limits
- dangerous content patterns
- permission grant patterns
- shell/network/MCP/plugin patterns
- mutation/apply flags
- preview_created lifecycle
- applied=false state
- dry_run_not_performed visibility
- review_not_approved visibility
- apply_gate_not_opened visibility
- draft/preview presence
- not_safe_to_apply policy consistency

dry_run_not_performed, review_not_approved, and apply_gate_not_opened are expected in v0.22.3. They are visible informational findings, not approvals.

## Status Rules

passed if required draft/preview sources exist, paths are workspace-relative, operation types are allowed, scope is within limits, dangerous patterns are absent, lifecycle is preview_created, applied=false, and write/apply flags are false.

warning if non-blocking risks such as generated file risk or truncation concerns are present.

failed if forbidden operations, scope limits, dangerous patterns, permission grant patterns, shell/network/MCP/plugin patterns, mutation flags, or inconsistent state are present.

blocked if patch draft or diff preview is missing, the candidate/draft/preview is already applied, secret/private/binary hard-block risk is present, or raw private content exposure would be required.

## Acceptance Criteria

- Static safety report can be created from patch draft/diff preview.
- Path, operation, scope, content, secret/private, lifecycle, and policy rules run.
- Static safety may mark eligible_for_dry_run.
- Static safety never marks safe_to_apply=true.
- No dry-run, review approval, apply gate, apply, file write, apply_patch, test/lint/shell execution, or LLM judge occurs.
- OCEL/PIG/OCPX visibility exists.
- v0.22.2, v0.22.1, v0.22.0, v0.21.x, and v0.20.x tests still pass.

## Withdrawal Conditions

Withdraw this version claim if v0.22.3 writes files, calls apply_patch, applies patches, runs dry-run, executes tests/lint/shell, approves review, opens an apply gate, performs rollback, performs post-apply verification, emits workspace_file_changed, marks safe_to_apply=true, exposes raw full file content, exposes private full paths, exposes raw secrets, or uses an LLM judge.

## Validity Horizon

This contract is valid until v0.22.4 Patch Dry-run / Applicability Check. v0.22.4 may check dry-run/applicability, but static safety remains distinct from dry-run, approval, and apply.
