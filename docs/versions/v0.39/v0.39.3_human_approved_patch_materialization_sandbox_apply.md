# v0.39.3 Human-approved Patch Materialization & Sandbox Apply

Release name: **v0.39.3 Human-approved Patch Materialization & Sandbox Apply**.

Parent track: **Human-approved Sandbox Repair Apply & Re-test Loop with PI-native Self-Prompting Mission Loop Boundary**.

v0.39.3 builds on v0.39.2 Sandbox Repair Workspace Isolation Contract, v0.39.1 Approval Artifact Intake & Authenticity Gate, v0.39.0 boundary foundation, and v0.38.9 Bounded Repair Proposal Loop v1. It consumes approval gate metadata, workspace isolation metadata, proposed patch envelope metadata, safety report metadata, and optional human review packet metadata.

## Boundary

v0.39.3 is the first stage that may mutate sandbox target files.

v0.39.3 mutates sandbox target files only. It does not mutate live workspace files, reference corpus files, secret paths, or files outside the validated sandbox root.

The only opened mutation surface is bounded sandbox target text replacement after approval gate validation, workspace isolation validation, sandbox root containment validation, exact original-text match, and target safety checks.

## Definitions

Human-approved patch materialization is the conversion of previously proposed and approved repair hunk metadata into an in-memory patch artifact. It is not patch file export.

Sandbox-only apply is bounded text replacement inside an explicitly supplied and validated sandbox root.

Bounded sandbox target read is the read of a specific target file under the validated sandbox root for exact original-text matching. It is not filesystem scanning.

Bounded sandbox target write is the write of a specific target file under the validated sandbox root after exact original-text replacement succeeds. It is not live workspace write.

Exact text replacement apply replaces one unambiguous occurrence of the approved original text with the approved replacement text.

Apply preflight validates approval gate metadata, workspace isolation metadata, safety report metadata, proposed patch metadata, target containment, target exclusions, and exact-match requirements before mutation.

Sandbox apply transaction records a single-cycle sandbox-only apply attempt. It records operation ids, target ids, atomicity metadata, rollback/discard metadata, and prohibited runtime confirmations.

Sandbox apply result records applied and failed operations, changed target refs, pre-apply digests, post-apply digests, failure kinds, and future post-apply re-test input readiness.

## Distinctions

Sandbox patch materialization is not patch file export.

Sandbox apply is not live apply.

Sandbox apply is not apply_patch or git apply.

Sandbox apply result is not repair correctness.

Future post-apply retest input is not test execution.

Sandbox apply is not autonomous repair.

Exact original-text match is required to prevent applying a stale, overbroad, or mismatched repair.

Path containment is required so a target relative path cannot escape the sandbox root.

Shell, subprocess, apply_patch, and git apply remain blocked because v0.39.3 is a bounded text replacement stage, not a command execution stage.

Human handoff remains mandatory after sandbox apply. Self-prompting remains draft/future-gated, not self-execution.

## Acceptance Criteria

The release is acceptable only if patch materialization is in-memory only, sandbox target validation rejects unsafe targets, preflight enforces approval/workspace/safety/patch gates, exact text replacement mutates only sandbox-contained target files, missing or ambiguous original text fails, transaction metadata remains single-cycle, result metadata does not claim tests or repair correctness, and focused tests pass.

## Withdrawal Conditions

Withdraw the v0.39.3 readiness claim if implementation code creates workspaces, runs git, scans directories, writes patch files, applies to live workspace, calls apply_patch, calls git apply, runs shell/subprocess, runs tests, executes repair loops, generates or executes self-prompts, invokes subagents, invokes model providers, invokes external agents, runs autonomous loops, implements Dominion runtime, certifies production, or sets any unsafe readiness flag true.

## Handoff

v0.39.3 hands off to v0.39.4 Post-Apply Controlled Re-test as design-stage metadata only. It supplies sandbox apply result metadata, pre/post digest metadata, changed sandbox target refs, and future retest input readiness. It does not run tests.
