## Definitions

Approval artifact means supplied human approval evidence represented as bounded metadata. It can include explicit human approval text, signed text metadata, reviewer attestation metadata, review packet attached approval, approval request response metadata, manual operator note, or test fixture approval. v0.39.1 does not perform external signature verification or identity-provider verification.

Approval artifact intake means creating a structured intake object from supplied approval text and supplied metadata. v0.39.1 may mark approval artifact received.

Approval authenticity assessment means checking supplied metadata signals such as reviewer id, reviewer role, approval phrase, timestamp, nonce, digest, scope, patch envelope reference, review packet reference, and safety report reference. Authenticity assessment is not external identity verification.

Approval scope binding means binding the approval artifact to proposed patch envelope, proposed file change, proposed hunk, safety-validated patch scope, human review packet scope, or sandbox apply future scope. Live apply scope is forbidden and must be blocked or rejected.

Approval expiration/freshness assessment means representing whether supplied approval timestamps and expiration metadata are fresh, near expiry, expired, missing, or invalid. Approval freshness is not test pass.

Approval-to-patch-envelope binding, approval-to-safety-report binding, and approval-to-human-review-packet binding are metadata consistency checks. Binding consistency does not grant apply.

Approval process-state transition gate means a PI-native process-state gate from review_packet_created to approval_artifact_received, approval_artifact_schema_valid, approval_authenticity_assessed, approval_scope_bound_to_patch, approval_freshness_validated, future_sandbox_workspace_isolation_eligible, and future_sandbox_apply_precondition_satisfied. This gate is not runtime authority.

Approval artifact differs from approval grant. Approval artifact validation differs from apply permission. Future sandbox apply precondition differs from sandbox apply. Reviewer metadata differs from external identity verification. Self-Prompting Mission Loop policy input differs from self-prompt execution.
