# v0.24.0 Internal Provider Contract

Korean name: 내부 Provider 계약

Track: Internal Provider / Local Runtime Provider

Previous: v0.23.9 Internal Dominion Consolidation / Release Readiness

Next: v0.24.1 Provider Registry & Capability Surface

## Objective

v0.24.0 defines the contract-only foundation for ChantaCore internal providers:
provider types, capability contracts, effect policies, permission policies,
invocation boundaries, observability requirements, safety boundaries, and roadmap
boundaries.

## Policy

v0.24.x is Internal Provider / Local Runtime Provider. v0.24.0 is contract-only.
It does not implement actual provider invocation, workspace file read execution,
repository search execution, file content extraction, OCEL/PIG/OCPX live
inspection execution, local command execution, external provider adapters,
General Agent Usability, Schumpeter split, GrowthKernel dependency, or LLM judge.

## Principles

- Provider is not arbitrary tool execution.
- Provider is not external adapter.
- Provider is not agent UX.
- Provider is an OCEL-visible internal capability surface.
- Provider invocation must be gated by effect type.
- Read-only provider is still governed by policy.
- Local runtime provider is not unrestricted shell.
- Internal provider must never expose credentials or raw secrets.
- Internal provider must remain public-core safe.

## Provider Types

v0.24.0 declares contract descriptors for workspace_read_provider,
repository_search_provider, file_read_provider, ocel_inspection_provider,
pig_inspection_provider, ocpx_projection_provider, local_runtime_provider,
diagnostic_provider, candidate_generation_provider, and unknown.

All v0.24.0 internal provider types are public-core descriptors, not external
adapters. They do not require provider API calls, external runtime touch, or
credential materialization.

## Data Models

The contract defines InternalProviderContract, InternalProviderTypeDescriptor,
InternalProviderCapabilityContract, InternalProviderEffectPolicy,
InternalProviderPermissionPolicy, InternalProviderInvocationBoundary,
InternalProviderObservabilityContract, InternalProviderSafetyBoundary,
InternalProviderRoadmapBoundary, InternalProviderContractFinding, and
InternalProviderContractReport.

## Service Design

Services build provider type descriptors, effect policy, permission policy,
invocation boundary, observability contract, safety boundary, roadmap boundary,
findings, and a contract report. No service invokes a provider or reads files.

## Effect Policy

Allowed effects in v0.24.0 are read_only_observation, state_candidate_created,
and provider_contract_declared. Future provider effects are listed as future
contract vocabulary only. v0.24.0 forbids provider_invoked,
workspace_file_read_executed, repository_search_executed, file_content_extracted,
local_command_executed, bounded_local_command_executed,
unrestricted_shell_executed, network_accessed, package_installed,
destructive_command_executed, external_runtime_touched,
external_control_dispatched, credential_exposed, raw_secret_output,
memory_mutated, persona_mutated, and external_provider_called.

## Permission Policy

The permission policy is deny-by-default. Read-only and candidate creation
surfaces require policy. Future local runtime execution requires gate,
allowlist, timeout, output cap, redaction, and side-effect scan. Network, package
install, destructive command, credential access, and secret output are forbidden
by default.

## Invocation Boundary

v0.24.0 disables provider invocation, workspace read execution, repository search
execution, file read execution, OCEL/PIG/OCPX inspection execution, local runtime
execution, local command execution, external provider invocation, shell,
network, MCP, plugin loading, and LLM judge.

## Observability

Provider contracts require OCEL, PIG, OCPX, and execution-envelope visibility.
Future provider invocation must emit an event, record effect type, sanitize
results, and provide explainable failures.

## Safety Boundary

All dangerous counts are expected to be zero: provider invocation, workspace
file read execution, repository search execution, file content extraction, local
command execution, bounded local command execution, unrestricted shell, provider
API call, external runtime touch, external dispatch, network access, package
install, destructive command, credential exposure, raw secret output, LLM judge,
external provider adapter, and Schumpeter split.

## Roadmap Boundary

The next version is v0.24.1 Provider Registry & Capability Surface. General
Agent Usability remains deferred to v0.25.x, Workspace Agent Workbench to
v0.26.x, Memory Candidate & Continuity to v0.27.x, Schumpeter split preparation
to v0.28.x, and External Provider Adapters to v0.29.x+.

## OCEL Mapping

v0.24.0 declares object, event, relation, and effect mappings for internal
provider contract visibility. OCEL remains the canonical substrate.

## Tests

## Acceptance Criteria

InternalProviderContract can be created. Provider types, effect policy,
permission policy, invocation boundary, observability contract, safety boundary,
roadmap boundary, and contract report exist. ready_for_v0_24_1 may be true,
ready_for_v0_25 remains false, and no provider invocation, workspace/repo/file
read execution, local command execution, external adapter, Schumpeter split,
credential output, or LLM judge is introduced.

## Withdrawal Conditions

Withdraw this contract readiness if provider invocation, read execution, local
command execution, credential exposure, raw secret output, external adapter
logic, Schumpeter split, GrowthKernel active dependency, or future-track UX work
appears in v0.24.0.

## Validity Horizon

Valid until provider registry implementation begins or internal provider policy
changes.
