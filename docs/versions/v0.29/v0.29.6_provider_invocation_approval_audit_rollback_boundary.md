# v0.29.6 Provider Invocation Approval / Audit / Rollback Boundary

## Version Definition

- Version: v0.29.6
- Name: Provider Invocation Approval / Audit / Rollback Boundary
- Korean name: Provider 호출 승인·감사·Rollback 경계
- Track: External Skill / External Provider Adapter Development
- Previous: v0.29.5 Adapter Invocation Candidate / Dry-Run Plan
- Next: v0.29.7 External Skill Packaging / Certification Matrix

v0.29.6 is approval/audit/rollback-boundary-only. It creates approval requirements, approval candidates, decision records, scope summaries, expiry and revalidation policies, audit plans, audit trails, OCEL trace plans, result boundaries, failure classifications, rollback plans, no-op boundaries, retry deferral policies, gates, and reports. It does not grant approval, invoke providers, call network, access credentials, retrieve secrets, execute commands, perform side effects, mutate files, execute rollback, automatically retry, or implement live adapters.

## Objective

The objective is to consume v0.29.5 invocation candidate and dry-run outputs, then define the approval, audit, rollback, no-op, failure, result, and OCEL boundaries required before v0.29.7 certification work. Approval/audit/rollback readiness is not live invocation readiness.

## Relationships

v0.29.5 produced invocation candidates, payload previews, effect previews, risk previews, dry-run plans, dry-run reports, no-op plans, and failure mode previews. v0.29.6 references those artifacts as metadata. Invocation candidate is not invocation, dry-run success is not approval, risk preview is not approval, effect preview is not side effect, and no-op remains a valid outcome.

v0.29.7 must create External Skill Packaging / Certification Matrix. v0.29.6 may prepare certification inputs, but approval boundary is not live invocation readiness, audit boundary is not provider result persistence, rollback plan is not rollback execution, and certification remains future work.

## Service Design

The prerequisite source service loads metadata and refs from v0.29.5 invocation candidate reports, v0.29.4 credential/network boundary reports, v0.29.3 permission/safety reports, v0.29.2 mock harness reports, v0.29.1 registry reports, and v0.29.0 audit/rollback/OCEL/prohibition contracts. It does not load raw secrets, raw traces, raw transcripts, raw provider outputs, or company-private material.

The approval service builds approval requirements, candidates, decision candidates, decision records, scope summaries, expiry policy, and revalidation policy. The audit service builds audit policy, event plans, and audit trail. The OCEL service builds trace policy and trace plans. Result, failure, rollback, no-op, retry deferral, gate, finding, report, PIG, OCPX, and CLI services are generated without live execution.

## Approval Boundary

Approval requirements record that explicit user approval is required later. Approval candidates are not grants and are not execution. Decision candidates may propose deny, defer, require_user_approval, require_certification, require_limited_preview_gate, mock_only, dry_run_only, no_op, or unknown. Decision records record those future-boundary decisions while approval_granted_now, provider_invoked_now, network_called_now, credential_accessed_now, command_executed_now, and rollback_executed_now remain false.

Scope summaries describe provider, credential, network, payload, and result scopes. Approval expiry forbids permanent approval, approval reuse without revalidation, expired approval reuse, and scope-change reuse. Revalidation covers permission, safety, credential, network, payload, and result boundaries, and revalidation is not execution.

## Audit And OCEL

Audit policy requires approval, invocation candidate, credential reference, network candidate, payload preview, effect preview, risk preview, failure classification, rollback plan, and future provider result audit. Audit forbids raw provider output, credential value, and raw payload inclusion. Audit trails keep raw_content_included, credential_value_included, raw_payload_included, and raw_provider_output_included false.

OCEL trace policy requires approval requirement, approval candidate, approval decision, audit plan, rollback plan, no-op boundary, failure classification, and future provider invocation event requirements. Provider invocation events are forbidden now and no provider invocation event is emitted in v0.29.6.

## Result / Failure / Rollback

Provider result boundary policy forbids raw provider output persistence and memory writes. Result persistence policy forbids raw result persistence and logging, and provider_result_persisted_now remains false. Failure classification records possible failures but does not enable retry or automatic retry. Failure handling defers retry policy and permits no-op on uncertain failure.

Rollback boundary policy requires rollback planning before side effects, keeps rollback execution disabled, forbids rollback without audit or original action refs, requires scope limits, and requires no-op fallback. Rollback plans are plans only; rollback_executed_now and rollback_plan_is_rollback_execution remain false. No-op boundary is a valid safety decision and not execution. Retry deferral keeps automatic retry disabled.
