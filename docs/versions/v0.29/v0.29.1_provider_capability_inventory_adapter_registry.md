Korean name: Provider Capability Inventory·Adapter Registry

Previous: v0.29.0 External Provider Adapter Contract

Next: v0.29.2 Mock Adapter Harness / No-Network Default

## Version Definition

v0.29.1 creates metadata-only provider capability inventory and adapter registry structures from the v0.29.0 External Provider Adapter Contract. It records provider kinds, capability declarations, registry entries, readiness states, disabled reasons, dependency boundaries, risk profiles, requirement declarations, OCEL visibility declarations, a registry gate, and an adapter registry report.

v0.29.1 is inventory/registry-only. It is not adapter implementation, provider registration, provider invocation, network access, credential handling, command execution, RPA integration, external dominion, or Schumpeter private runtime work.

## Objective

The objective is to make v0.29.2 possible by defining the inventory and registry metadata that a future mock/no-network harness can consume. Registry readiness may prepare the next mock-harness step, but it does not authorize real providers, provider SDKs, network access, credentials, or commands.

## Core Principles

- Inventory is not implementation.
- Registry is not execution.
- Adapter registry entry is not provider registration.
- Provider availability is not invocation readiness.
- Capability declaration is not permission.
- Capability status is not approval.
- Provider kind catalog is not provider SDK dependency.
- Adapter readiness is not network access.
- Credential need declaration is not credential storage.
- Network need declaration is not network access.
- RPA capability declaration is not RPA integration.
- Schumpeter future consumer reference is not Schumpeter runtime.
- Unknown capability status is not available.
- Blocked capability remains blocked until a future gate changes it.
- No-adapter and no-provider-invocation remain valid outcomes.

## Provider Kind Catalog

The provider kind catalog records provider kinds without importing provider SDKs or creating runtime bindings:

- llm_provider
- search_provider
- storage_provider
- calendar_provider
- email_provider
- workflow_provider
- rpa_provider
- internal_mock_provider
- local_file_provider
- unknown

Implementation and invocation are disabled for every provider kind. Network and credential needs are declarations for future gates only.

## Provider Capability Inventory

The provider capability inventory records capability metadata and status counts. Capability statuses are available, gated, disabled, future_track, blocked, or unknown. Unknown is never available. Capability declaration is not permission and not invocation.

## Adapter Registry

The adapter registry contains metadata-only entries. A registry entry may reference contracts, capability declarations, availability status, readiness status, disabled reason, risk profile, dependency boundary, requirement declarations, and OCEL visibility declarations. A registry entry is not provider registration.

## Capability Declarations

AdapterCapabilityDeclaration records input schema refs, output schema refs, effect boundary refs, risk level, permission scope, safety scope, credential boundary need, network boundary need, audit boundary need, rollback boundary need, and OCEL visibility need. The declaration does not grant permission and does not invoke a provider.

## Status And Disabled Reasons

AdapterAvailabilityStatus and AdapterReadinessStatus separate registry availability from implementation, invocation, and network readiness. AdapterDisabledReason records why an entry remains disabled, gated, blocked, or future-track and points to a future follow-up version when known.

## Dependency Boundary

AdapterDependencyBoundaryReport records whether a provider SDK may be required later. v0.29.1 must keep provider_sdk_required_now=false and provider_sdk_runtime_dependency_added_now=false.

## Risk Profile

AdapterRiskProfile records credential exposure, network access, private data exposure, provider side effects, command execution, data exfiltration, audit gap, rollback gap, OCEL visibility gap, RPA scope creep, and external dominion creep. Risk can block registry, and it always blocks implementation and invocation in v0.29.1.

## Requirement Declarations

v0.29.1 records future permission, safety, credential, network, and OCEL visibility requirements:

- Permission requirements do not grant permission.
- Safety requirements do not pass the safety gate.
- Credential declarations do not store, log, or create credentials.
- Network declarations do not enable or perform network calls.
- OCEL declarations list required registry and status events.

## Registry Gate

AdapterRegistryGate checks declaration completeness and confirms no provider registration, provider invocation, network call, credential storage, or command execution occurred. The gate may mark ready_for_v0_29_2=true for the mock harness, while ready_for_adapter_implementation, ready_for_provider_registration, and ready_for_provider_invocation remain false.

## Non-Goals

v0.29.1 does not implement adapters, external skills, provider registration, provider invocation, provider SDK invocation, external API calls, network calls, credential storage or logging, .env file creation, command execution expansion, shell execution surface, subprocess expansion, RPA/A360/Brity/UiPath adapters, external dominion, Schumpeter private runtime, private data use, raw provider output persistence, reference runtime dependency, reference code copy, PIG execution authority, or LLM judge as sole authority.
