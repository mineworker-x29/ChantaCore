# v0.35.5 Patch Risk & Conformance Scanner

Parent track: Controlled Patch Proposal Layer.

Release name: v0.35.5 Patch Risk & Conformance Scanner.

## Relationships

v0.35.5 consumes v0.35.4 `DiffProposalEnvelope`, `UnifiedDiffProposal`, `StructuredPatchProposal`, `PatchFileProposal`, and `PatchHunkProposal` as proposal artifact metadata.

It may also consume v0.35.3 `PatchPlan` and `PatchChangeSetGraph`, v0.35.2 `PatchContextSnapshot` and `PatchContextEvidenceBundle`, v0.35.1 `PatchIntentScopeBundle` and `PatchScopePolicy`, and the v0.35.0 `ReferencePatternDigest` as static metadata only.

These inputs are not executed, imported, installed, tested, applied, or treated as runtime authority.

## Definition

The Patch Risk & Conformance Scanner performs bounded in-memory inspection of supplied diff proposal artifacts. It may produce:

- `PatchRiskSignal`
- `PatchConformanceRule`
- `PatchConformanceFinding`
- `PatchSafetyRegressionReport`
- `PatchScopeViolationReport`
- `PatchDiffRiskSummary`
- `PatchProposalRiskReport`
- `PatchRiskScanDecision`
- validation, preview, guarantee, and readiness artifacts

Patch risk scan is not patch approval. Patch conformance scan is not patch application. Safety regression report is not remediation execution. Scope violation report is not file access. Risk signal is not runtime enforcement by itself. Review-required classification is not human approval. `acceptable_for_review` is not `approved_for_apply`.

## Scanner Policy

`PatchRiskScannerPolicy` may allow `acceptable_for_review` as a human-review input classification. It must keep false:

- `allow_approved_for_apply`
- `allow_patch_apply`
- `allow_workspace_write`
- `allow_code_edit`
- `allow_test_execution`
- `allow_shell`
- `allow_dependency_install`
- `allow_reference_execution`
- `allow_provider_invocation`
- `allow_network_access`
- `allow_credential_access`
- `allow_secret_read`

The scanner policy is not apply permission.

## Hard Prohibitions

v0.35.5 may scan bounded diff proposal artifact strings and may produce risk/conformance reports.

v0.35.5 does not approve patch application. It does not apply patches. It does not call `apply_patch`. It does not call `git apply`. It does not write or edit files. It does not execute tests. It does not execute shell, subprocess, commands, package scripts, dependency installs, OpenCode, Hermes, OpenClaw, reference imports, provider SDKs, network calls, credential reads, secret reads, persistent trace writes, UI runtime, external control, or authority grants.

## Acceptance Criteria

- Risk, conformance, safety regression, and scope violation scanner models exist.
- Scanner policy blocks approved-for-apply, apply, write, edit, test, shell, install, reference, provider, network, credential, and secret access.
- Scanner detects unsafe readiness flags, apply/write/edit/shell/test/install/reference/provider/network/credential patterns in bounded proposal text.
- Clean metadata-only proposal may become `acceptable_for_review` but never `approved_for_apply`.
- `PatchProposalRiskReport` is review-input metadata, not apply approval.
- All unsafe readiness flags remain false.

## Withdrawal Conditions

Withdraw this release boundary if any runtime path is introduced for patch approval, patch application, workspace write, code edit, `apply_patch`, `git apply`, shell/subprocess/command execution, test execution, dependency install, reference execution/import/install, provider/network/credential/secret access, persistent trace writes, UI runtime, external control, or authority grant.

## Handoff

v0.35.5 is ready only for v0.35.6 Human Review Packet & Approval Gate Metadata and v0.35.7 Patch Proposal OCEL Trace Packet design-stage handoff. `ready_for_execution` remains false.
