# v0.35.1 Patch Intent & Scope Policy

Parent track: Controlled Patch Proposal Layer.

Release name: v0.35.1 Patch Intent & Scope Policy.

## Relationship to v0.35.0

v0.35.1 builds on v0.35.0 Controlled Patch Proposal Boundary Foundation and its ReferencePatternDigest. The digest at `docs/versions/v0.35/v0.35.0_reference_pattern_digest.md` is consumed as design metadata only. v0.35.1 does not execute, import, install, or test OpenCode, Hermes, OpenClaw, or any reference corpus.

v0.35.0 defined the boundary between patch proposal and patch apply. v0.35.1 defines why a future proposal might be requested and what scope metadata may constrain future read-only context collection.

## Definition

Patch Intent & Scope Policy is a metadata layer made of:

- `PatchIntentEnvelope`: why a patch proposal is requested, with rationale, expected outcome, source refs, non-goal summary, and reference-pattern use notes.
- `PatchScopePolicy`: which roots, path patterns, and target kinds are allowed or blocked as future metadata.
- `PatchTargetSelector`: a future target reference selector. It is not file access and not file write.
- `PatchAllowedTarget` and `PatchBlockedTarget`: bounded target metadata outcomes.
- `PatchNonGoalRegister`: explicit constraints that prevent accidental scope expansion.
- validation reports, intent-scope bundles, run previews, no-apply guarantees, and readiness reports.

Patch intent is not patch proposal. Patch scope is not file access. A target selector is not file write. A non-goal register is not runtime enforcement by itself. Scope validation is not write permission. `PatchIntentScopeBundle` is not a patch plan, change-set graph, diff proposal, or patch proposal.

## ReferencePatternDigest Consumption Policy

The v0.35.0 ReferencePatternDigest may provide pattern IDs, summaries, adaptation notes, rejected patterns, future-track items, confidence levels, and evidence refs. v0.35.1 maps those metadata items into `PatchReferencePatternUse` records.

Reference-informed uses must distinguish:

- observed pattern,
- ChantaCore adaptation,
- rejected reference pattern,
- future-track item,
- evidence source from digest or relative path.

Rejected reference patterns remain rejected. Future-track reference patterns remain future-track. No implementation body is copied from references.

## Taxonomies

Patch intent kinds include missing contract models, validation rules, safety boundary tightening, test coverage, documentation, no-behavior refactor, reference alignment, inconsistency fixes, release consolidation, future-track preparation, no-op, and unknown.

Patch scope kinds include source module scope, test module scope, documentation scope, version document scope, agent runtime scope, reference digest scope, read-only reference scope, blocked reference execution scope, blocked secret scope, blocked external scope, no-op scope, and unknown.

Patch target kinds include source files, test files, documentation files, version documents, package metadata, reference digest documents, reference corpus paths, secret-like files, credential-like files, binary files, external paths, and unknown. Secret, credential, binary, and external targets are blocked.

Decision kinds allow metadata only, design-stage handoff, denial, block, no-op, review, future gate, and unknown. Allow decisions do not permit write or apply.

Risk kinds cover vague intent, overbroad scope, unsafe reference patterns, patch apply, workspace write, code edit, test execution, dependency install, shell execution, credential exposure, secret exposure, provider/network opening, reference execution, autonomous loop, authority grant, and unknown.

## Flow

`v0.35.0 boundary + ReferencePatternDigest + user/model intent metadata -> PatchIntentEnvelope -> PatchScopePolicy -> TargetSelectors -> NonGoalRegister -> ValidationReports -> PatchIntentScopeBundle -> V0351ReadinessReport -> v0.35.2 Read-only Patch Context Collector`

The v0.35.3 Reference-informed Patch Plan handoff is design-stage only. v0.35.1 does not create a patch plan.

## Hard Prohibitions

v0.35.1 does not collect patch context. It does not create patch plans. It does not generate diff proposals. It does not generate patch proposals. It does not apply patches. It does not write or edit files. It does not call apply_patch. It does not run git apply. It does not run tests. It does not execute shell, subprocess, or commands. It does not install dependencies. It does not execute or import OpenCode, Hermes, or OpenClaw. It does not use provider SDKs, network libraries, credentials, secrets, persistent trace writes, UI runtime, external control, or authority grants.

## Acceptance Criteria

- Patch intent and scope taxonomies exist.
- Intent, source, reference use, scope policy, target selector, allowed target, blocked target, and non-goal register models exist.
- Validation/report/bundle/preview/guarantee/readiness artifacts exist.
- ReferencePatternDigest can be consumed as metadata.
- Unknown or unsafe intent/scope states require review, block, or future gate.
- All write/apply/proposal/context/plan/diff/execution flags remain false.

## Withdrawal Conditions

This release boundary must be withdrawn if any runtime path is introduced for patch context collection, patch planning, diff proposal generation, patch proposal generation, patch application, workspace write, code edit, apply_patch, git apply, shell, subprocess, command execution, test execution, dependency installation, reference execution/import/install, provider/network/credential/secret access, persistent trace write, UI runtime, external control, or authority grant.

## Handoff

v0.35.1 is ready only for v0.35.2 Read-only Patch Context & Reference Corpus Collector and v0.35.3 Reference-informed Patch Plan design-stage handoff. `ready_for_execution` remains false.
