# v0.35.3 Reference-informed Patch Plan & Change Set Graph

Parent track: Controlled Patch Proposal Layer.

Release name: v0.35.3 Reference-informed Patch Plan & Change Set Graph.

## Relationships

v0.35.3 consumes v0.35.1 Patch Intent & Scope Policy metadata and v0.35.2 Read-only Patch Context Collector metadata. It may also consume the v0.35.0 ReferencePatternDigest as design metadata.

These inputs are metadata only. v0.35.3 does not read files directly, execute reference code, import reference modules, install dependencies, run tests, or use provider/network/credential paths.

## Definition

`PatchPlan` is a non-mutating planning artifact. It may contain a title, summary, source refs, gaps, reference-informed pattern uses, risks, and a `PatchChangeSetGraph`.

`PatchChangeSetGraph` is non-mutating graph metadata. It may contain:

- `PatchChangeNode`
- `PatchDependencyEdge`
- `PatchTargetFilePlan`
- `PatchTestPlan`
- `PatchDocumentationPlan`

A patch plan is not a diff proposal. A patch plan is not a patch proposal. A change-set graph is not code edit. A target-file plan is not file write. A test plan is not test execution. A documentation plan is not documentation write. Reference-informed planning is not reference code import.

## Taxonomies

Patch plan modes include intent/scope only, context snapshot informed, reference digest informed, combined context and reference informed, blocked, no-op, and unknown.

Source kinds include v0.35.2 context snapshot, v0.35.2 evidence bundle, v0.35.1 intent/scope bundle, v0.35.1 intent envelope, v0.35.1 scope policy, v0.35.0 ReferencePatternDigest, reference pattern adaptation, user request, model-backed step output, test fixture, and unknown.

Patch plan statuses include draft, plan created, plan validated, plan validated with gaps, change-set graph created, blocked, review required, future gated, no-op, and unknown.

Readiness levels include plan contract ready, change-set graph ready, reference-informed plan ready, design handoff for v0.35.4, design handoff for v0.35.5, blocked, future track, and not ready.

Change kinds include contract model additions/extensions, validation helpers, stronger validation rules, readiness reports, boundary policy, tests, documentation, internal refactor metadata, reference-pattern alignment, trace artifact metadata, CLI surface metadata, no-op change, and unknown.

Dependency kinds describe relationships only. They do not execute order.

## Planning Policy

`PatchPlanningPolicy` may require context snapshot, intent/scope bundle, non-goal register, and reference-informed patterns. It may allow future diff handoff metadata.

It must keep false:

- `allow_diff_text`
- `allow_patch_hunks`
- `allow_patch_apply`
- `allow_workspace_write`
- `allow_code_edit`
- `allow_test_execution`
- `allow_dependency_install`
- `allow_shell`

Planning policy is not diff, hunk, apply, write, test, install, or shell permission.

## Flow

`PatchIntentScopeBundle + PatchContextSnapshot + EvidenceBundle + ReferencePatternDigest -> PatchPlanningInput -> ReferenceInformedPatternUse -> ChangeNodes -> DependencyEdges -> ChangeSetGraph -> PatchPlan -> V0353ReadinessReport -> v0.35.4 Diff Proposal Envelope`

v0.35.5 Patch Risk & Conformance Scanner is also a design-stage handoff. v0.35.3 does not perform the risk scan.

## Hard Prohibitions

v0.35.3 does not generate diff proposals. It does not generate patch proposals. It does not generate patch hunks. It does not apply patches. It does not write or edit files. It does not call apply_patch. It does not run git apply. It does not execute tests. It does not execute shell, subprocess, commands, package scripts, dependency installs, OpenCode, Hermes, OpenClaw, reference imports, provider SDKs, network calls, credential reads, secret reads, persistent trace writes, UI runtime, external control, or authority grants.

## Acceptance Criteria

- Patch plan and change-set graph models exist.
- Change nodes, dependency edges, target-file plans, test plans, documentation plans, validation reports, plan reports, previews, guarantees, and readiness reports exist.
- ReferencePatternDigest and context snapshots are consumed only as supplied metadata.
- Rejected reference patterns remain rejected.
- Future-track reference patterns remain future-track.
- Diff/proposal/hunk/apply/write/edit/execution readiness remains false.

## Withdrawal Conditions

Withdraw this release boundary if any runtime path is introduced for diff generation, patch proposal generation, patch hunk generation, patch application, workspace write, code edit, apply_patch, git apply, shell/subprocess/command execution, test execution, dependency install, reference execution/import/install, provider/network/credential/secret access, persistent trace writes, UI runtime, external control, or authority grant.

## Handoff

v0.35.3 is ready only for v0.35.4 Diff Proposal Envelope and v0.35.5 Patch Risk & Conformance Scanner design-stage handoff. `ready_for_execution` remains false.
