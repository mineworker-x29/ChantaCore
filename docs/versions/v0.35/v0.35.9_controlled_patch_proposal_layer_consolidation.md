# v0.35.9 Controlled Patch Proposal Layer Consolidation

Foundation release name: Controlled Patch Proposal Layer v1.

Parent track: Controlled Patch Proposal Layer.

v0.35.9 is consolidation only. Controlled Patch Proposal Layer v1 is a bounded foundation, not production runtime, not patch application, and not a workspace write surface.

## Relationship to Prior Gates

- v0.34.9 established the Controlled Model Invocation Boundary v1, including provider, network, credential, and execution prohibitions inherited by this track.
- v0.35.0 established the controlled patch proposal boundary and ReferencePatternDigest for read-only OpenCode/Hermes pattern digestion.
- v0.35.1 established Patch Intent & Scope Policy metadata.
- v0.35.2 established read-only Patch Context collection and evidence bundles.
- v0.35.3 established reference-informed Patch Plans and PatchChangeSetGraph metadata.
- v0.35.4 established DiffProposalEnvelope, UnifiedDiffProposal, StructuredPatchProposal, file proposal, and hunk proposal artifacts.
- v0.35.5 established Patch Risk & Conformance Scanner reports, safety regression scans, and scope violation scans.
- v0.35.6 established Human Review Packet, approval gate metadata, reviewer decision placeholders, and review decision records.
- v0.35.7 established returned/in-memory Patch Proposal OCEL Trace Packets.
- v0.35.8 established the bounded CLI Patch Proposal Surface for safe previews and denied unsafe commands.

## What v0.35.x Accomplished

v0.35.x enables patch proposal artifacts, not patch application. It enables:

- controlled patch proposal boundary definition;
- OpenCode/Hermes read-only ReferencePatternDigest outcomes;
- patch intent and scope metadata;
- read-only patch context snapshots;
- reference-informed patch plan and change-set graph metadata;
- diff proposal artifacts, including bounded unified diff proposal text and structured patch proposal metadata;
- patch risk, conformance, safety regression, and scope violation reports;
- human review packet metadata;
- approval gate metadata without apply permission;
- reviewer decision placeholders and review decision metadata;
- returned/in-memory patch proposal trace packets, not persistent trace writes;
- CLI patch proposal preview surface, not shell execution;
- Digestion-first / Dominion-fallback metadata;
- external-agent-control-pattern awareness as blocked/future-gated metadata.

## What v0.35.x Explicitly Did Not Accomplish

v0.35.x does not enable workspace write, code edit, patch application, apply_patch, git apply, file creation/deletion/rename/chmod/chown runtime behavior, test execution, shell execution, subprocess execution, command execution, dependency install, OpenCode/Hermes/OpenClaw execution, reference code import, reference dependency install, reference test execution, Claude Code invocation, Codex CLI invocation, external coding agent orchestration, infinite agent loops, Dominion runtime, direct provider SDK invocation, direct network access, credential/env/secret reads, autonomous general agent runtime, general tool execution, unquarantined action execution, persistent trace files/database/log writes, external trace sinks, UI runtime, external control, authority grant, or D4-D9 authority grant.

Diff proposal artifacts are not git apply or apply_patch. Human review metadata is not apply permission. CLI preview is not shell. Trace packet readiness is not persistent trace readiness. Digestion/Dominion awareness is not Dominion runtime readiness.

## Digestion / Dominion Consolidation

ChantaCore uses Digestion first: read, observe, summarize, model, adapt, and internalize static patterns safely. Dominion remains future-gated unless digestion is insufficient and later gates add strict boundaries, human approval, audit trace, and sandboxing.

v0.35.9 consolidates:

- digestion_first_policy_applied;
- external_agent_control_pattern_observed;
- dominion_like_loop_detected;
- dominion_escalation_rejected;
- dominion_escalation_future_gated;
- external_agent_execution_blocked;
- infinite_agent_loop_blocked;
- reference_harness_execution_blocked.

OpenCode, Hermes, OpenClaw, Claude Code, Codex CLI, and external coding agents are not executed or invoked. Codex-to-Claude-Code-like loops and OpenCode/Hermes execution loops are recorded only as blocked/future-gated metadata.

## Recommended v0.36 Handoff

Recommended v0.36 track: Human-approved Patch Apply Sandbox.

Recommended first step: dry-run apply simulation. v0.36 should not immediately open unrestricted write/apply authority. Patch application should require human approval, scope gate, sandbox policy, no-secret/no-shell/no-network checks, rollback plan, audit trace, explicit apply boundary, and staged dry-run simulation before any real write.

## Hard Prohibitions

v0.35.9 does not add patch application behavior, runtime file write, runtime code edit, apply_patch, git apply, shell execution, subprocess execution, command execution, test execution, dependency install, provider SDK imports, network access, credential/secret reads, reference execution/import, external agent invocation, Claude Code invocation, Codex CLI invocation, Dominion runtime, infinite loops, registry/memory mutation, persistent trace writes, OCEL/JSONL/log/database writes, UI runtime, external control, authority grant, or production certification.

## Gate Inheritance

v0.35.9 inherits v0.30, v0.31, v0.32, v0.33, v0.34, and v0.35 gates. Any attempt to treat proposal artifacts as apply permission, CLI previews as shell execution, trace packets as persistence, or Digestion/Dominion awareness as runtime authority violates this consolidation.
