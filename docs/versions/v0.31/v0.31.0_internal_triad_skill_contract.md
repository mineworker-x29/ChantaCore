# v0.31.0 Internal Triad Skill Contract

Release name: v0.31.0 Internal Triad Skill Contract

Parent track: Internal Triad Skill Foundation

## Relationship To v0.30.9

v0.30.9 closed the External Dominion Control Plane Foundation v1. It consolidated external target representation, capability observations, digestion candidates, dominion authority, delegation dry-run candidates, approval/audit/rollback boundaries, certification matrices, limited preview gate reviews, and the v0.31 handoff packet.

v0.31.0 does not change v0.30 runtime readiness flags. It consumes the v0.30.9 handoff language as contract input and begins the internalization track by defining the common model layer for ChantaCore's Internal Triad:

- Observation
- Digestion
- Dominion

The v0.30 gates remain inherited. D0-D3 remain the only grantable dominion levels. D4-D9 remain future-track. External execution readiness remains false.

## Internal Triad

The Internal Triad is the internal skill grammar ChantaCore uses to reason about future internal skills without activating those skills.

Observation means structure and interpret input artifacts. ObservationSkillContract is not external scan and does not fetch source_ref.

Digestion means produce internalization candidates only. DigestionSkillContract is not active internalization and does not mutate registries, install tools, install missions, activate policies, or activate memory writers.

Dominion means produce governance and control candidates only. DominionSkillContract is not external runtime control and does not invoke providers, networks, credentials, commands, browser runtimes, RPA runtimes, gateways, packet sends, rollback, or retry.

## Contract Versus Runtime Skill

Internal triad skill contract is not runtime execution. A TriadSkillContract describes allowed input and output artifact kinds, required evidence and OCEL planning references, readiness status, boundary policy, withdrawal conditions, and no-execution flags.

A runtime skill would execute behavior. v0.31.0 does not define runtime skills, does not activate internal skills, and does not enable read-only tools.

## Input Envelope Versus Execution Request

TriadSkillInputEnvelope is an input artifact descriptor. It records source artifact refs, target refs, capability refs, candidate refs, report refs, evidence refs, task summary, and prohibited runtime actions.

The input envelope is not an execution request. It does not authorize external scanning, source_ref fetch, read-only local tool execution, provider invocation, network calls, command execution, browser automation, RPA control, gateway send, packet send, rollback, retry, registry mutation, or memory mutation.

## Result Envelope Versus Active Artifact Registration

TriadSkillResultEnvelope is a result descriptor. It records produced artifact refs, candidate refs, decision refs, evidence refs, blocked reasons, optional no-op reason, and next-stage design readiness.

The result envelope is not active artifact registration. It does not register artifacts into an active runtime, mutate the skill registry, write memory, promote memory, or mark a runtime task complete.

## Boundary Policy

TriadSkillBoundaryPolicy is a contract boundary, not permission.

It records:

- allowed input artifact kinds
- allowed output artifact kinds
- prohibited runtime actions
- prohibited external surfaces
- max dominion level
- evidence ref and OCEL trace plan requirements
- no-execution, no-external-contact, no-registry-mutation, and no-memory-mutation guarantees

The required prohibited runtime actions include external execution, internal tool execution, provider invocation, network, credential, command, browser, RPA, gateway, packet send, rollback, retry, active registry mutation, and active memory mutation.

The max dominion level must be None or no higher than D3_SIMULATE.

## No-Execution Guarantee

TriadSkillNoExecutionGuarantee records contract metadata that all no_* runtime flags remain true in v0.31.0:

- no external execution
- no internal tool execution
- no network access
- no credential access
- no command execution
- no provider invocation
- no browser automation
- no RPA control
- no gateway control
- no packet send
- no registry mutation
- no memory mutation
- no rollback execution
- no retry execution

This guarantee is contract metadata, not runtime enforcement by itself.

## Contract Set

TriadSkillContractSet groups the Observation, Digestion, and Dominion contracts with their no-execution guarantees and shared boundary policy refs.

ready_for_v0311_observation_skill_foundation may be true only as next design stage handoff. ready_for_execution remains false. ready_for_skill_activation remains false.

## Readiness Report

V0310ReadinessReport summarizes the v0.31.0 contract set. It may say the repository is ready for v0.31.1 Observation Skill Foundation design work.

It must not say v0.31.0 is execution-ready. ready_for_execution remains false. ready_for_skill_activation remains false.

## Persistence And Canonical-State Policy

v0.31.0 introduces no persistence layer.

There is no canonical runtime state, no database table, no registry write, no memory write, no file-backed artifact registration, and no OCEL event persistence in this release. All objects are immutable dataclass values (`frozen=True`) used as in-memory contract descriptors.

Any future release that persists triad contract artifacts must define a separate persistence policy, evidence provenance policy, lifecycle policy, and revoke/forget/migration path before writing durable state.

Future report support may read these contract descriptors as metadata, but it must not infer runtime readiness, external scan readiness, skill activation readiness, or memory write readiness from the presence of a triad contract.


The unsafe interpretation must be rejected.

## Future Work

The next bounded step is v0.31.1 Observation Skill Foundation.

That future release may define richer observation skill foundation artifacts, but it must still preserve:

- no external scan unless a later gate explicitly opens and tests that behavior;
- no source_ref fetch unless a later source access policy is defined;
- no read-only tool execution unless a later tool execution boundary is defined;
- no active skill activation;
- no v0.30 runtime readiness changes.

Later v0.31.x releases may define digestion and dominion foundations. Those releases must continue to distinguish candidate generation from installation, and governance/control candidates from runtime control.

## Acceptance Criteria

v0.31.0 is acceptable when:

- TriadSkillKind, TriadSkillStatus, and TriadSkillExecutionMode exist
- TriadSkillBoundaryPolicy validates no-execution boundaries
- TriadSkillContract validates no execution and no skill activation
- ObservationSkillContract prohibits external scan and source_ref fetch
- DigestionSkillContract prohibits active skill/tool/mission/policy/memory mutation
- DominionSkillContract prohibits external runtime control and keeps max dominion level <= D3
- TriadSkillInputEnvelope is not an execution request
- TriadSkillResultEnvelope is not active artifact registration
- TriadSkillNoExecutionGuarantee keeps all no_* fields true
- TriadSkillContractSet contains observation/digestion/dominion contracts
- V0310ReadinessReport remains not runtime-ready
- focused tests pass

## Withdrawal Conditions

The v0.31.0 judgment must be withdrawn if:

- a v0.31.0 helper executes, fetches, invokes, registers, writes, emits OCEL events, sends packets, rolls back, retries, or mutates state
- ObservationSkillContract becomes external scan or source_ref fetch
- DigestionSkillContract becomes active internalization
- DominionSkillContract becomes external runtime control
- ready_for_execution becomes true
- ready_for_skill_activation becomes true
- v0.31.0 enables read-only tools
- v0.31.0 activates internal skills
- D4-D9 become grantable
- v0.30 runtime readiness flags are changed by this track
