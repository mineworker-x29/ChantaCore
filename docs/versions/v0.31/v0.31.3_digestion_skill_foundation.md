# v0.31.3 Digestion Skill Foundation

## Version Name And Scope

Release name: `v0.31.3 Digestion Skill Foundation`

Parent track: `v0.31.x Internal Triad Skill Foundation`

Immediate parent: `v0.31.2 Observation Report / Capability Map`

Inherited foundations:

- `v0.31.1 Observation Skill Foundation`
- `v0.31.0 Internal Triad Skill Contract`
- `v0.30.9 External Dominion Control Plane Foundation Consolidation`

This release defines the second Internal Triad skill foundation: Digestion.

The scope is contract/control-plane only. It defines how observation reports and
capability maps can be interpreted as route signals for later internalization
design. It does not perform internalization.

## Restore Goal

A maintainer restoring this version should be able to reconstruct:

1. the Digestion source/focus/signal/route/posture/blocker taxonomies;
2. the Digestion input, source ref, pattern signal, finding, blocker, route
   decision, output, no-op, run preview, foundation report, and readiness report
   models;
3. the pure helper functions used to build those models;
4. the interaction with `v0.31.2` capability-map classifications;
5. the no-runtime, no-internalization, no-registry, no-memory, no-candidate, and
   no-dominion boundaries;
6. the focused tests that prove the boundary.

Restore is invalid if route signals become active candidates, route decisions
become implementations, or readiness flags become runtime readiness.

## Definition Of Digestion Skill

Digestion Skill means the internal triad layer that interprets already-available
observation artifacts and identifies whether an observed pattern is suitable for
later design-stage internalization review.

In `v0.31.3`, Digestion Skill can:

- receive observation report and capability-map refs;
- receive capability entry refs;
- classify focus patterns;
- infer route signals;
- record blockers;
- produce route decisions;
- produce a non-runtime output artifact.

In `v0.31.3`, Digestion Skill cannot:

- create internal skill candidates;
- create tool contract candidates;
- create mission candidates;
- create policy candidates;
- create memory schema candidates;
- create internalization plans;
- mutate registries;
- write memory;
- execute tools;
- scan external systems.

## Digestion Foundation Versus Active Internalization

Digestion Skill Foundation is not active internalization.

Active internalization would create or install internal skill/tool/mission/policy
or memory-schema artifacts. This release only records route signals and
diagnostic blockers for later review.

Searchable invariant:

- `ready_for_internalization=False`
- `ready_for_registry_mutation=False`
- `ready_for_execution=False`
- `ready_for_skill_activation=False`

## DigestiblePatternSignal Versus Internal Candidate

`DigestiblePatternSignal` is a signal object. It is not an internal skill
candidate, not a tool contract candidate, not a mission candidate, not a policy
candidate, not a memory schema candidate, and not an internalization plan.

The signal may say that internalization is possible, but that statement only
means a later release may review the pattern. It is not generated code,
registration, installation, or activation.

## internalization_candidate_signal Versus Actual Internalization

`internalization_candidate_signal` is a route value. It is not an
implementation. It does not create an object reserved for `v0.31.4 Digestion
Candidate / Internalization Plan`.

`ready_for_v0314_digestion_candidate_internalization_plan=True` means next design
stage only. It does not mean execution readiness or registry mutation readiness.

## dominion_required_signal Versus DominionTarget

`dominion_required_signal` is a route value. It does not create a dominion
target and does not create a dominion decision.

It only means a later dominion foundation may review the boundary. The inherited
max grantable dominion level remains no higher than `D3_SIMULATE`, and this
release grants no new dominion authority.

## Safe Defaults

Safe defaults:

- all runtime/internalization/registry readiness flags stay false;
- helpers build model artifacts only;
- source refs are never fetched;
- route signals never instantiate candidates;
- blockers never remediate;
- readiness is limited to later design-stage handoff;
