# v0.38.2 Read-only Sandbox Source Context Snapshot

Parent track: Bounded Repair Proposal Loop.

v0.38.2 builds on v0.38.1 Repair Proposal Evidence Contract and v0.38.0 Bounded Repair Proposal Boundary Foundation. It creates read-only sandbox source context metadata and bounded read-only sandbox source snapshot helpers. It may perform bounded text reads only for explicit relative paths under a validated sandbox root.

## Definitions

Read-only sandbox source context: an in-memory snapshot of explicitly requested source files under a validated sandbox root. It is bounded, redacted, and non-mutating.

Validated sandbox root: a supplied directory that exists, is a directory, is declared as sandbox/work area metadata, is not the live workspace, is not reference corpus, and does not contain disallowed markers such as `.git`.

Explicit relative path candidate: a path string provided by the caller that is not absolute, has no parent traversal, and is not trusted until validated against the sandbox root.

Bounded source file snapshot: metadata containing a digest, bounded preview, redaction marker, truncation marker, file kind, and path validation reference. It is not edit permission.

Bounded source excerpt: a bounded and redacted excerpt from a validated source file snapshot. It is not a code hunk and not edit permission.

Symbol context hint: non-executing text heuristic metadata such as a detected `def` or `class` line. It does not import or execute source.

Source context sufficiency assessment: metadata deciding whether the bounded context is sufficient for future scope planning or future patch metadata input.

## Explicit Distinctions

- Sandbox source read is not live workspace read.
- Bounded source read is not unbounded source read.
- Validated sandbox read is not general file read.
- Source excerpt is not edit permission.
- Source context snapshot is not repair proposal generation.
- Symbol hint is not source execution.
- Future patch metadata input is not proposed diff or code hunk generation.
- Do-nothing alternative remains valid.
- Human review remains mandatory before any future apply.

## Allowed Read-only Rules

All conditions must hold before a bounded read:

- v0.38.1 evidence metadata supports future source context input or human review exploration.
- `sandbox_root` is supplied.
- `sandbox_root` exists and is a directory.
- `sandbox_root` is validated as sandbox/work area metadata, not live workspace.
- candidates are explicit relative paths.
- candidates normalize under `sandbox_root`.
- no absolute path.
- no parent traversal.
- no symlink traversal.
- no reference corpus path.
- no `.git`, credential, secret, env, token, key, private key, or certificate path.
- no binary or unsupported file.
- file size is within `max_file_bytes`.
- output is bounded by snapshot and excerpt limits.
- output is redacted for secret-like patterns.
- output is returned as in-memory metadata only.

## Denied Rules

v0.38.2 denies implicit full project scans, unbounded recursive directory walks, absolute path reads, parent traversal reads, symlink reads, reference corpus reads, live workspace reads, credential or secret reads, binary reads, writes, edits, patch application, proposal generation, diff generation, code hunk generation, patch envelope generation, repair execution, tests, subprocess, shell, dependency install, network, model providers, external agents, and Dominion runtime.

Reference corpus policy remains unchanged: `references/OpenCode`, `references/Hermes`, and `references/OpenClaw` are not read as repair source context, executed, imported, installed, tested, or used as adapters. Reference patterns may only remain digested, adapted, rejected, or future-gated metadata.

## Acceptance Criteria

- Valid sandbox root can be accepted.
- Missing, live workspace, reference corpus, or disallowed-marker roots are invalid.
- Explicit relative path candidates can be validated.
- Absolute, traversal, symlink, outside-root, reference, `.git`, secret-like, binary, oversized, directory, missing, and unsupported paths are denied.
- Bounded read-only snapshots are created only after root and path validation.
- Secret-like content is redacted.
- Excerpts are bounded and redacted.
- Symbol hints are generated from text heuristics without import or execution.
- Context assessment conservatively gates future scope planning and patch metadata inputs.
- No-mutation guarantee confirms no write, edit, apply, repair, or generation.

## Withdrawal Conditions

Withdraw v0.38.2 readiness if live workspace read, outside-root read, reference source read, secret read, unbounded read, write, proposal generation, diff generation, hunk generation, patch envelope generation, file edit, patch application, test execution, subprocess, shell, dependency install, network, model invocation, external agent execution, Dominion runtime, production certification, or `ready_for_execution=True` appears in implemented v0.38.2 runtime/helper behavior.

## Handoff

v0.38.2 hands off only design-stage metadata to v0.38.3 and v0.38.4. v0.38.3 may later create repair scope planning and change intent metadata. v0.38.4 may later create proposed diff and code hunk metadata under its own gate. v0.38.2 does not implement those stages.
