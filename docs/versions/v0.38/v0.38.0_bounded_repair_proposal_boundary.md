## Definitions

A bounded repair proposal is a future metadata artifact that may describe evidence, source context, repair scope, change intent, proposed diff metadata, proposed code hunk metadata, proposed patch envelope metadata, safety validation, human review packet, and do-nothing comparison. In v0.38.0 this is only a boundary.

A repair evidence contract is a future contract for evidence inputs. Its boundary is not evidence sufficiency.

A read-only sandbox source context is a future bounded source snapshot. Its boundary is not source read execution.

A repair scope plan and change intent model are future planning artifacts. Their boundary is not edit permission or code modification.

Proposed diff metadata, proposed code hunk metadata, and proposed patch envelope metadata are future metadata surfaces. Their boundaries are not diff generation, code hunk generation, patch envelope generation, code edit, or patch application.

A human review packet boundary defines that review is required. It is not human approval. An approval request boundary is not approval.

A future sandbox repair apply input boundary is not sandbox repair apply.

The do-nothing alternative remains a valid comparison and must remain present.

## Explicit Non-Goals

v0.38.0 does not generate repair proposals. It does not generate proposed diffs. It does not generate code hunks. It does not generate patch envelopes. It does not read source files. It does not read sandbox source. It does not edit files. It does not write files. It does not apply patches. It does not call apply_patch. It does not call git apply. It does not execute repair.

v0.38.0 does not execute tests, run subprocess, execute shell/subprocess/commands, install dependencies, access network, invoke model providers, invoke Codex CLI, invoke Claude Code, invoke external agents, or implement Dominion runtime.

v0.38.0 does not certify production readiness.

## Acceptance Criteria

Acceptance requires boundary models, policies, surfaces, permission/gate/risk models, no-execution guarantee, roadmap, readiness report, docs, and tests. All helpers must remain pure. No helper may read or write files, read source, generate diffs/hunks/patch envelopes, apply patches, run tests, invoke subprocess, shell out, install dependencies, use network, invoke providers, invoke external agents, implement Dominion runtime, read credentials/secrets/env, persist traces, or set unsafe readiness flags true.

## Withdrawal Conditions

Withdraw v0.38.0 if source read, repair proposal generation, proposed diff generation, proposed code hunk generation, proposed patch envelope generation, file edit/write, patch application, apply_patch, git apply, repair execution, automatic repair, test execution, subprocess/shell/command execution, dependency install, network access, model provider invocation, external agent execution, Dominion runtime, production certification, or ready_for_execution is introduced.
