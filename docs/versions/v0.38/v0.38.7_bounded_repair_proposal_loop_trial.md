# v0.38.7 Bounded Repair Proposal Loop Trial

Parent track: Bounded Repair Proposal Loop.

v0.38.7 builds on v0.38.6 Human Review Packet & Approval Request Contract, v0.38.5 Repair Proposal Safety & Static Validation, v0.38.4 Proposed Diff / Code Hunk Metadata Generation, v0.38.3 Repair Scope Planner & Change Intent Model, v0.38.2 Read-only Sandbox Source Context Snapshot, v0.38.1 Repair Proposal Evidence Contract, and v0.38.0 Bounded Repair Proposal Boundary Foundation.

## Definition

The bounded repair proposal loop trial creates one-shot loop trial metadata only. It assembles already-created evidence, source context snapshot, scope plan, proposed patch envelope, safety report, human review packet, and approval request contract metadata into a bounded in-memory loop packet.

The one-shot loop packet is not runtime execution. A loop step record is not step execution. A loop artifact bundle is not artifact generation. A stop condition is not a repair result. Future CLI repair proposal surface input is not CLI execution. Future v0.39 handoff metadata is not sandbox apply.

## One-Shot Policy

`max_cycle_count` is exactly `1`. `retry_count` is exactly `0`. Human handoff is mandatory. Stop reason is mandatory. Approval remains absent. Do-nothing comparison remains mandatory and represented.

The trial sequence is:

`v0.38.6 HumanReviewPacket + v0.38.5 SafetyReport + v0.38.4 ProposedPatchEnvelope + v0.38.3 ScopePlan + v0.38.2 SourceContext + v0.38.1 EvidenceBundle -> LoopInput -> ArtifactBundle -> StepRecords -> BoundaryAudit -> StopCondition -> DoNothingComparison -> LoopDecision -> LoopPacket -> V0387ReadinessReport -> v0.38.8 CLI Repair Proposal Surface`

## Models

`RepairProposalLoopPolicy` requires one cycle, zero retries, required prior artifacts, do-nothing comparison, stop reason, and human handoff. It allows only one-shot packet, artifact bundle, step records, boundary audit, stop condition, do-nothing comparison, and future handoff metadata.

`RepairProposalLoopInput` is a loop packet request, not a loop execution request.

`RepairProposalLoopArtifactBundle` records present, missing, and invalid existing artifacts. It does not create missing artifacts.

`RepairProposalLoopStepRecord` records metadata steps only. It does not execute steps.

`RepairProposalLoopBoundaryAudit` confirms no approval, no apply permission, no source read, no new patch generation, no file write, no patch apply, no repair execution, no test execution, no model invocation, no external agent, no Dominion runtime, and no production certification.

`RepairProposalLoopStopCondition` records why the packet stops. It requires cycle count <= 1, retry count 0, human handoff, approval absent, apply not allowed, repair not executed, and tests not run.

`RepairProposalLoopDoNothingComparison` keeps do-nothing valid and may prefer or require it when artifacts are missing or risks block handoff.

`RepairProposalLoopDecision` may prepare future CLI surface, consolidation, and v0.39 handoff metadata. It never permits autonomous loops, retries, approval capture, approval grant, apply permission, source read, new patch generation, file write, patch application, repair execution, tests, model provider invocation, external agents, or production certification.

`RepairProposalLoopPacket` is bounded in-memory metadata. It is not execution readiness.

`RepairProposalLoopValidationReport`, `RepairProposalLoopNoExecutionGuarantee`, and `V0387ReadinessReport` confirm the one-shot/no-execution posture.

## Digestion And Reference Policy

The digestion-first policy is recorded as metadata. Dominion runtime, external agent execution, infinite agent loops, recursive self-invocation, automatic repair loops, repair execution, model provider invocation, and tool execution remain blocked.

Reference corpus patterns may be represented only as digested, adapted, rejected, or future-gated metadata. v0.38.7 does not execute, import, install, test, run, or read reference corpus code as repair source context.

## Acceptance Criteria

The release is acceptable when the loop policy is one-shot only, the loop input is a packet request, artifact completeness is represented, step records are metadata only, boundary audit confirms no runtime actions, stop condition and human handoff are present, do-nothing comparison is represented, loop decision preserves no approval/apply/execution, loop packet is bounded and not execution, and focused tests pass.

## Withdrawal Conditions

Withdraw v0.38.7 readiness if any helper starts a loop, retries, runs multiple cycles, captures approval, grants approval, allows apply, reads source, generates new patch metadata, writes files, applies patches, runs tests, invokes subprocess/shell/network/provider/external agents, implements Dominion runtime, or sets execution readiness true.

## Handoff

v0.38.7 hands off only metadata to v0.38.8 CLI Repair Proposal Surface, v0.38.9 Bounded Repair Proposal Loop Consolidation, and future v0.39 human-approved sandbox repair apply contract work. These handoffs are not CLI execution, loop consolidation execution, or sandbox apply.
