# v0.38.3 Repair Scope Planner & Change Intent Model

Parent track: Bounded Repair Proposal Loop.

v0.38.3 builds on v0.38.2 Read-only Sandbox Source Context Snapshot, v0.38.1 Repair Proposal Evidence Contract, v0.38.0 Bounded Repair Proposal Boundary Foundation, and the v0.37.9 Controlled Sandbox Test Runner & Vera-Codex Agent Evaluation v1 consolidation.

## Definition

v0.38.3 creates repair scope planning and change intent metadata only. It consumes v0.38.2 `RepairSourceContextSnapshot`, `RepairSourceContextAssessment`, `RepairSourceExcerpt`, and `RepairSymbolContextHint` metadata, plus v0.38.1 evidence and eligibility metadata. It does not read source files.

A repair scope plan records the likely scope category, candidate affected files, candidate affected symbols, supporting and missing evidence, scope risk, change intent, do-nothing comparison, and future patch metadata input readiness.

An affected file candidate is metadata about a file path already present in source context metadata. It is not file modification.

An affected symbol candidate is metadata about a symbol already described by source context hints. It is not source execution.

A scope evidence map links scope candidates to evidence references, contradictory evidence, and missing items. It is not source read or proposal generation.

A change intent records a future-gated direction such as aligning implementation with a test, reviewing a test expectation, adjusting an import path, inspecting configuration, inspecting a missing dependency without install, or inspecting a timeout without retry. Change intent is not code modification.

A scope risk assessment records overbroad, underbroad, wrong-scope, dependency-install confusion, timeout-retry confusion, edit-permission confusion, generation confusion, execution confusion, external-agent, Dominion, and production-certification risks.

A do-nothing scope comparison is mandatory. Do-nothing remains valid even when a scope plan is better supported than no action.

## Distinctions

Scope planning is not edit permission.

An affected file candidate is not file modification.

An affected symbol candidate is not source execution.

Change intent is not code modification.

Future patch metadata input is not proposed diff or code hunk generation.

Scope confidence is not patch correctness.

## Model Layer

`RepairScopePlanningPolicy` allows only scope planning, affected file candidates, affected symbol candidates, scope evidence map, change intent model, scope risk assessment, do-nothing comparison, and future patch metadata input. It blocks source read, source write, repair proposal generation, proposed diff generation, proposed code hunk generation, proposed patch envelope generation, repair patch proposal, repair execution, patch application, apply_patch, git apply, test execution, subprocess, shell, dependency install, network, model provider invocation, external agent execution, and Dominion runtime.

`RepairScopePlanningInput` is a scope planning request, not a proposal generation request. Its prohibited runtime actions include source read, source write, proposal generation, diff generation, hunk generation, patch envelope generation, patch apply, repair execution, test execution, subprocess, shell, install, network, model provider, external agent, and Dominion.

`RepairAffectedFileCandidate` and `RepairAffectedSymbolCandidate` are metadata only. They must keep source read, import/execute, edit, proposal, diff, hunk, and repair permissions false.

`RepairScopeEvidenceMap` records supporting evidence, contradictory evidence, and missing evidence.

`RepairScopeRiskAssessment` review-gates or blocks high-risk scope states.

`RepairChangeIntent` is future-gated and grants no source read, edit, proposal, diff, hunk, patch envelope, or repair permission now.

`RepairScopeDoNothingComparison` is always represented and keeps the do-nothing alternative valid.

`RepairScopePlan` may mark future patch metadata input ready only when the evidence map, risk assessment, change intent, and do-nothing comparison support it. It must keep source read, proposal generation, diff generation, hunk generation, patch envelope generation, file edit, repair execution, production certification, and execution readiness false.

`RepairScopePlanningDecision` is not approval. It may allow future patch metadata input as metadata only and must not allow source read, proposal generation, diff generation, hunk generation, patch envelope generation, or repair now.

Validation reports confirm metadata-only scope planning, no source read, no source write, no proposal/diff/hunk/envelope generation, no edit, no repair, do-nothing comparison, and human review when needed.

The no-generation guarantee confirms no source read, no source write, no repair proposal generation, no proposed diff generation, no proposed code hunk generation, no proposed patch envelope generation, no file edit, no patch application, no repair execution, no test execution, no subprocess execution, no shell execution, no model provider invocation, no external agent execution, and no Dominion runtime.

## Safe Defaults

v0.38.3 records digestion-first policy, Dominion runtime blocked, external agent execution blocked, infinite agent loop blocked, recursive self-invocation blocked, automatic repair loop blocked, repair execution blocked, model provider invocation blocked, tool execution blocked, bounded repair proposal only, bounded scope planning only, no independent autonomous agent runtime, and mandatory human review before any apply in metadata.

If evidence is insufficient, v0.38.3 prefers do-nothing, human review, or blocked. If source context is insufficient, it prefers human review or more context. If a missing dependency appears likely, it creates dependency inspection scope only and does not install. If timeout appears likely, it creates performance investigation scope only and does not retry. If no repair is needed, it prefers do-nothing. If a safety boundary violation exists, future patch metadata input is blocked.
