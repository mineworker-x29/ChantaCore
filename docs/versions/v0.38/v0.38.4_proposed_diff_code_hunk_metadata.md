# v0.38.4 Proposed Diff / Code Hunk Metadata Generation

Parent track: Bounded Repair Proposal Loop.

v0.38.4 builds on v0.38.3 Repair Scope Planner & Change Intent Model, v0.38.2 Read-only Sandbox Source Context Snapshot, v0.38.1 Repair Proposal Evidence Contract, v0.38.0 Bounded Repair Proposal Boundary Foundation, and v0.37.9 Controlled Sandbox Test Runner & Vera-Codex Agent Evaluation v1.

## Definition

v0.38.4 creates proposed diff, proposed code hunk, proposed file change, and proposed patch envelope metadata only. It may create bounded in-memory proposed diff text for review. It does not write that text to a file and does not apply it.

Proposed diff metadata is bounded review text, not an applied diff.

Proposed code hunk metadata is a proposed change description, not a code edit.

Proposed file change metadata is an affected-file proposal, not file modification.

Proposed patch envelope metadata groups proposed hunks, diffs, rationale, evidence, review requirement, and do-nothing comparison. It is not patch application.

Proposed change rationale explains intent and evidence. It is not proof of correctness.

Proposed change evidence map links scope plan, change intent, affected file/symbol, source excerpt, source context snapshot, evidence bundle, feedback, test result, and do-nothing references. It is metadata only.

Future safety validation input is not safety validation in v0.38.4.

Future human review packet input is not approval.

Do-nothing comparison remains mandatory. Human review remains mandatory before any future apply.

## Policy

`RepairPatchMetadataPolicy` allows proposed diff metadata, proposed code hunk metadata, proposed file change metadata, proposed patch envelope metadata, change rationale metadata, patch evidence map, future safety validation input, and future human review packet input.

It blocks source file read, sandbox source read, live workspace read, source write, sandbox source write, patch file write, file edit, patch application, apply patch, git apply, repair execution, automatic repair, test execution, subprocess, shell, dependency install, network access, model provider invocation, external agent execution, and Dominion runtime.

Unsafe operations are blocked by default: file deletion, file rename, permission change, binary modification, dependency install, network call, subprocess or shell addition, model provider call, external agent call, credential or secret read, and Dominion runtime addition.

Missing dependency does not create an install patch.

Timeout does not create a retry patch.

## Model Flow

v0.38.3 `ScopePlan` + `ChangeIntent` + v0.38.2 `SourceContextSnapshot` + v0.38.1 `EvidenceBundle` -> `PatchMetadataInput` -> `EvidenceMap` -> `Rationale` -> `ProposedCodeHunks` -> `ProposedDiffMetadata` -> `ProposedFileChanges` -> `DoNothingComparison` -> `ReviewRequirement` -> `ProposedPatchEnvelope` -> `V0384ReadinessReport` -> v0.38.5 Repair Proposal Safety & Static Validation.

`RepairPatchMetadataInput` is a proposed patch metadata request, not an apply request.

`ProposedChangeEvidenceMap` records supporting evidence, contradictory evidence, missing evidence, scope refs, and source excerpt refs.

`ProposedChangeRationale` is not proof of correctness and defaults to human review required.

`ProposedCodeHunk` must be bounded and must keep source read by v0.38.4, write, edit applied, patch applied, and repair execution false.

`ProposedDiffMetadata` must be bounded, in-memory only, not written to file, not applied, and not repair execution.

`ProposedFileChange` must keep creates new file, deletes file, renames file, changes permissions, writes file, applies patch, and repair executed false by default.

`ProposedPatchReviewRequirement` is not approval. `human_approval_present` remains false in v0.38.4.

`ProposedPatchDoNothingComparison` is always represented. If do-nothing is preferred or required, the envelope is blocked or review-gated.

`ProposedPatchEnvelope` may mark future safety validation input and future human review packet input ready only when not blocked. It must keep source read by v0.38.4, file write, patch file write, file edit, patch applied, apply patch called, git apply called, tests run, repair executed, production certified, and execution readiness false.

`RepairPatchMetadataDecision` does not allow source read, write, patch file write, edit, apply, apply patch, git apply, repair execution, test execution, model provider invocation, or external agent now.

Validation reports confirm metadata-only proposal, bounded diff text, no file write, no patch write, no edit, no apply, no repair, no tests, do-nothing comparison, and human review requirement.

The no-apply guarantee confirms no write, no patch file, no edit, no apply, no repair, no test, no source read, no subprocess, no shell, no network, no model provider, no external agent, and no Dominion runtime.
