## Definitions

Repair proposal evidence contract: a metadata contract that states which prior artifacts are acceptable evidence for future bounded repair proposal stages.

Repair proposal evidence bundle: a bounded packet of source references, evidence items, evidence gaps, a do-nothing evidence record, an evidence assessment, and an eligibility decision.

Evidence sufficiency assessment: a conservative metadata assessment of whether prior evidence is adequate for later design-stage inputs.

Evidence gap: missing, contradictory, inconclusive, or incomplete metadata that blocks or weakens future source context, scope planning, or patch metadata inputs.

Do-nothing evidence: mandatory metadata that keeps no repair, review, or waiting as a valid alternative.

Eligibility decision: metadata that may mark future source context, scope planning, or patch metadata input as eligible. It is not approval, source read permission, proposal generation permission, or repair execution permission.

## Explicit Distinctions

- Evidence contract is not repair proposal generation.
- Evidence sufficiency is not source read permission.
- Evidence bundle is not a patch proposal.
- Eligibility decision is not approval.
- Eligibility for future source context is not source read in v0.38.1.
- Eligibility for future patch metadata is not diff or code hunk generation in v0.38.1.
- Missing dependency evidence does not grant install permission.
- Timeout evidence does not grant retry permission.
- Do-nothing evidence remains mandatory.
- Human review remains mandatory before any future apply.

## Evidence Policy

Sufficient failed-test, feedback, diagnosis, repair suggestion, and cold scorecard metadata may support future repair proposal path input. Missing evidence produces insufficient or blocked metadata. Contradictory evidence produces review-required or blocked metadata. Inconclusive test evidence must not produce high-confidence eligibility. Passed or no-failure evidence should prefer no repair or do-nothing. Safety boundary violation evidence blocks future repair proposal path.

## Acceptance Criteria

- Evidence contract, bundle, assessment, gaps, do-nothing evidence, and eligibility decision can be created as pure metadata.
- Missing required evidence creates gaps.
- Contradictory evidence creates review-required or blocked metadata.
- Sufficient evidence may mark future input metadata ready.
- Future input readiness never grants current source read, generation, edit, apply, test, model, external-agent, Dominion, or execution permission.
- No-generation guarantee confirms all no_* fields.

## Withdrawal Conditions

Withdraw v0.38.1 readiness if source-read permission, repair proposal generation, diff or hunk generation, patch envelope generation, file edit, patch application, test execution, subprocess, shell, dependency install, network, model invocation, external agent execution, Dominion runtime, production certification, or `ready_for_execution=True` appears in the implemented v0.38.1 runtime/helper behavior.
