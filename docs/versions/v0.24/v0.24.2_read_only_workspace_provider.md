# v0.24.2 Read-only Workspace Provider

Korean name: 읽기 전용 워크스페이스 Provider

Track: Internal Provider / Local Runtime Provider

Previous: v0.24.1 Provider Registry & Capability Surface

Next: v0.24.3 Repository Search / File Read Provider

## Version Definition

v0.24.2 is the first implemented read-only internal provider stage. It activates only the workspace provider surface declared in v0.24.1, and only for workspace root discovery, directory tree observation, and file metadata observation.

This release does not read file contents, extract file excerpts, run repository search, create local command candidates, execute local commands, call external providers, or introduce General Agent UX.

## Objective

The objective is to make workspace structure visible through a policy-bound internal provider report while preserving the boundary established by v0.24.0 and v0.24.1:

- `WorkspaceReadProviderPolicy`
- `WorkspaceRootDescriptor`
- `WorkspaceIgnorePattern`
- `WorkspaceIgnorePolicy`
- `WorkspacePathSanitizationPolicy`
- `WorkspaceTreeRequest`
- `WorkspaceDirectoryNode`
- `WorkspaceFileMetadata`
- `WorkspaceTreeSnapshot`
- `WorkspaceMetadataSummary`
- `WorkspaceReadFinding`
- `WorkspaceReadReport`

## Relationship To Earlier Versions

v0.24.0 declared the Internal Provider Contract. v0.24.1 declared the provider registry and capability surfaces. v0.24.2 activates only `workspace_read_provider` for metadata and tree observation.

v0.23.9 remains the previous Internal Dominion Foundation v1 consolidation source. v0.24.2 does not reopen Dominion dispatch, authorization consumption, live provider status, or real external outcomes.

## Core Principles

- Workspace provider is read-only.
- Workspace tree observation is not file content reading.
- File metadata is not file content.
- Ignore policy must be applied before output.
- Private paths must be sanitized.
- Secret-like paths must be masked or excluded.
- Workspace observation must be OCEL-visible.
- No command execution is allowed.

## Included Scope

v0.24.2 may:

- discover declared workspace roots through sanitized labels;
- inspect directory structure with bounded depth and entry limits;
- observe file metadata such as extension, size, modified-time reference, hidden status, symlink status, generated/binary/secret-like classification;
- summarize extension counts, file kind counts, ignored counts, hidden counts, and truncation state;
- emit deterministic findings for skipped, masked, truncated, or blocked cases.

## Non-Goals

v0.24.2 must not implement:

- file content read;
- file excerpt read;
- repository text or symbol search;
- file content extraction;
- local command candidate creation;
- local runtime static safety, preflight, or execution;
- shell, network, package install, destructive command, MCP, plugin loading;
- external provider adapter;
- ask/repl/general agent UX;
- Workspace Agent Workbench beyond this provider's read-only report;
- Memory Candidate / Continuity;
- ChantaCore-Schumpeter split or company-specific wrapper;
- credential/token reading, storage, or output;
- GrowthKernel runtime dependency;
- vendor-specific runtime logic;
- LLM judge.

## Data Models

`WorkspaceReadProviderPolicy` fixes read-only behavior, disabled content/excerpt/search/command execution, symlink policy, hidden/ignored defaults, traversal limits, private path sanitization, credential path masking, and raw secret prohibition.

`WorkspaceRootDescriptor` represents roots by sanitized labels and path refs, not private full paths.

`WorkspaceIgnorePolicy` contains default ignored directories including `.git`, `.venv`, `venv`, `node_modules`, `__pycache__`, `.pytest_cache`, `.pytest-tmp`, `.mypy_cache`, `.ruff_cache`, `dist`, `build`, `data`, `letters`, and `references`.

`WorkspacePathSanitizationPolicy` requires relative/sanitized paths, home/user/drive masking where needed, secret-like path masking, and max display length.

`WorkspaceTreeSnapshot` records directories and metadata-only files, count summaries, truncation, and fixed false boundary flags for file content, repository search, local command, external runtime touch, credentials, and raw secrets.

`WorkspaceReadReport` is the release-facing readiness artifact and points next to v0.24.3 Repository Search / File Read Provider.

## Service Design

- `WorkspaceProviderContractSourceService` loads the v0.24.0 contract, v0.24.1 registry, and workspace provider surface.
- `WorkspaceReadPolicyService` builds the read-only provider policy.
- `WorkspaceRootDiscoveryService` declares and validates roots without reading content.
- `WorkspaceIgnorePolicyService` applies default ignored directories and secret-like patterns before output.
- `WorkspacePathSanitizationService` emits relative/sanitized paths and masks secret-looking names.
- `WorkspaceTreeTraversalService` observes directory tree and metadata using filesystem metadata only.
- `WorkspaceFileMetadataService` classifies metadata without opening files.
- `WorkspaceMetadataSummaryService` summarizes metadata-only observations.
- `WorkspaceReadFindingService` builds deterministic findings.
- `WorkspaceReadReportService` builds report, PIG report, OCPX projection, and CLI output.
- `ReadOnlyWorkspaceProviderService` is the provider-facing read-only entry point.

## Traversal Policy

Traversal defaults:

- max depth: 4
- max entries: 2000
- follow symlinks: false
- include hidden files: false
- include ignored files: false
- include file metadata: true
- include file content: false
- include file excerpt: false

Limits are enforced before unbounded output. Truncation is reported through `truncated` and `truncation_reason`.

## Ignore And Sanitization Policy

Ignore policy is applied before output. Secret-like names such as `.env`, `*.pem`, `*.key`, `id_rsa`, `secrets.*`, `credentials.*`, and `token*` are masked or excluded. Reports and CLI output use relative paths or sanitized root labels, not private full paths.

## Status Rules

`passed` means the workspace provider surface exists, at least one allowed root is available, traversal completes within limits, metadata summary is produced, paths are sanitized, ignore policy is applied, and no forbidden action occurs.

`warning` means traversal was truncated, hidden/ignored paths were skipped, secret-like paths were masked, symlinks were not followed, or optional metadata was unavailable.

`failed` means required policy or usable root is missing, or repository/file-content behavior was requested but not yet blocked at critical severity.

`blocked` means credentials, raw secrets, private full paths, local commands, provider API calls, external runtime touch, vendor runtime logic, Schumpeter split, or LLM judge are detected.

- Private paths are sanitized.
- Secret-like paths are masked or excluded.
- max_depth and max_entries are enforced.
- No file content read occurs.
- No file excerpt read occurs.
- No repository search occurs.
- No local command execution occurs.
- No external provider adapter is implemented.
- No Schumpeter split or company-specific wrapper is introduced.
- No credential value is stored or output.
- No private full path is output.
- No LLM judge is used.
- OCEL/PIG/OCPX visibility exists.
- v0.24.1, v0.24.0, v0.23.9, v0.23.x, v0.22.x, v0.21.x, and v0.20.x tests remain valid.

## Withdrawal Conditions

Withdraw this release if file content or excerpts are read, repository search is performed, local command candidates are created, local commands are executed, external providers are invoked, credentials or raw secrets are emitted, private full paths are output, vendor runtime logic is added, Schumpeter split is introduced, GrowthKernel becomes an active runtime dependency, or LLM judge is used.

## Validity Horizon

Valid until workspace layout, ignore policy, sanitization policy, provider policy, or v0.24.x roadmap boundaries change.
