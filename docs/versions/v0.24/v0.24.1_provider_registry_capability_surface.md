# v0.24.1 Provider Registry & Capability Surface

Version: v0.24.1
Version name: Provider Registry & Capability Surface
Korean name: Provider 레지스트리·Capability 표면
Track: Internal Provider / Local Runtime Provider
Previous: v0.24.0 Internal Provider Contract
Next: v0.24.2 Read-only Workspace Provider

## Objective

v0.24.1 declares the Internal Provider registry and capability surfaces that
later v0.24.x versions may activate. It reads the v0.24.0
`InternalProviderContract`, preserves v0.23.9 Internal Dominion release
readiness as a predecessor boundary, and creates provider refs, capability
surface descriptors, provider statuses, registry snapshots, and registry
reports.

## Policy

v0.24.x is Internal Provider / Local Runtime Provider. v0.24.1 is
registry/surface-only.

v0.24.1 does not implement provider invocation, workspace file read execution,
directory tree reading execution, repository search execution, file content
extraction, file excerpt read execution, OCEL/PIG/OCPX live inspection
execution, local command candidate creation, local runtime execution, local
command execution, external provider adapters, General Agent UX, Schumpeter
split, GrowthKernel runtime dependency, credential output, or LLM judge.

## Principles

- Provider registry is not provider invocation.
- Capability surface is not capability execution.
- Provider registration is not workspace read.
- Provider registration is not repository search.
- Provider registration is not local runtime execution.
- Provider surface must declare effect type, permission boundary, OCEL
  visibility, and implementation status.
- Every provider capability is deny-by-default until its activation version.

## Service Design

- `InternalProviderContractSourceService` loads the v0.24.0 contract.
- `InternalProviderRefFactory` builds provider refs from the request and
  contract.
- `InternalProviderCapabilitySurfaceFactory` builds capability descriptors and
  surfaces.
- `InternalProviderStatusService` builds provider statuses.
- `InternalProviderRegistryService` builds the registry.
- `InternalProviderRegistrySnapshotService` builds a snapshot.
- `InternalProviderRegistryFindingService` builds deterministic findings.
- `InternalProviderRegistryReportService` builds registry and surface reports,
  PIG reports, OCPX projections, and CLI output.

## Status Rules

Passed if the v0.24.0 contract exists, required providers and capability
surfaces are declared, invocation/execution flags are false, external adapter
counts are zero, dangerous counts are zero, capability descriptors define
effect/permission/OCEL fields, and roadmap boundaries are aligned.

Warning if optional descriptions are partial or candidate generation activation
is future-track but no dangerous count is nonzero.

Failed if a required provider, capability surface, effect policy, permission
policy, or OCEL mapping is missing, or if invocation/read/local-runtime
execution is enabled too early.

Blocked if credentials, raw secrets, external adapters, vendor runtime logic,
network/shell/package/destructive command, Schumpeter split, GrowthKernel
active dependency, or LLM judge are introduced.

`provider_surface_declared`.

## Acceptance Criteria

The registry, snapshot, registry report, and capability surface report can be
created. Required provider refs and surfaces are declared. All invocation and
execution flags remain false. No provider invocation, workspace/repo/file read
execution, local command candidate creation, local command execution, external
provider adapter, Schumpeter split, credential output, or LLM judge is
introduced. OCEL/PIG/OCPX visibility exists. `ready_for_v0_24_2` is calculated
and `ready_for_v0_25=false`.

## Withdrawal Conditions

Withdraw v0.24.1 readiness if provider invocation, workspace/repo/file read
execution, local command candidate creation, local command execution, local
runtime execution, external provider adapter logic, vendor runtime logic,
credential exposure, raw secret output, GrowthKernel active dependency,
Schumpeter split, General Agent UX, or LLM judge appears.

## Validity Horizon

Valid until provider registry implementation changes or provider capability
surfaces are activated.
