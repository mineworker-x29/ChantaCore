## Version Definition

v0.23.7 records explicit human review decisions and evaluates the Dominion Gate after foundation-level runtime preflight. It may create a scoped, single-use authorization artifact for the next stage, but it does not consume that authorization and does not dispatch.

## Principles

- Human review is not dispatch.
- Dominion gate is not dispatch.
- Gate authorization is not execution.
- Gate authorization is not provider API permission by itself.
- Gate authorization is single-use and scoped.
- Approval is not sufficient without v0.23.8 boundary handling.
- No-action and needs-more-input are valid review outcomes.

## Included Scope

- Read-only source bundle loading.
- Human-readable review summary.
- Deterministic review decision normalization.
- Gate condition evaluation.
- Gate state creation.
- Optional single-use, unconsumed authorization artifact for v0.23.8.
- Rejection, no-action, and needs-more-input candidate/record models.
- OCEL, PIG, OCPX, and CLI visibility.

## Non-Goals

## Data Models

The version defines project-style dataclasses for review request creation, source bundle, review summary, decision input, review decision, gate condition, gate state, gate authorization, gate finding, gate report, needs-more-input candidate, no-action candidate, and rejection record.

## Service Design

- `DominionGateSourceService` loads read-only source states.
- `DominionReviewSummaryService` builds review summaries.
- `DominionReviewDecisionService` records deterministic explicit decisions.
- `DominionGateConditionService` creates gate conditions.
- `DominionGateEvaluationService` evaluates gate state.
- `DominionGateAuthorizationService` creates unconsumed single-use authorization only for open gates.
- `DominionGateFindingService` builds findings.
- `DominionHumanReviewGateService` coordinates the flow.

## Gate Condition Policy

Required gate conditions include preflight existence, preflight gate eligibility, static safety pass or acceptable warning, control plan existence, action candidate existence, no provider API call, no external runtime touch, no dispatch, no credential exposure, explicit approval checks, high-risk approval phrase checks, production strong gate checks, and roadmap/migration continuity checks.

## Decision And Status Rules

`open` requires explicit approve, eligible preflight, acceptable static safety, all hard gate conditions passed, and single-use unconsumed authorization creation.

`rejected`, `no_action`, and `needs_more_input` are terminal review outcomes for this stage and do not create authorization.

`blocked` applies to hard violations such as provider API call, runtime touch, dispatch attempt, credential exposure, static safety failure, preflight ineligibility, vendor hardcoding, GrowthKernel active dependency, premature Local Runtime Provider, premature General Agent Usability, or Schumpeter split.

## OCEL Mapping

v0.23.7 adds human review, gate condition, gate state, gate authorization, gate report, candidate, and rejection object/event/relation types. Effect types are `read_only_observation`, `state_candidate_created`, and `gate_state_created`.

## CLI Commands

- `python -m chantacore.cli dominion gate review --preflight-report-id <id> --decision approve --approval-phrase "<phrase>"`
- `python -m chantacore.cli dominion gate review --preflight-report-id <id> --decision reject`
- `python -m chantacore.cli dominion gate report --report-id <id>`
- `python -m chantacore.cli dominion gate state --gate-id <id>`
- `python -m chantacore.cli dominion gate authorization --authorization-id <id>`
- `python -m chantacore.cli dominion gate findings --report-id <id>`

CLI output is sanitized and does not print credential values, raw secrets, or private full paths.

## PIG / OCPX

PIG subject is `human_review_dominion_gate`, with `gate_state_created=True`, `authorization_created=conditional`, `authorization_consumed=False`, `safe_to_dispatch=False`, and `bounded_dispatch_allowed_now=False`.

OCPX state is `dominion_gate_opened_or_review_recorded`, targeting human review, gate, gate authorization, v0.23.8 eligibility, and roadmap boundary read models.

## Acceptance Criteria

- `DominionGateReport` can be created from `DominionRuntimePreflightReport`.
- Human review decisions can be recorded.
- Gate conditions are evaluated.
- Approved eligible plans can create single-use unconsumed authorization.
- Reject/no-action/needs-more-input produce appropriate non-dispatching outcomes and no authorization.
- No dispatch, provider API call, external runtime touch, authorization consumption, credential output, local command execution, LLM judge, Schumpeter split, or v0.24+ implementation occurs.
- OCEL/PIG/OCPX visibility exists.

## Withdrawal Conditions

Withdraw this version if gate authorization is consumed, bounded dispatch starts, provider APIs are called, external runtime is touched, credentials are read or output, local commands are executed, LLM judge approval is introduced, GrowthKernel becomes an active dependency, vendor runtime logic enters core Dominion, or Schumpeter split/company-specific logic is added.

## Validity Horizon

Valid until review decision, preflight report, control plan, risk profile, or Dominion policy changes.
